# Архитектура AI-пайплайна для анализа задач Jira по направлению «Сопровождение 1С»

> Практическое руководство по проектированию production-grade пайплайна, обрабатывающего 1000–3000+ задач за квартальный прогон, устойчивого к лимитам облачных LLM и грязным данным.
> Все рекомендации даны в формате, пригодном для прямой передачи разработчику.

---

## 1. Общая архитектура решения

Пайплайн строится как ETL/ELT-конвейер с несколькими LLM-этапами и классическим ML-слоем (embeddings + clustering). Жёстко разделяем «детерминированную обработку» (Python-код) и «вероятностную обработку» (LLM): чем больше работы можно сделать кодом — тем дешевле, стабильнее и быстрее прогон.

**Полный конвейер этапов:**

```
[1] Выгрузка из Jira (REST API / JQL)
        │
        ▼
[2] Сырое сохранение  ──►  jira_tasks_raw  (PostgreSQL + JSONB)
        │
        ▼
[3] Очистка и нормализация (regex, BeautifulSoup, dedup)
        │   • снимаем HTML/Wiki-разметку
        │   • удаляем подписи, цитаты, служебные системные сообщения
        │   • нормализуем 1С-термины (словарь синонимов)
        │   • токенизация, подсчёт длины
        ▼
[4] Структурирование полей  ──►  jira_tasks_cleaned
        │
        ▼
[5] Чанкование длинных полей (Журналы работ, Комментарии)
        │   • RecursiveCharacterTextSplitter, ~1500–2000 токенов
        │   • группировка по дате/автору
        ▼
[6] Этап 1: Map — анализ ОДНОЙ задачи LLM-ом
        │   • строгий JSON по схеме (Pydantic + Instructor)
        │   • при длинных текстах — внутренний map_reduce полей
        │   • запись в ai_task_analysis + model_runs (трейс)
        ▼
[7] Embeddings задач (multilingual-e5-large / BGE-M3)
        │   • вектор по композиции title + short_summary + tags
        │   • запись в pgvector / Qdrant
        ▼
[8] Кластеризация (UMAP → HDBSCAN → c-TF-IDF / BERTopic)
        │   • многомерные разрезы: тип проблемы, область 1С, причина и т.д.
        │   • LLM-разметка кластеров (короткое имя + описание)
        ▼
[9] Reduce — иерархическая агрегация по кластерам и командам
        │   • суммаризация внутри кластера (map_reduce)
        │   • защита редких кейсов (anomaly buckets)
        ▼
[10] Финальное управленческое summary (executive prompt)
        │
        ▼
[11] Контроль качества (QA-слой)
        │   • валидность JSON, полнота полей, confidence_score
        │   • LLM-as-judge на случайной выборке
        │   • проверка соответствия исходному тексту
        ▼
[12] Сохранение артефактов: ai_reports, ai_clusters, экспорт в BI/дашборд
```

Ключевые принципы:

- **Идемпотентность**: каждый этап можно перезапустить отдельно по списку `task_id`.
- **Инкрементальность**: новые задачи входят в пайплайн без полного пересчёта.
- **Двухскоростная обработка**: дешёвая модель — для каждой задачи (Map), сильная модель — только для финального reduce и спорных случаев.
- **Трейсинг**: каждый LLM-вызов логируется (промпт, ответ, токены, модель, latency, стоимость) в Langfuse/Helicone.

## 2. Какие данные передавать в модель

Для каждого поля задачи Jira формулируем: зачем оно, как чистить, какие риски, как ужимать.

### 2.1 Наименование (Summary)
- **Зачем**: самый плотный носитель смысла; часто содержит ключевую категорию задачи («Ошибка проведения документа», «Доработка отчёта», «Обновление БП 3.0»).
- **Очистка**: убрать префиксы команд/проектов, номера тикетов, эмодзи, повторяющиеся метки в квадратных скобках.
- **Риск**: формальные/копипастные заголовки («Проблема», «Помогите»). Нельзя слепо доверять.
- **Сжатие**: не сжимаем — поле короткое, передаём целиком.

### 2.2 Цель (бизнес-цель/Goal)
- **Зачем**: позволяет извлечь бизнес-процесс и зачем задача нужна (ключевой вход для `business_goal`).
- **Очистка**: убрать общие фразы («чтобы было удобнее», «для работы»), оставить конкретику.
- **Риск**: часто пустое или формально заполненное. Помечать как `missing_information`.
- **Сжатие**: до 1–2 предложений, при отсутствии — `null`.

### 2.3 Текущее поведение
- **Зачем**: основа для `current_behavior` и `issue_type`. Часто содержит «как ошибка проявляется».
- **Очистка**: HTML→text; вырезать стектрейсы, оставив их сигнатуру (первые 2 строки + хеш) — стектрейс целиком модели не нужен.
- **Риск**: смешение «что делал пользователь» и «что система ответила». Нельзя выкидывать сообщения об ошибке 1С (например, «Ошибка СУБД», «Конфликт блокировок», «Объект не найден»).
- **Сжатие**: оставлять признаки ошибок 1С (Microsoft SQL Error, ORA-, «Конфликт блокировок при выполнении транзакции», номера форм/обработок).

### 2.4 Описание
- **Зачем**: контекст, расширяющий заголовок; часто содержит ссылки на конфигурации, релизы, документы.
- **Очистка**: убрать вложенные подписи, скриншоты-ссылки, повторы заголовка.
- **Риск**: бывает либо пусто, либо «простыня». Нужно ограничивать длиной и предварительной summary.
- **Сжатие**: если >1500 токенов — предварительный summary через дешёвую модель → подставить в промпт результата.

### 2.5 Журналы работ (Worklog)
- **Зачем**: фактические действия сотрудников, время, шаги решения. Источник для `employee_actions` и `complexity_level`.
- **Очистка**: дедупликация (часто одинаковые автокомментарии «начал работу/завершил»); удаление пустых строк, статусных переходов («Status → In Progress»).
- **Риск**: длинные «лог переписки» без итога. Самый шумный источник.
- **Сжатие**: иерархическая суммаризация (см. §13). На вход финальной модели — только суммарные действия и ключевые шаги.

### 2.6 Комментарии сотрудников
- **Зачем**: рассуждения, гипотезы, обмен между сопровождением и пользователем; часто содержат root cause.
- **Очистка**:
  - убирать quoted reply («> писал:»);
  - удалять подписи («С уважением, …»);
  - снимать боты (Jira Automation, e-mail handler);
  - дедуплицировать по нормализованному тексту (lower, без пунктуации, fuzzy hash).
- **Риск**: «переписки без итога», флуд, ссылки на скрины. Высокий риск галлюцинации, если модель примет вопрос за факт.
- **Сжатие**: оставлять только комментарии с содержательной плотностью (длина >40 символов и ≥1 термин 1С/ИТ); группировать по автору и дате; map-reduce по чанкам.

### 2.7 Общая стратегия сжатия
1. **Truncate by relevance, not by length**: сначала фильтр по эвристикам, потом отрезание по токенам.
2. **Сохранять «технические сигналы»**: коды ошибок 1С, имена объектов метаданных (Документы.\*, Регистры.\*, Обработки.\*), названия конфигураций (БП, ЗУП, УТ, ERP, КА, Розница), версии платформы (8.3.x).
3. **Бюджет на одну задачу**: ~3–5k токенов входа в Map-этапе. Если поле этого превышает — ему нужен предварительный summary.

## 3. Промежуточная JSON-структура одной задачи

Это основной артефакт Map-этапа, единый контракт для кластеризации и Reduce-этапа.

```json
{
  "task_id": "SUP1C-12345",
  "title": "Ошибка проведения реализации в БП 3.0 после обновления",
  "short_summary": "После обновления конфигурации БП 3.0 до 3.0.156 пользователи не могут провести документ 'Реализация (акты, накладные)'; возникает ошибка при списании партий.",
  "business_goal": "Восстановить регулярное оформление продаж и закрытие месяца",
  "current_behavior": "При проведении документа возникает сообщение 'Не найдена партия для списания' для номенклатуры из группы 'Услуги'",
  "issue_type": "bug",
  "work_type": "incident_fix",
  "affected_1c_area": ["БП 3.0", "Учёт реализации", "Партионный учёт"],
  "root_cause_hypothesis": "После обновления изменилась логика подбора партий; настройка учётной политики не пересохранена",
  "employee_actions": [
    "Воспроизвели ошибку на копии базы",
    "Сравнили учётную политику до/после обновления",
    "Перенастроили списание партий по среднему",
    "Перепровели документы за период"
  ],
  "final_status_or_outcome": "resolved_workaround",
  "recurring_problem_markers": ["обновление_конфигурации", "партионный_учёт", "учётная_политика"],
  "risk_level": "high",
  "complexity_level": "medium",
  "automation_potential": "medium",
  "documentation_needed": true,
  "tags": ["БП3.0", "обновление", "проведение", "партии", "учётная_политика"],
  "confidence_score": 0.82,
  "missing_information": ["точная_версия_до_обновления", "число_затронутых_пользователей"]
}
```

**Пояснения по полям:**

| Поле | Тип | Назначение | Допустимые значения / правила |
|---|---|---|---|
| `task_id` | string | Ключ Jira | копируется как есть |
| `title` | string | Очищенный заголовок | до 200 символов |
| `short_summary` | string | Управленческое резюме сути | 2–4 предложения, без жаргона, по-русски |
| `business_goal` | string\|null | Бизнес-цель, ради которой обращались | null если не выводится из текста |
| `current_behavior` | string\|null | Что сейчас происходит / симптом | пишется языком симптома, не гипотезы |
| `issue_type` | enum | Природа задачи | `bug` / `enhancement` / `consultation` / `routine_op` / `access_request` / `integration_issue` / `data_fix` / `other` |
| `work_type` | enum | Характер выполненной работы | `incident_fix` / `config_change` / `code_change` / `update_install` / `report_dev` / `integration_setup` / `user_consult` / `rights_setup` / `data_load` / `regulatory` |
| `affected_1c_area` | array<string> | Область 1С | контролируемый словарь: конфигурация (БП/ЗУП/УТ/ERP/КА/Розница/УНФ/ДО/самописная), подсистема, объект |
| `root_cause_hypothesis` | string\|null | Гипотеза о причине | модель ОБЯЗАНА пометить как «гипотеза», иначе запретить |
| `employee_actions` | array<string> | Шаги, которые делали сотрудники | 3–10 пунктов, глаголами совершенного вида |
| `final_status_or_outcome` | enum | Чем закончилось | `resolved` / `resolved_workaround` / `escalated` / `not_reproduced` / `cancelled` / `in_progress` / `unknown` |
| `recurring_problem_markers` | array<string> | Признаки повторяющейся проблемы | snake_case, словарь нормализованных меток |
| `risk_level` | enum | Бизнес-риск | `low` / `medium` / `high` / `critical` |
| `complexity_level` | enum | Трудоёмкость | `low` / `medium` / `high` |
| `automation_potential` | enum | Возможность автоматизации | `none` / `low` / `medium` / `high` + комментарий |
| `documentation_needed` | bool | Нужна ли статья KB | true/false |
| `tags` | array<string> | Свободные теги для поиска | 3–10 |
| `confidence_score` | float [0..1] | Самооценка модели | по сумме признаков (см. §12) |
| `missing_information` | array<string> | Чего не хватило | если поле пустое — выводить из текста явно |

Контракт оформляется как Pydantic-модель и используется через **Instructor** или native structured outputs OpenRouter (`response_format: json_schema`) — это даёт автоматический retry при невалидном ответе.

## 4. Этап 1: промпты для анализа одной задачи 1С

### 4.1 Системный промпт (на русском, под 1С)

```
Ты — аналитик службы сопровождения 1С: Предприятие. Тебе дают карточку
задачи Jira: заголовок, цель, текущее поведение, описание, журналы работ,
комментарии. Твоя задача — извлечь структурированные сведения для
управленческой аналитики.

ПРАВИЛА:
1. Пиши строго на русском, профессиональным языком сопровождения 1С.
2. Не выдумывай факты. Если данных нет — ставь null или пустой массив,
   а в поле "missing_information" перечисли, чего не хватило.
3. Гипотезы помечай явно («предположительно», «вероятно»), они идут
   только в поле "root_cause_hypothesis".
4. Используй контролируемые словари для перечислений (issue_type,
   work_type, final_status_or_outcome, risk_level, complexity_level,
   automation_potential). Если значение не подходит — выбирай
   ближайшее, в "missing_information" указывай неоднозначность.
5. Для affected_1c_area используй короткие нормализованные термины:
   "БП 3.0", "ЗУП 3.1", "УТ 11", "ERP 2", "КА 2", "Розница 3",
   "УНФ", "ДО", "самописная конфигурация", "платформа 8.3",
   "регламентные операции", "права и роли", "интеграция",
   "обмен данными", "отчётность", "печатные формы", "БСП".
6. recurring_problem_markers — это короткие snake_case ярлыки
   повторяющихся симптомов: ошибка_обновления, конфликт_блокировок,
   некорректные_права, ошибка_обмена, медленный_отчёт,
   повисший_фоновый_процесс и т.п.
7. confidence_score: 1.0 — все ключевые поля выведены из текста явно;
   0.7 — часть выведена косвенно; 0.4 и ниже — данных мало, выводы
   неуверенные.
8. Ответ — ОДИН валидный JSON-объект по заданной схеме. Никакого
   текста до или после.
```

### 4.2 Пользовательский промпт (шаблон)

```
КАРТОЧКА ЗАДАЧИ JIRA:

[task_id]: {{task_id}}
[Команда]: {{team}}
[Наименование]:
{{title}}

[Цель]:
{{goal_or_null}}

[Текущее поведение]:
{{current_behavior_or_null}}

[Описание]:
{{description_compressed}}

[Журналы работ — суммаризовано]:
{{worklog_summary}}

[Комментарии — суммаризовано]:
{{comments_summary}}

ЗАДАНИЕ:
Проанализируй задачу и верни JSON по схеме TaskAnalysis.

ОБЯЗАТЕЛЬНО:
- short_summary — 2–4 предложения управленческим языком, без жаргона.
- employee_actions — список шагов, реально выполненных сотрудниками
  (по журналам работ и комментариям). Если шагов нет — пустой массив.
- root_cause_hypothesis — формулируй как гипотезу, не как факт.
- Если задача — типовая консультация без действий, ставь work_type
  "user_consult" и complexity_level "low".
- Если задача — регламентная операция (закрытие месяца, выгрузка в
  ФНС/ПФР, обновление классификаторов), ставь work_type "regulatory".
- Если в комментариях идёт переписка без итога — отметь это в
  missing_information и поставь final_status_or_outcome "unknown".

Верни только JSON.
```

### 4.3 Защита от галлюцинаций на уровне Map-этапа
- Pydantic-валидация + `Instructor.max_retries=2` с автоматическим reasking.
- Whitelisting перечислений: модель не может выдать `work_type` вне списка.
- Постпроверка: если `confidence_score >= 0.7`, но `missing_information` непустое — снижаем до 0.5 (несоответствие).
- При ответе с `final_status_or_outcome = "resolved"` проверяем, что в `employee_actions` есть ≥1 элемент или явное упоминание результата.

## 5. Этап 2: пакетная обработка 100 / 1000 / 10000+ задач

### 5.1 Размер батча и стратегия запросов

| Объём | Стратегия | Параллельность | Комментарий |
|---|---|---|---|
| <100 | По одной задаче, sequential | 1–3 | Дёшево, отладочно |
| 100–1000 | По одной задаче, async | 5–10 одновременных запросов | Лимит 20 RPM на free-моделях OpenRouter |
| 1000–3000 (квартал) | По одной задаче, async + очередь | 8–16 (paid модели) | Целевой режим |
| 3000–10000+ | Очередь Celery/RQ + несколько моделей-воркеров | 16–32 | Шардирование по командам |

**Главное правило: одна задача = один LLM-вызов на Map-этапе.** Это даёт:
- стабильный JSON;
- независимый retry на сбое;
- идемпотентность (можно перезапустить только сломавшиеся);
- честный confidence_score.

**Группировка** возникает только на Reduce-этапе: по кластеру, по команде, по типу проблемы.

### 5.2 Хранение промежуточных результатов
- Каждый Map-результат немедленно записывается в `ai_task_analysis` (см. §11).
- В `model_runs` фиксируем: `task_id`, `model`, `prompt_version`, `tokens_in/out`, `latency_ms`, `cost`, `status`, `raw_response`.
- Если задача уже проанализирована текущей версией промпта той же моделью — пропускаем (idempotent skip).

### 5.3 Защита от потери редких но важных проблем

Опасность Reduce-этапа — «среднее» поглощает выбросы. Меры:

1. **Anomaly bucket**: задачи с `risk_level in ('high','critical')` или маленьких кластеров (<3 задач) сохраняем отдельным разделом отчёта.
2. **Двойной проход кластеризации**: первая — крупная (top-N тем), вторая — внутри «шумного кластера −1» HDBSCAN, чтобы редкое не утонуло.
3. **Stratified sampling**: при суммаризации кластера всегда включаем хотя бы 1 пример с самым высоким `risk_level`.
4. **Long-tail report**: отдельный список из 20–50 «уникальных» задач, не попавших ни в один крупный кластер.

### 5.4 Защита от доминирования популярных задач
- Перед суммаризацией кластера применяем **near-duplicate dedup** по embeddings (cosine ≥ 0.92 → одна репрезентативная).
- В Reduce-промпт явно передаём «частоту» как число, а не как N копий текста: `{"cluster_size": 87, "examples": [3 типичных, 2 пограничных, 1 экстремальный]}`.

### 5.5 Инкрементальный анализ
- При новом квартальном прогоне: сравниваем хеши задач (`hash(title + description + worklog_last_n)`); пересчитываем только изменённые/новые.
- Кластеры пересчитываются полностью (быстро на embeddings), но имена и описания кластеров сохраняем стабильными через **cluster matching** по центроидам с предыдущим прогоном.

## 6. Этап 3: кластеризация по разрезам

### 6.1 Используемые поля
Для построения вектора задачи объединяем: `title + short_summary + " ".join(tags) + " ".join(affected_1c_area) + " ".join(recurring_problem_markers)`. Это даёт «семантически плотное» представление, очищенное от шума исходного описания.

### 6.2 Технологический стек кластеризации
- **Embeddings**: `intfloat/multilingual-e5-large` (универсально, отличный русский) или `BAAI/bge-m3` (длинный контекст до 8192 токенов, мульти-функциональный, очень силён на русском). Для большей точности на русском техническом тексте можно взять русский файнтюн `deepvk/USER-bge-m3` (вариант BGE-M3, дообученный на русском).
- **Снижение размерности**: UMAP (n_neighbors=15, min_dist=0.0, n_components=5–10).
- **Кластеризация**: HDBSCAN (`min_cluster_size=5..15`, `metric='euclidean'`).
- **Описание тем**: c-TF-IDF (BERTopic) для извлечения ключевых терминов; затем LLM-наименование кластера (короткий промпт «дай заголовок и описание группы по этим примерам»).
- **Альтернатива «всё в одном»**: BERTopic с custom `embedding_model=SentenceTransformer('intfloat/multilingual-e5-large')` или BGE-M3 — он внутри использует UMAP+HDBSCAN+c-TF-IDF.

### 6.3 Многомерная кластеризация (несколько «разрезов»)
Один embedding → несколько проекций под разные вопросы. На практике лучше работают **несколько отдельных кластеризаций** на «фокусированных» текстах:

| Разрез | Текст для embedding | Пример вывода |
|---|---|---|
| Тип проблемы | `issue_type + current_behavior + recurring_problem_markers` | «Ошибки проведения после обновления» |
| Характер работ | `work_type + employee_actions[:3]` | «Перенастройка прав по запросу» |
| Область 1С | `affected_1c_area + tags` | «БП 3.0 → закрытие месяца» |
| Бизнес-процесс | `business_goal + title` | «Закрытие периода», «Расчёт зарплаты» |
| Источник проблемы | `root_cause_hypothesis + recurring_problem_markers` | «Связаны с регламентными обновлениями» |
| Повторяемость | `recurring_problem_markers + tags` | «Типовая повторяющаяся» |
| Потенциал автоматизации | `automation_potential + work_type + employee_actions` | «Можно автоматизировать (выгрузки)» |
| Потребность в KB | `documentation_needed=true + issue_type` | «Темы для базы знаний» |

Каждый разрез сохраняется отдельной кластеризацией в `ai_clusters` с пометкой `cluster_dimension`.

### 6.4 Объединение похожих формулировок
- Перед кластеризацией: **canonicalization** через словарь синонимов 1С (например: «1Ска», «1С-ка», «база» → нормализуются к контексту).
- Семантический dedup: для каждой пары задач в одном кластере с cosine ≥ 0.95 — помечаем `duplicate_of`.
- Для кластеров с очень похожими центроидами (≥ 0.9 cosine между центрами) — merge на отчётном уровне.

### 6.5 Псевдокод кластеризации

```python
from sentence_transformers import SentenceTransformer
import umap, hdbscan
from bertopic import BERTopic
from bertopic.vectorizers import ClassTfidfTransformer

model = SentenceTransformer("intfloat/multilingual-e5-large")
texts = [build_cluster_text(t, dim="issue_type") for t in tasks]
embeddings = model.encode(texts, batch_size=64, normalize_embeddings=True)

umap_model = umap.UMAP(n_neighbors=15, n_components=8,
                      min_dist=0.0, metric="cosine", random_state=42)
hdb = hdbscan.HDBSCAN(min_cluster_size=8, metric="euclidean",
                     cluster_selection_method="eom")

topic_model = BERTopic(
    embedding_model=model,
    umap_model=umap_model,
    hdbscan_model=hdb,
    ctfidf_model=ClassTfidfTransformer(reduce_frequent_words=True),
    language="multilingual",
    calculate_probabilities=False,
    verbose=True,
)
topics, _ = topic_model.fit_transform(texts, embeddings)
```

Затем для каждого кластера вызываем LLM с промптом «дай короткое имя (3–6 слов) и описание (1–2 предложения) для группы задач 1С по этим примерам».

## 7. Этап 4: финальное управленческое summary

### 7.1 Структура отчёта
1. **Executive summary** (10–15 строк): главные цифры, главные тренды, главный риск.
2. **Распределение типов работ** (таблица: work_type × количество × % × средний complexity).
3. **Топ-частых проблем** (топ-15 кластеров по `issue_type`/`recurring_problem_markers` + примеры task_id).
4. **Трудоёмкие/рискованные категории** (high `complexity_level` × high `risk_level`).
5. **Причины обращений** (агрегация `root_cause_hypothesis` через подкластеризацию).
6. **Что отправить в KB** (задачи с `documentation_needed=true` → темы статей).
7. **Что автоматизировать** (`automation_potential ∈ {medium, high}` → инициативы).
8. **Что менять в процессе** (повторяющиеся обращения по правам, повторяющиеся ошибки после обновлений и т.д.).
9. **Плохо заполненные поля Jira** (метрики из `missing_information`: какие поля чаще пустые).
10. **Команды-аномалии** (по медиане complexity, доле резолвед, доле «unknown»).
11. **Рекомендации руководителю** — 5–10 пунктов, приоритизированных.
12. **План улучшений 30/60/90 дней** — с ответственными и метрикой успеха.

### 7.2 Промпт финального reduce-этапа

```
Ты — старший консультант 1С и руководитель практики сопровождения.
Ниже — агрегированные данные за квартал по N задачам 10+ команд:

[CLUSTERS]: список кластеров с именем, размером, описанием, типичными
            примерами, метриками complexity и risk.
[DIM_BREAKDOWNS]: распределения по issue_type, work_type,
                  affected_1c_area, final_status_or_outcome.
[ANOMALIES]: задачи high/critical risk, не вошедшие в крупные кластеры.
[QUALITY]: метрики заполненности полей Jira и confidence_score.

Сформируй управленческий отчёт по структуре:
1. Executive summary (10–15 строк)
2. Распределение типов работ
3. Частые проблемы (топ-15)
4. Трудоёмкие и рискованные категории
5. Причины обращений
6. Темы для базы знаний
7. Кандидаты на автоматизацию
8. Изменения в процессах
9. Качество заполнения Jira
10. Рекомендации руководителю
11. План 30/60/90 дней

Требования:
- Опирайся ТОЛЬКО на переданные данные. Не выдумывай статистику.
- Все цифры — из агрегированных метрик; в каждом пункте указывай
  размер группы (N=...) и долю.
- Рекомендации — конкретные, измеримые, с ответственным (если
  определимо: «руководитель команды X», «ответственный за обновления»).
- Стиль — деловой, без воды, на русском.
- Формат — Markdown.
```

Финальный отчёт всегда генерируется **сильной моделью** (top-tier reasoning) — это единственный вызов, который реально требует «качество», и он один раз на прогон.

## 8. Архитектура с учётом OpenRouter.ai (актуально на 2026)

### 8.1 Что важно знать про OpenRouter API в 2026

- Единый OpenAI-совместимый эндпоинт `https://openrouter.ai/api/v1`. Всё работает через стандартный OpenAI SDK с заменой `base_url` и `api_key`.
- **Free-тиры (на май 2026)**: модели с суффиксом `:free` имеют лимит 20 запросов/минуту; 50 запросов/день — для аккаунтов без купленных кредитов; 1000 запросов/день — после покупки от $10 кредитов (даже если баланс позже опустился ниже). Эти цифры регулярно меняются, всегда сверяйтесь с `https://openrouter.ai/docs/api/reference/limits` и `/api/v1/key` (GET, чтобы посмотреть лимиты ключа).
- **Pay-as-you-go**: платформенных rate-limit нет, но действуют upstream-лимиты провайдеров. Платформенный fee — порядка 5.5% поверх цены модели.
- **Structured outputs**: поддерживается `response_format: { type: "json_object" }` (свободный JSON) и `response_format: { type: "json_schema", json_schema: {...} }` (строгая схема). Не все модели поддерживают строгую схему — фильтр по флагу на странице моделей.
- **Fallbacks**: параметр `models: [primary, fallback1, fallback2]` или `route: "fallback"` — при ошибке primary автоматически выбирается следующий. При routing/fallback вы платите только за успешный вызов.
- **Plugins**: `web` (поиск), `file-parser` (PDF), `response-healing` (авто-починка JSON), `context-compression` (middle-out компрессия). `response-healing` особенно полезен для JSON-этапа.
- **Reasoning models**: поддержка `reasoning` параметра и поля `reasoning_details` в ответе — нужно сохранять при многошаговых диалогах.
- **`openrouter/free` router**: автоматический выбор случайной free-модели с подходящими фичами (image, tools, structured outputs). Удобно для тестов, не подходит для production (непредсказуемое качество).
- **429 Too Many Requests**: неудачные попытки тоже считаются в дневной лимит. Поэтому ретрай должен быть с экспоненциальной задержкой и проверкой headers.

### 8.2 Распределение моделей по этапам (без указания конкретных «гарантированно бесплатных»)

| Этап | Класс модели | Критерии выбора |
|---|---|---|
| Предварительный summary длинных полей (§13) | дешёвая средняя | поддержка русского, контекст ≥32k, JSON mode |
| Map (анализ одной задачи) | средняя по цене, надёжная по JSON | хорошее следование инструкциям на русском, native structured outputs / function calling |
| Embeddings | open-source, локально (e5/BGE-M3) | качество на русском, длина контекста, лицензия |
| Имена кластеров | дешёвая | короткие выходы, JSON-mode |
| Финальный отчёт (Reduce) | топовая reasoning-модель | длинный контекст ≥128k, сильное reasoning, отличный русский |
| QA-judge | средняя или сильная (другая, чем Map) | независимая «второе мнение» |

### 8.3 Критерии выбора бесплатных/дешёвых моделей
1. **Поддержка русского языка** — проверить на пилоте 30–50 задач (см. §12).
2. **Соблюдение JSON-схемы** — % валидных ответов в пилоте должен быть ≥98%.
3. **Стабильность доступа** — free-модели часто перегружены: смотреть процент 429/5xx по логам последних 24 часов.
4. **Контекст** — минимум 32k для Map, 128k+ для Reduce.
5. **Цена в долларах за 1M токенов** — должна укладываться в бюджет: 2000 задач × ~3000 токенов вход × ~600 токенов выход = ~7.2M токенов/прогон. Считать в обе стороны.
6. **Лицензия данных** — можно ли передавать персональные/корпоративные данные. Для 1С это критично: фамилии, контрагенты, договоры.
7. **Поддержка structured outputs или хотя бы tool/function calling**.

### 8.4 Слабые vs сильные модели
- **Слабые** (дешёвые/free): отлично подходят для Map-этапа на нормально очищенных данных. Но они чаще:
  - срываются в JSON («забывают» закрыть скобку);
  - переводят русский на английский в выводе;
  - галлюцинируют в `root_cause_hypothesis`;
  - выбирают значение enum, которого нет в схеме.
  - **Митигация**: Instructor + retries + `response-healing` + жёсткий enum.
- **Сильные** (top-tier): для Reduce-этапа и QA. Использовать выборочно.

### 8.5 Fallback-паттерн (production)

```python
# pseudo
PRIMARY = "vendor/strong-cheap-model"
SECONDARY = "vendor2/another-cheap"
FINAL_FALLBACK = "vendor3/free-model:free"

response = client.chat.completions.create(
    model=PRIMARY,
    extra_body={
        "models": [PRIMARY, SECONDARY, FINAL_FALLBACK],
        "route": "fallback",
        "provider": {"sort": "throughput", "allow_fallbacks": True},
        "plugins": [{"id": "response-healing"}]
    },
    response_format={"type": "json_schema", "json_schema": TASK_SCHEMA},
    messages=[...],
    temperature=0.1,
)
```

### 8.6 Логирование и retry
- Полный трейс через **Langfuse** (декоратор `@observe`) — промпт, ответ, модель (фактически выбранная, из заголовка `OpenRouter-Provider`), токены, цена, latency.
- Retry: tenacity `retry(stop=stop_after_attempt(3), wait=wait_exponential(min=2, max=30))`, с пропуском на 429 (увеличиваем задержку до 60+ секунд).
- На каждой неудаче: пишем в `quality_checks` причину; через 3 попытки — переводим задачу в статус `failed` для ручного разбора.
- При невалидном JSON: первый retry — Instructor reasking, второй — переключение модели в fallback-цепочке, третий — ручная очередь.

### 8.7 Сравнение моделей на тестовой выборке
- Зафиксировать «золотой набор» из 30–50 задач, размеченных вручную (с правильными `issue_type`, `work_type`, `affected_1c_area`, `risk_level`).
- Прогнать через каждую кандидатную модель.
- Метрики: % валидного JSON; accuracy enum-полей; пересечение списков `tags` (Jaccard); семантическое сходство `short_summary` с эталоном (cosine на e5-large); среднее `confidence_score`; цена; latency; % 429.
- Запоминать `prompt_version` — менялся промпт ⇒ перепрогон бенчмарка.

## 9. Альтернативы OpenRouter.ai

### 9.1 Локальные open-source модели (Ollama / LM Studio / vLLM)

**Когда выбирать**: чувствительные данные 1С (контрагенты, ФОТ, договоры), требование «не уходит в облако», стабильная нагрузка.

**Модели для русского (state of the art на 2026, выбирать по критериям, не по имени):**
- **T-pro 2.0** (T-Tech, на базе Qwen3, Apache-2.0, оптимизированный под русский, hybrid reasoning) — топ-выбор для сильных задач на русском.
- **T-Lite** — лёгкий вариант той же линейки для Map-этапа.
- **Qwen3-семейство** (от 8B до MoE 235B-A22B) — отличный мультиязычный, в т.ч. русский; широкая экосистема.
- **Vikhr / ruadapt / Saiga** — русские файнтюны Llama/Mistral; зрелые, но к 2026 уступают свежим Qwen3-based решениям.
- **GigaChat (Сбер)** — закрытая, но через API; хороший русский; рассматривать как российский cloud-аналог OpenRouter.
- **YandexGPT** — аналогично, через Yandex Cloud Foundation Models.

**Критерии выбора локальной модели:**
1. Качество на пилоте 30–50 задач (см. §12).
2. Размер vs железо (см. таблицу ниже).
3. Лицензия (Apache-2.0 / MIT — лучше всего для коммерческого использования).
4. Длина контекста (минимум 32k для Map, 128k для Reduce).
5. Поддержка структурированного вывода (function calling / JSON mode), либо хотя бы хорошо работает с Outlines/XGrammar для constrained generation.

**Ориентир по железу (4-bit quant):**

| Размер модели | GPU VRAM | Use case |
|---|---|---|
| 7–8B | 8–12 GB (RTX 3060/4060) | Map, embeddings, тестирование |
| 13–14B | 16–24 GB (RTX 4090, A4000) | Map в продакшене |
| 30–34B | 24–48 GB (RTX 6000 Ada / 2×4090) | Сильный Map / средний Reduce |
| 70B+ | 80 GB (A100/H100) или 2×48 GB | Reduce / финальный отчёт |
| MoE (Qwen3-235B-A22B) | 80–140 GB | Только серверы / шеринг |

**Плюсы**: данные не покидают периметр; нет лимитов; единая стоимость владения; отсутствует зависимость от провайдера. **Минусы**: капекс, MLOps, обновление моделей, нужен SRE.

### 9.2 Hugging Face Inference Providers / Inference Endpoints
- Удобно для serverless-инференса open-source моделей без своего железа.
- Биллинг по токенам / часам.
- Поддержка multi-provider routing (как у OpenRouter, но с уклоном в HF-экосистему).
- Хорошо для embeddings (multilingual-e5, BGE-M3), которые можно выставить как endpoint.

### 9.3 «Бесплатные» API
- Free-тиры Google AI Studio (Gemini), Cerebras, Groq и др. — лимиты 1–60 RPM. Подходят для пилотов, не для квартальных прогонов 3000 задач без пагинации.
- Не закладывать в SLA.

### 9.4 Гибрид (рекомендуемый подход для 1С-сопровождения)
- **Локально** (vLLM/Ollama, T-Lite/Qwen3-14B): Map-этап (анализ 1 задачи), embeddings, имена кластеров, предварительные summary длинных полей.
- **Облако** (OpenRouter с топовой моделью): Reduce-этап (финальный отчёт), QA-judge, спорные случаи.
- Преимущество: минимизация уходящих данных (в облако идут уже агрегированные кластерные сводки, без сырых ФИО/контрагентов), при этом качество финального отчёта остаётся высоким.

### 9.5 Embeddings + vector DB (без LLM)
Для базовой кластеризации и поиска дублей задач достаточно embeddings без LLM:
- multilingual-e5-large / BGE-M3 → Qdrant / pgvector / Weaviate.
- Дубли, похожие задачи, semantic search по архиву.
- Минусы: не даёт `short_summary`, `root_cause_hypothesis`, `recommendations` — только группировку.

### 9.6 Классические NLP-методы (как baseline и фильтры)
- Регулярные выражения для извлечения версий конфигураций («БП 3.0.156»), кодов ошибок, имён метаданных (`Документ\.[А-ЯA-Z]\w+`).
- TF-IDF + KMeans/LDA как baseline для сравнения с BERTopic.
- Rule-based классификатор issue_type по ключевым словам («ошибка», «не проводится», «доработать», «новая роль») — даёт ~60% точности и стабильно работает без LLM; можно использовать как pre-filter и сравнение с LLM-предсказанием (если расходятся — флаг для ревью).
- Плюсы: быстро, бесплатно, объяснимо. Минусы: не понимает контекста.

| Подход | Стоимость | Скорость | Приватность | Качество | Сложность ops |
|---|---|---|---|---|---|
| OpenRouter (paid) | $$ | высокая | средняя (TOS) | высокое | низкая |
| OpenRouter (free) | $0 | средняя | средняя | нестабильное | низкая, но 429 |
| Локально (Ollama/vLLM) | капекс | средняя | максимальная | от среднего до высокого | высокая |

### 10.3 Ключевые модули
- `ingestion/` — Jira REST client (с pagination, retries, since-watermark).
- `cleaning/` — нормализация HTML/Wiki, дедуп, словари 1С-синонимов.
- `chunking/` — `RecursiveCharacterTextSplitter` для длинных полей.
- `llm_gateway/` — единая точка вызова LLM (OpenRouter/local), с retry, schema, healing.
- `analysis/` — Map-воркер (одна задача → JSON).
- `embeddings/` — батчевая генерация и upsert в pgvector.
- `clustering/` — BERTopic-пайплайн по разрезам.
- `reduce/` — иерархическая агрегация и финальный отчёт.
- `qa/` — валидаторы, LLM-judge, метрики.
- `api/` — FastAPI endpoints для UI и интеграций.
- `prompts/` — версионирование промптов (Git, prompt_version в БД).

### 10.4 Мониторинг качества
- Langfuse: traces, scores, evals datasets.
- Дашборд KPIs: % валидного JSON, среднее `confidence_score`, доля задач с `missing_information`, среднее latency, $ на прогон.
- Алерты: drop валидности JSON > 5 п.п. за час → переключиться на fallback.

### 10.5 Повторная обработка
- Эндпоинт `POST /reprocess` с фильтрами (`team`, `cluster_id`, `confidence_score < 0.5`, `date_range`).
- Каждая задача пересчитывается только на тех этапах, что выбраны (Map / cluster / reduce).
- Результаты не перезаписываются «in place», а добавляются как новая версия (`analysis_version`, `analyzed_at`); UI показывает diff между версиями.

## 12. Контроль качества AI-анализа

### 12.1 Многоуровневая валидация
1. **JSON-уровень**: парсинг через `json.loads`; на ошибке — `response-healing` plugin или Instructor reasking.
2. **Schema-уровень**: Pydantic-модель + Instructor `max_retries=2`. Невалидные enum → отбрасываются.
3. **Полнота полей**: обязательные поля заполнены; для `null`-полей — есть запись в `missing_information`.
4. **Cross-field consistency** (свои валидаторы):
   - `final_status_or_outcome = "resolved"` ⇒ `len(employee_actions) >= 1` ИЛИ `confidence_score < 0.7`.
   - `documentation_needed = true` ⇒ `issue_type != "consultation"` либо явная пометка.
   - `automation_potential = "high"` ⇒ `work_type ∈ {regulatory, data_load, rights_setup, report_dev}` (мягкая проверка).
5. **Source faithfulness**: для каждой записи `employee_actions` проверяем, что её ключевые слова встречаются в `worklog_compressed + comments_compressed` (recall-эвристика). Если 0% слов из действия — флаг галлюцинации.

### 12.2 Confidence score (формула)
Рекомендуемая композитная формула, которую модель должна оценивать сама + post-correction:

```
confidence = 0.4 * field_evidence_score
           + 0.3 * (1 - missing_information_ratio)
           + 0.2 * cross_field_consistency
           + 0.1 * outcome_clarity
```

Где:
- `field_evidence_score` — доля выходных полей, у которых LLM смог сослаться на конкретный фрагмент входа (можно требовать в промпте: «приведи 3-5 слов из исходника как опору»).
- `missing_information_ratio` — `len(missing_information) / 5`.
- `cross_field_consistency` — пройдены ли валидаторы выше.
- `outcome_clarity` — `final_status_or_outcome != "unknown"`.

После расчёта overrideит самооценку модели, если она расходится > 0.2.

### 12.3 LLM-as-judge на случайной выборке
- Случайные 5–10% задач прогоняются через **другую** модель с промптом «оцени по шкале 1–5 точность каждого поля относительно исходного текста».
- Записываем в `quality_checks` с `check_type='judge'`.
- Если средний score кластера < 3.5 — кластер уходит в ручной ревью.

### 12.4 Ручная разметка золотого набора
- 30–50 задач, размеченных аналитиком с экспертизой 1С.
- Прогон любой новой версии промпта/модели → метрики на golden set:
  - exact match для enum (issue_type, work_type, final_status_or_outcome): целевое ≥85%.
  - Jaccard для `affected_1c_area` и `tags`: целевое ≥0.6.
  - Семантическое сходство `short_summary` (cosine, e5-large): ≥0.75.
  - Полнота `employee_actions`: F1 по ключевым шагам ≥0.7.
- Без прохождения порога — релиз новой версии запрещён.

### 12.5 Перепрогон спорных задач
- Очередь `quality_checks.passed = false OR confidence_score < 0.5`.
- Прогон на сильной модели + изменённом (более строгом) промпте.
- Diff в UI: до/после.

### 12.6 Метрики панели качества
- **JSON validity rate** (за прогон, скользящее окно).
- **Mean confidence** по командам.
- **Hallucination rate** (% задач с `source faithfulness < 0.3`).
- **Coverage of mandatory fields** (% непустых).
- **Stability**: при перепрогоне той же задачи той же моделью с temperature=0.1 — % полей, которые меняются.
- **Cost per analyzed task** ($/task).
- **Latency p50/p95**.

## 13. Обработка длинных текстов (журналы работ, комментарии)

### 13.1 Очистка служебных сообщений (детерминированно, до LLM)
- Удалить системные комментарии Jira Automation, e-mail handler, smart commits.
- Удалить «status changed», «assignee changed», pure-watch-сообщения.
- Удалить quoted-блоки (строки начинающиеся с `>`, или включённые в `[quote]…[/quote]`).
- Удалить подписи (heuristic: 2+ строки в конце, начинающиеся с «С уважением», «Best regards», «—» и т.п.).
- Удалить ссылки на скриншоты без описания (regex на attachment-URL без контекста).

### 13.2 Дедупликация
- Нормализация (lowercase, без пунктуации) → SimHash или MinHash.
- Cosine на embeddings внутри переписки (e5-small для скорости): дубль — cosine ≥ 0.92.
- Сохранить только первое уникальное вхождение, отметить число повторов.

### 13.3 Выделение содержательных комментариев
Эвристики, ранжирующие комментарий по «содержательности»:
- Длина ≥40 символов;
- Наличие ≥1 термина из словаря 1С (имена объектов, коды ошибок, версии, имена обработок);
- Наличие глаголов действия («перенастроил», «перепровёл», «обновил», «откатил», «скорректировал»);
- Не вопрос (не заканчивается на `?`, не начинается на «А как», «Подскажите»).

Каждый комментарий получает `salience_score`. Для LLM оставляем top-N по этому score, пока укладываемся в бюджет токенов.

### 13.4 Группировка по датам и авторам
- Сортировка по дате; объединение в «эпизоды» по разрывам ≥48 часов.
- Внутри эпизода: «диалог» как блок «{author, date}: текст».
- В промпт идут эпизоды, а не отдельные комментарии — так модель лучше восстанавливает развитие задачи.

### 13.5 Иерархическая суммаризация (Map-Reduce внутри одного поля)

```
1. Если len(worklog_compressed) <= 4k токенов → передаём как есть.
2. Иначе:
   a. Разбить RecursiveCharacterTextSplitter (chunk_size≈1500, overlap=150)
      по разделителям ["\n\n", "\n", ". "]
   b. Для каждого chunk-а — дешёвая модель с промптом:
       "Извлеки из фрагмента журнала работ по задаче 1С только:
        (1) выполненные действия сотрудников, (2) обнаруженные причины,
        (3) промежуточные результаты, (4) итог если есть.
        Без воды. Верни маркированный список на русском."
   c. Объединить summaries → если снова > 4k → ещё один уровень reduce.
   d. Финальный summary → подставить в Map-промпт основной задачи.
3. Кешировать summary по hash(chunk) — экономит при перепрогонах.
```

### 13.6 Извлечение действий и финального результата
- На уровне Map-промпта основной задачи (§4) явно требуем `employee_actions` и `final_status_or_outcome`.
- При наличии нескольких эпизодов: модель должна указать, кто из них «закрывающий» (последний по дате с резолюцией). Это снижает риск, что промежуточный workaround будет принят за итог.

### 13.7 Лимиты токенов
- Бюджет на одну задачу в Map: ~3–5k токенов вход (для cheap-моделей с 32–128k контекстом — с большим запасом).
- Если после всех summary всё ещё превышает — обрезаем самые слабые эпизоды (по `salience_score`), а не «по середине».
- В `missing_information` явно отмечаем «обрезано N эпизодов комментариев».

   - топ-кластеров с drill-down до конкретных задач;
   - индикаторы качества AI-анализа (mean confidence, JSON validity);
   - визуализация в темной dashboard-эстетике (TailwindCSS `slate-900` + акценты `cyan-400`/`amber-400`) — для финального инфографика.
10. **Технический артефакт**: dataset.parquet всех `ai_task_analysis` за период — для повторного использования в BI/ML.

## 15. End-to-end пример: 500 задач одной команды за квартал

Команда «Сопровождение 1С Бухгалтерия», период Q1 2026.

### Шаг 1. Выгрузка из Jira
JQL: `project = SUP1C AND component = "БП 3.0" AND updated >= 2026-01-01 AND updated < 2026-04-01`. Через REST `search` с `expand=changelog,renderedFields` и пагинацией по 100. Результат: 500 issues → `jira_tasks_raw`. Время: 2–4 минуты.

### Шаг 2. Очистка
- Снимаем Wiki-разметку Jira (`{noformat}`, `{code}`, `[~user]`).
- Из 500 задач: у 78 пустое поле «Цель», у 142 пустое «Текущее поведение», у 29 — описание короче 30 символов. Все эти факты пойдут в общий отчёт о качестве заполнения Jira.
- Удалили 1814 служебных комментариев Jira Automation, ~3200 quoted-reply, ~890 подписей.
- Дубликатов комментариев (cosine ≥ 0.92): 412 удалено.

Пример очистки одной задачи (`SUP1C-12345`):
- Worklog был 14 800 символов → после очистки 4 200 → после summary 1 100 символов.
- Комментариев было 23 → содержательных 7 → после группировки в 3 эпизода.

### Шаг 3. Анализ Map (LLM)
- Бюджет: 500 задач × ~3500 токенов вход × ~600 токенов выход = ~2.05M токенов.
- Модель: средняя дешёвая модель с надёжным русским и structured outputs. Параллельность: 10 одновременных async-запросов через Celery.
- Время прогона: ~25–40 минут (зависит от лимитов провайдера).
- Невалидных JSON на первом проходе: 6 (1.2%). После reasking — 1 (0.2%) ушёл в ручную очередь.
- Средний `confidence_score`: 0.74. Задач с `confidence < 0.5`: 38 (7.6%) — кандидаты на перепрогон сильной моделью.

Пример вывода (см. §3) — задача `SUP1C-12345` об ошибке проведения после обновления БП 3.0.

### Шаг 4. Embeddings и кластеризация
- BGE-M3 на GPU: 500 векторов за ~6 секунд.
- BERTopic по разрезу `issue_type + current_behavior`: 14 кластеров + 47 «шумовых».
- Топ-3 кластера команды:
  - `cluster_07` «Ошибки проведения после обновления БП 3.0» — 31 задача (6.2%), avg_risk 0.78, avg_complexity 0.62.

## Резюме архитектурных решений

1. **Map-Reduce, а не один большой запрос**: одна задача = один LLM-вызов; агрегация — отдельный этап через embeddings + clustering + LLM-reduce.
2. **Контракт через Pydantic + Instructor**: единственный надёжный способ получить структурированный JSON от любой модели, включая дешёвые.
3. **OpenRouter — для гибкости и fallback, но рассмотрите гибрид**: локальные модели семейств T-pro/Qwen3 для Map, облако только для финального отчёта и QA.
4. **Embeddings — open-source и локально**: multilingual-e5-large или BGE-M3 (русский файнтюн USER-bge-m3 — для максимума на 1С-терминологии).
5. **BERTopic как готовое решение для multidimensional clustering**: модульный (можно подменить любой блок).
6. **Postgres + pgvector** покрывает 95% потребностей до объёмов в миллионы задач.
7. **Celery + Redis + Prefect** — стандарт для прогона очередей с DAG-оркестрацией.
8. **Langfuse self-hosted** для трейсинга — данные не уходят наружу.
9. **Контроль качества — обязательный этап, не опция**: golden set, LLM-judge, source faithfulness, перепрогон спорных.
10. **Защита редких сигналов**: anomaly buckets и stratified sampling важнее красивого «среднего» — именно critical-задачи дают руководителю практическую ценность отчёта.

Эта архитектура масштабируется от 500 задач (одна команда, один прогон) до 10000+ задач (10+ команд, мульти-квартал) без принципиальной переработки — меняются только лимиты параллельности воркеров и набор моделей.
