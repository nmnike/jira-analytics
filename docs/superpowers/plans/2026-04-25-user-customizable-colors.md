# План: пользовательские настройки цветов

**Статус:** в задумке (не запланирована к разработке).
**Создано:** 2026-04-25 (после правки QA-цвета).

## Зачем

Сейчас цветовая палитра прибита гвоздями в двух местах:
- роли (`roles.color`) — редактируется в БД, но без UI;
- семантические токены (`DARK_THEME.amber` для перегруза, `#f5222d` для дефицита,
  `CATEGORY_COLORS.*`, `CHART_COLORS.*`) — хардкод в `frontend/src/utils/constants.ts`.

Когда меняется одно (например, цвет роли), оно может случайно совпасть с
другим (так QA-оранжевый совпал с amber-перегрузом → потребовалась миграция 033).
Пользователь не может сам разрулить — приходится править код и катить релиз.

## Что сделать

UI «Настройки → Внешний вид» с полем выбора цвета (HEX-picker) для каждого:

1. **Роли** (analyst, dev, qa, consultant, RP, …) — уже редактируется через
   `roles.color`; не хватает фронта.
2. **Семантические индикаторы** — перенести из `constants.ts` в новую таблицу
   `app_settings` (или `theme_tokens`) с ключами:
   - `theme_overload_color` (сейчас `#f5a524`)
   - `theme_deficit_color` (сейчас `#f5222d`)
   - `theme_success_color` (сейчас `#1D9E75`)
   - `theme_warning_color` (сейчас `#f5c842`)
3. **Категории работ** (`CATEGORY_COLORS`) — на отдельной вкладке или вместе с
   справочником категорий (`/categories`).

## Контракт «не дать выстрелить себе в ногу»

При сохранении цвета — backend проверяет:
- цвет не совпадает (по recpdistance / Δ Lab < threshold) с уже занятыми
  семантическими токенами и активными ролями;
- 422 с человеческим сообщением: «Цвет слишком близко к перегрузу — выбери
  другой» + предлагает 2-3 свободных альтернативы из дефолтной палитры.

## Объём (грубо)

- Frontend: страница «Внешний вид» с превью (~1 день).
