# Спек: Иерархический отчёт «Аналитика»

**Дата:** 2026-05-01
**Статус:** утверждён к реализации

## Контекст

Текущая страница `/analytics` — пять плоских вкладок (по сотрудникам / проектам / категориям / периодам / переключения контекста). Не покрывает реальные сценарии разбора часов: невозможно от виджета на дашборде перейти к подробной декомпозиции часов сотрудника, нельзя одновременно увидеть структуру работы команды по уровням «роль → сотрудник → вид работ → категория → задачи».

Свежий эпизод: бэкенд виджета «Нормированные работы» теряет 9 ч у Шутова Сергей за апрель 2026 — выяснилось вручную через прямые запросы в БД. Спек 1 (`docs/superpowers/specs/2026-05-01-orphan-worklogs-row-design.md`) починил конкретно эту течь, но для регулярного анализа таких потерь нужен полноценный отчёт.

## Цель

Заменить страницу `/analytics` на единый иерархический отчёт со связкой с виджетами дашборда. Пользователь должен иметь возможность:

1. Видеть структуру часов **Команда → Роль → Сотрудник → Вид работ → Категория → Задача → (опц.) Ворклоги**.
2. Кликнуть в виджете дашборда на любую строку/плитку и попасть в отчёт с уже применёнными фильтрами.
3. Фильтровать дополнительно по сотруднику, поиску задачи, видам работ, категориям.
4. Настраивать набор видимых колонок персонально под себя.
5. Выгружать текущий вид в XLSX.

## Архитектура

### Layout (вариант C — master-detail)

- **Слева:** список команд с переключателем «Все команды» (агрегированный режим).
- **Справа:** иерархическая таблица с раскрытием уровней.
- **Над таблицей:** строка фильтров (Сотрудник / Поиск задачи / Вид работ / Категория) + кнопки [Настройка столбцов] / [Экспорт XLSX] / переключатель [Ворклоги: inline / drawer].

### Глобальный пикер периода в шапке сервиса

Помимо уже существующего глобального фильтра команд, в шапку добавляется второй контрол: пикер квартала + месяца. Состояние сохраняется в профиле пользователя (по аналогии с `User.selected_teams`) — поле `selected_period` (`year`, `quarter`, `month`).

Действует на все страницы, где есть понятие периода (Дашборд, Аналитика, Capacity, Backlog, Planning). Локальные пикеры на этих страницах сохраняются как override (например, на Аналитике — диапазон дат поверх глобального периода).

### Бэкенд

**Новый эндпоинт:** `GET /api/v1/analytics/report`

Параметры:
- `year`, `quarter` (обязательные)
- `month` (опц.)
- `start_date`, `end_date` (опц., локальный override)
- `teams` (CSV, опц.)
- `employee_id` (опц.)
- `task_query` (опц.) — substring по `Issue.key` и `Issue.summary` (case-insensitive)
- `work_type_codes` (CSV, опц.)
- `category_codes` (CSV, опц.)

Ответ — единое дерево:

```typescript
{
  teams: AnalyticsTeamNode[],
  grand_totals: NodeTotals,
}

AnalyticsTeamNode {
  team: string | null,            // null = "Без команды"
  totals: NodeTotals,
  roles: AnalyticsRoleNode[],
}

AnalyticsRoleNode {
  role_code: string | null,
  role_label: string,
  role_color: string,
  totals: NodeTotals,
  employees: AnalyticsEmployeeNode[],
}

AnalyticsEmployeeNode {
  employee_id: string,
  name: string,
  initials: string,
  totals: NodeTotals,
  work_types: AnalyticsWorkTypeNode[],
}

AnalyticsWorkTypeNode {
  work_type_id: string,           // включая виртуальный "__unmapped__"
  label: string,
  totals: NodeTotals,
  categories: AnalyticsCategoryNode[],
}

AnalyticsCategoryNode {
  category_code: string | null,
  label: string,
  color: string,
  totals: NodeTotals,
  issues: AnalyticsIssueNode[],
}

AnalyticsIssueNode {
  id: string,
  key: string,
  summary: string,
  status: string,
  status_category: string,        // 'done' | 'indeterminate' | 'new' | 'cancel'
  issue_type: string,
  category: string | null,        // унаследованная
  last_worklog_at: datetime | null,
  assignee_name: string | null,
  totals: NodeTotals,
}

NodeTotals {
  fact_hours: float,
  plan_hours: float | null,       // null на уровнях, где план не определён
  pct_plan: float | null,         // null если plan_hours = null или 0
  pct_total: float,
  worklog_count: int,
  issue_count: int,
  employee_count: int,
  avg_worklog_minutes: float,
}
```

Реализация: `AnalyticsService.get_hierarchical_report` в `app/services/analytics_service.py`. Один SQL-агрегат `worklogs JOIN issues JOIN employees + employee_teams + categories + mandatory_work_types`, разворачиваемый в Python-дерево. Использует общий orphan-bucket из Спека 1.

**Второй эндпоинт:** `GET /api/v1/analytics/report/issue/{issue_id}/worklogs?start=..&end=..`

Возвращает плоский список ворклогов задачи за период:
```typescript
[
  {
    worklog_id: string,
    started_at: datetime,
    hours: float,
    employee_name: string,
    comment: string | null,
  }
]
```

- `AnalyticsColumnSettings` — модалка/popover для настройки видимости колонок
- `AnalyticsWorklogsExpand` — inline-блок с ворклогами под строкой задачи

### Drill-down с виджетов

URL-контракт: `/analytics?employee=<id>&work_type=<code>&category=<code>&task=<key>`. Период и команды подхватываются из глобальных шапочных фильтров.

Точки клика:
- **Виджет «Нормированные работы»:**
  - Клик по строке вида работ внутри блока сотрудника → `?employee=<id>&work_type=<code>`
  - Клик по итогу сотрудника → `?employee=<id>`
  - Клик по строке «Не указана категория/вид работ» → `?employee=<id>&work_type=__unmapped__`
- **Виджет «Ворклоги по категориям задач»:**
  - Клик по плитке категории → `?category=<code>`
  - Клик по карточке сотрудника в блоке «Последний ворклог» → `?employee=<id>`

### Настройка видимости колонок

Каждый пользователь хранит свой набор видимых колонок. Поле `User.analytics_columns: JSON` (или отдельная таблица `user_settings(user_id, key, value)` если будем расширять — на этом этапе кладём в `User` для простоты).

API: `GET/PUT /api/v1/users/me/analytics-settings`.

Дефолтный набор — все колонки видимы.

### Колонки

Универсальные на всех уровнях группировки:
1. Σ часов факт
2. Σ часов план *(только где определён — Сотрудник × Вид работ)*
3. % факт/план
4. % от итога отчёта
5. Кол-во ворклогов
6. Кол-во задач *(на листе-задаче = 1)*
7. Кол-во сотрудников
8. Средняя длительность ворклога (мин)

На уровне задачи добавляются:
9. Ключ + ссылка на Jira (всегда видим)
10. Заголовок задачи (всегда видим)
11. Статус с цветом (как в Категориях задач)
12. Тип задачи
13. Категория (унаследованная)
14. Дата последнего ворклога
15. Исполнитель

Поля 9-10 нельзя скрыть. Остальные 13 — настраиваемые.

На уровне ворклога (внутри задачи):
- Дата+время начала
- Длительность (часы)
- Сотрудник
- Комментарий

(ворклог-уровень — фиксированный набор, без настройки видимости)

### Поведение

- **Сортировка:** дефолт — внутри каждого уровня по убыванию `fact_hours`. Заголовки числовых колонок кликабельны для пересортировки внутри уровня.
- **Развёрнутость по умолчанию:** все Команды свёрнуты, кроме команды текущего пользователя. В master-detail режиме при загрузке выбран команда пользователя (если есть). Иначе — «Все команды».
- **Поиск задачи:** substring по key и summary, case-insensitive.
- **Пустое состояние:** «Нет данных за выбранный период».
- **Виртуализация таблицы:** включается при > 200 видимых строк.

### Экспорт XLSX

`GET /api/v1/analytics/report/export.xlsx?<те же params + columns=...>`

Структура файла — плоская: каждая строка задачи + колонки Команда / Роль / Сотрудник / Вид работ / Категория + значения видимых числовых колонок + базовые задача-поля. Без иерархических строк-итогов; пользователь делает свои фильтры/сводные в Excel.

## Out of scope (планы на потом)

Записываем сюда, чтобы не потерять; реализация — в отдельных спеках:

1. **Серверная пагинация при «Все команды»** — если объём при агрегированном режиме станет тяжёлым (> 10000 задач), добавить пагинацию верхних уровней.
2. **Сохранённые пресеты фильтров пользователя** — «Мои часто используемые наборы фильтров» с возможностью именования и быстрого выбора.
3. **Графики/диаграммы поверх таблицы** — pie / bar / timeline в дополнение к табличному виду.
4. **Перетаскиваемая иерархия** — позволить пользователю менять порядок группировки (например, поставить Категория выше Сотрудника). Сейчас иерархия фиксирована.
5. **Дополнительный поиск задачи** — расширить до поиска по комментариям ворклогов и тегам/целям.
6. **Drill-up из задачи в отчёт** — на странице задачи (если такая появится) — кнопка «Показать все ворклоги по этой задаче в Аналитике».
7. **Автоматическая привязка категорий без вида работ** — после стабилизации Спека 1 решить, надо ли автоматически связывать «Архив» / «Незаполненные» с конкретным видом работ через настройки.
8. **Перепривязка категории «Технический долг»** — сейчас привязана к виду работ «Сопровождение и консультация», а не «Технический долг». Это правка справочника, не кода — отдельная задача.
9. **Экспорт XLSX с иерархическими подитогами** — если пользователи захотят форматирование с группировкой как в отчёте, а не плоско.

## Acceptance

- Страница `/analytics` показывает master-detail layout с командами слева и иерархической таблицей справа.
- В шапке появился глобальный пикер периода (квартал+месяц), сохраняемый в профиле пользователя.
- На Аналитике есть локальный пикер «Уточнить даты».
- 4 фильтра над таблицей (Сотрудник / Поиск задачи / Вид работ / Категория) применяются на сервере.
- Иерархия Команда → Роль → Сотрудник → Вид работ → Категория → Задача раскрывается, на каждом уровне числа агрегируются.
- Клик по задаче разворачивает её ворклоги inline или в Drawer (по выбору пользователя).
- Все 5 точек drill-down с виджетов открывают Аналитику с правильными фильтрами в URL (3 в NormWork, 2 в CategoryWidget).
- Настройка видимости колонок сохраняется per-user.
- Кнопка «Экспорт» отдаёт XLSX с применёнными фильтрами и видимыми колонками.
- Старая Аналитика (5 вкладок) полностью удалена.
- Тесты: интеграционные на `/analytics/report` (проверка структуры дерева, фильтров, агрегатов), на ленивые ворклоги, на drill-down фильтры.
