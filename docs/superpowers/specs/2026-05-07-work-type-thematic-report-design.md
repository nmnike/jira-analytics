# Спек: Тематический отчёт по виду работ

**Дата:** 2026-05-07
**Статус:** утверждён к реализации
**Маршрут:** `/analytics/work-type-report`

## Контекст

PM-у нужно регулярно понимать, на что уходят часы внутри одного вида работ — особенно «Сопровождение и консультация», где сейчас видны только агрегаты «480 ч за месяц» без декомпозиции. Существующие отчёты дают разрезы по сотрудникам/проектам/категориям, но не видят *тематику внутри* (например, что 32% сопровождения уходит на ошибки обмена с 1С, а 21% — на проводки).

Текущий `/analytics` — иерархическая таблица *Команда → Роль → Сотрудник → Вид работ → Категория → Задача*. Темы — это новый уровень *внутри вида работ*, который нельзя получить из категорий: они слишком общие.

Цель — дать PM-у инструмент *тематического разбора одного вида работ* за период, с AI-нарративом «что именно делали» и привязкой каждой задачи к теме. Аудитория — PM при подготовке отчёта руководству; экспортируемые форматы — XLSX (для дальнейшей сводки) и PDF (для презентации).

Полный лог брейнсторма с обоснованиями: 8 ключевых решений Q1–Q8 (см. секцию «Зафиксированные решения»).

## Зафиксированные решения

| ID | Вопрос | Выбор |
|---|---|---|
| Q1 | Откуда берутся темы | **E.** AI предлагает на первом запуске → пользователь курирует → словарь живёт сам. Дальше как гибрид «словарь + Другое» |
| Q2 | К чему привязан словарь | **A.** Один словарь на вид работ (общий для всех команд) |
| Q3 | Когда работает LLM | **A.** On-demand с кэшем по ключу `(work_type, период, набор команд)`. Кнопка «Пересчитать» |
| Q4 | LLM-пайплайн | **B.** Map-Reduce: фаза 1 — разметка задач per-issue с кэшем; фаза 2 — синтез на агрегированных findings |
| Q5 | Гибкость группировки | **C.** Пресеты + drag-and-drop pivot + сохранение «★ Мой layout» per-user |
| Q6 | Что пишет AI | **C.** Заголовок + per-theme нарратив + топ-выбросы + рекомендация. Сравнение периодов — ◇ |
| Q7 | Форматы выгрузки | **D.** UI + XLSX + PDF |
| Q8 | Скоп видов работ | **A.** Один вид работ за прогон |

## Что увидит пользователь

### Главный экран

Раскладка сверху вниз:

1. **Toolbar** — выбор вида работ (radio с одним активным), выбор команд (multi-select поверх глобального фильтра), выбор периода (год/квартал/месяц с локальным override относительно глобального), бейдж свежести снимка («● снапшот свежий, 2 ч назад» / «снимок мог устареть»), кнопки `↻ Пересчитать`, `⤓ XLSX`, `⤓ PDF для руководства`.

2. **AI-заголовок** — карточка с левой бирюзовой полоской: 1 предложение от AI («Команды Платформа+BI за апрель — 540 ч сопровождения; основное — обмены 1С (32%) и проводки (21%); пиковая нагрузка в неделю w15 после релиза 02.04»), под ней метаданные (модель, объём данных, версия промпта, бейдж уверенности).

3. **KPI-карточки** — 4 шт. по горизонтали: «Всего часов», «Тем», «Задач», «Сотрудников». Каждая показывает значение, дельту vs прошлый период (когда есть второй снапшот) и sparkline (◇).

4. **Левая колонка** (≈70% ширины):
   - **Распределение по темам** — donut слева (5 фокусных сегментов + «Другое»), горизонтальные progress-bars справа со списком тем и долями.
   - **Иерархия** — раскрывающаяся таблица. Сверху — строка пресетов (По темам / По сотрудникам / По командам / По ролям / По проектам / ★ сохранённые) и pivot-чипы текущей раскладки (drag-and-drop). Внутри — дерево с группировками. Под каждой темой 1–2 строки AI-нарратива курсивом с кликабельными ссылками на задачи. У новых тем — жёлтый бейдж «★ новая».

5. **Правая колонка** (≈30% ширины):
   - **Топ-выбросы** — 3–5 карточек: ключ задачи, заголовок, причина выброса (бейдж «съела 86 ч» / «переоткрыта 4×» / «7 разных правок»), метаданные (число сотрудников, ворклогов, дней в работе).
   - **Рекомендация** — жёлтая outlined-карточка с одним действием и ожидаемым эффектом.
   - **vs прошлый период** (◇) — компактный блок с дельтами по часам/задачам и списками «Растут / Падают / Новое».
   - **Панель кандидатов** — лента «★ N кандидатов в словарь». Клик — выдвижная панель с действиями.

### Drill-down

- **Клик по теме** в дереве → раскрытие до сотрудников и задач.
- **Клик по задаче** → выдвижная панель: заголовок, поля «Цель задачи» / «Описание текущего поведения» / «Описание», список ворклог-комментов, классификация AI (тема, уверенность, объяснение), ссылка «Открыть в Jira».
- **Клик по сотруднику в нарративе** → подсветка его задач во всех темах.
- **Клик по ключу задачи в AI-тексте** → переход к ней в дереве + раскрытие.

### Drawer управления словарём

Шестерёнка возле списка тем → выдвижная панель:
- Вкладка **«Активные темы»** — таблица: имя, описание, цвет, использовалось в N задачах, дата создания. Клик → редактирование. Кнопка «Архивировать». Кнопка «Слить в…» с выпадающим списком.
- Вкладка **«Архивные»** — для восстановления.
- Вкладка **«Кандидаты»** — общая накопительная очередь предложений из всех построенных отчётов.

Любое изменение словаря (создание / переименование / архивирование / слияние / принятие кандидата) поднимает версию словаря этого вида работ → инвалидирует кэш классификации затронутых задач.

### Empty-state (первый запуск)

При открытии страницы с пустым словарём — приглашение в полный экран:

> Словарь тем для **«Сопровождение и консультация»** ещё пустой.
> Постройте первый отчёт — AI предложит темы, вы их подтвердите.

PM выбирает период + команды → жмёт **«Построить первый отчёт»**. AI работает в режиме чистой авто-кластеризации (без словаря). Результат показывается с **всеми темами помеченными как кандидаты** (жёлтый бейдж). Справа — панель «Подтвердите словарь». После принятия хотя бы одной темы она становится постоянной.

### Сценарий низкой уверенности

Если в теме <10 задач или средняя уверенность AI <60% — на шапке темы жёлтый бейдж «низкая уверенность». Нарратив генерируется, но PM видит сигнал.

### Сценарий отказа AI

Задачи, на которых AI упал (rate limit / схема не валидна / faithfulness не прошёл) попадают в специальный блок **«Требуют ручной разметки»** внизу дерева. PM может вручную выбрать тему из словаря или нажать «Повторить» — пересчитывается только этот сегмент.

### Кросс-навигация

- На **Dashboard** в виджете «Распределение по видам работ» — клик по строке *«Сопровождение и консультация»* открывает этот отчёт с уже выставленными командой/периодом из глобального фильтра.
- В существующей **Аналитике** в строке вида работ — кнопка `Тематический разбор →`.

### Сохранённые раскладки

Перетаскивание pivot-чипов + выбор колонок → кнопка **★ Сохранить как…** → диалог имени → раскладка появляется в строке пресетов с золотой звездой. Опция «По умолчанию для меня» — открывается при следующем заходе.

## Архитектура

### Высокоуровневый поток данных

```
Issue + Worklog (БД) ─┐
                      ├─→ map (per-issue) ─→ findings JSON ─→ reduce ─→ Snapshot ─→ UI / XLSX / PDF
ThemeDictionary ──────┘
```

LLM работает только на findings JSON. Сырые поля задач и комменты ворклогов уходят только в Map-фазу (там они нужны для классификации). Reduce-фаза не видит сырого — только агрегаты.

### Map-фаза (разметка задач)

Для каждой задачи периода — отдельный мини-промпт:
- **Вход:** заголовок, тип, статус, поля «Цель задачи» / «Описание текущего поведения» / «Описание», конкатенированные комментарии всех её ворклогов за период; текущий словарь тем (имя + описание + примеры из 2–3 уже размеченных задач).
- **Выход:** ID темы из словаря **или** `null` + предложенное имя (если ни одна тема не подходит); 1 строка «что делали»; уверенность 0–1; (◇) тег природы — `systemic` / `one_off` / `consultation` / `new_feature`.

Кэш per-issue:
- Ключ кэша = `sha256(заголовок + цель + поведение + описание + конкатенация ворклог-комментов)`.
- Версия словаря отдельно.
- При повторном построении: задача с прежним хэшем и прежней версией словаря — берётся из кэша. Все изменения — переразмечаем.

Параллелизация: до 5 запросов одновременно (учитывая лимиты бесплатных моделей).

### Reduce-фаза (синтез)

Один промпт, на вход — структурированный findings JSON:
```jsonc
{
  "totals": {"hours": 540, "tasks": 78, "employees": 12},
  "themes": [
    {"id": "...", "name": "Ошибки обмена с 1С", "hours": 173, "pct": 32,
     "tasks_count": 18, "employees_count": 5,
     "top_tasks": [
       {"key": "PROJ-321", "summary": "...", "hours": 86, "contribution": "ручной разбор расхождений по контрагентам"},
       {"key": "PROJ-345", "summary": "...", "hours": 38, "contribution": "..."}
     ]}
  ],
  "outliers": [
    {"key": "PROJ-321", "reason": "high_hours", "value": 86, "context": "5 сотрудников, 23 ворклога, 18 дней"}
  ]
}
```

Выход — структурированный JSON по схеме:
```jsonc
{
  "headline": "≤180 chars",
  "themes_narratives": [
    {"theme_id": "...", "narrative": "≤2 предложения, упоминать ключи задач", "evidence_keys": ["PROJ-321","PROJ-345"]}
  ],
  "outliers_explanations": [
    {"key": "PROJ-321", "explanation": "одно предложение почему она в выбросах"}
  ],
  "recommendation": {"text": "одно действие", "expected_impact": "оценка эффекта"}
}
```

System prompt запрещает:
- Сравнения конкретных людей.
- Числа, которых нет в findings.
- Упоминание задач, которых нет в findings.

### Faithfulness-валидатор

После reduce — проверка:
- Каждое число в `headline` / `themes_narratives` / `outliers_explanations` присутствует в findings (с допуском ±10% для округлений).
- Каждый ключ задачи (`PROJ-XXX`) в текстах есть в findings.
- В текстах нет ФИО (regex по списку сотрудников из findings).

Если проверка не прошла — одна попытка регенерации с прицепом «previous output failed validation: <причина>». Если повторно — fallback-нарратив из шаблона: «AI summary unavailable. Раскладка по темам ниже».

### Снапшот

Сохраняется по ключу *(вид работ × период × хэш набора команд)*. Хранит:
- Полную структуру отчёта (totals, themes с нарративами, outliers, recommendation).
- Версию словаря на момент построения.
- ID модели + версию промпта.
- Дату и автора построения.

Свежесть: снапшот считается свежим если *(а)* версия словаря не поднималась с момента построения, *(б)* ни одна задача из периода не правилась после построения, *(в)* не появилось новых задач в периоде.

### Детект выбросов (без AI)

Каждая тема набирает кандидатов в выбросы по формальным правилам:
1. Часы задачи >85-го процентиля по теме.
2. Задача переоткрывалась ≥3 раза (transitions из Done → не-Done).
3. >5 разных людей логали часы в задачу за период.
4. Задача в работе >14 дней без перехода в Done.

Топ-5 кандидатов передаются в Reduce-фазу — AI пишет одно предложение «почему».

## Что сервис помнит между запусками

Без технических деталей структуры — в человеческих терминах:

1. **Словарь тем** на каждый вид работ — общий список (имя, описание, цвет, статус «активная/архивная»). Курируется через UI.
2. **Карточка темы для каждой задачи** — какую тему AI ей назначил, с какой уверенностью, какой моделью, какая формулировка «что делали». Не пересчитывается, пока задача и словарь не менялись.
3. **Готовый снимок отчёта** — полный результат под уникальную связку «вид работ + период + набор команд». Открывается мгновенно.
4. **Личные раскладки** — сохранённые pivot-конфигурации per-user.

Технические детали (имена таблиц, FK, индексы, JSON-схемы хранения) — в плане реализации.

## Скоп

### MVP (★)

**Бэкенд:**
- Хранилище словаря тем + CRUD-эндпоинты + слияние тем.
- Map-фаза с кэшем (по содержимому задачи + версии словаря) + параллелизация.
- Reduce-фаза с faithfulness-валидатором + fallback на шаблон.
- Снапшоты отчёта с детектом свежести.
- Детект выбросов без AI.
- Бейджи низкой уверенности.
- Блок «требуют ручной разметки» при сбое AI + ручная разметка.

**Фронт:**
- Страница `/analytics/work-type-report` + маршрут.
- Toolbar с выбором вида работ / команд / периода + бейдж свежести + кнопки экспорта.
- AI-заголовок с метаданными.
- 4 KPI-карточки (без sparkline-трендов в MVP).
- Donut + горизонтальные бары распределения.
- Иерархическая таблица с пресетами + drag-чипы pivot + сохранение «★ Мой layout».
- AI-нарратив под каждой темой + кликабельные ссылки.
- Drawer drill-down на задачу.
- Топ-выбросы (правая панель).
- Карточка рекомендации.
- Панель кандидатов + действия (принять / слить / игнорировать / переименовать).
- Drawer управления словарём (3 вкладки).
- Empty-state «словарь пуст».
- Кросс-навигация: с Dashboard и существующей Analytics.
- XLSX-выгрузка (3 листа: Темы / Задачи / Текст).
- PDF-выгрузка через `window.print()` с печатной раскладкой.

### Вторая итерация (◇)

- Sparkline-тренды в KPI и под темами (требуют ≥2 снапшота — естественно набирается).
- Блок «vs прошлый период»: дельты по темам, что выросло/упало, что появилось.
- Тег природы инцидента (systemic / one_off / consultation / new_feature) в Map-фазе + дополнительный разрез в UI.
- Кросс-тематическая корреляция в нарративе (AI находит общую первопричину между темами).
- Стоимость в рублях (часы × ставка роли) если ставки заведены.
- Stacked area chart истории тем за 6 месяцев.
- Заметки PM к темам (свободный текст, сохраняется per-snapshot).
- **Локальная модель как опция в настройках LLM** (Ollama / LM Studio через OpenAI-совместимый API). Гибрид: Map-фаза локально, Reduce-фаза в облаке.

### На будущее (·)

- SLA per-тема (целевое время + % выходов).
- Burn-rate / линейный прогноз.
- Heatmap «день недели × час».
- Разрез по reporter-у задач.
- Кнопка «Создать задачу-улучшение» из рекомендации (Jira deep link).
- Подписки на расписание (cron-снапшоты).
- Сравнение нескольких видов работ side-by-side.

## Защита от галлюцинаций

1. **AI не считает числа** — все агрегаты деривируются из БД до вызова Reduce.
2. **AI получает только findings JSON** в Reduce — нет доступа к сырым полям и комментам.
3. **Faithfulness-валидатор** проверяет каждое число и ключ задачи.
4. **Запрет ФИО** в нарративах (regex по списку сотрудников).
5. **Бейдж низкой уверенности** при <10 задач в теме или средней уверенности <60%.
6. **Каждое числовое утверждение в UI кликабельно** — drill-down к источнику (задаче).

## Визуализация — палитра решений

| Секция | Тип визуала | Обоснование |
|---|---|---|
| AI-заголовок | Карточка с акцентной полоской + текст | Триада «значение + контекст + AI-почему» |
| KPI | Карточки с числом / дельтой / sparkline | Каноник executive readability |
| Распределение тем | Donut (5+ «Другое») + horizontal bars | Donut ≤5 сегментов, остальное в прогресс-барах |
| Иерархия | Раскрывающееся дерево с inline-нарративом | Reuse паттерна существующей `/analytics` |
| Тренды | Sparkline (KPI) + stacked area (◇ вторая итерация) | Тренд > значение |
| Выбросы | Список карточек с бейджами причин | Action-oriented |
| Рекомендация | Жёлтая outlined-карточка | McKinsey one-pager pattern |

Анти-паттерны (не используем):
- Pie-chart с >5 сегментами.
- 3D-графики, gauges.
- Velocity-сравнения людей.
- Голые числа без AI-объяснения.
- Burndown.
- Radar / spider chart.

## Риски и митигации

| Риск | Митигация |
|---|---|
| Качество классификации на коротких/пустых описаниях задач | В Map-фазу даём не только описания, но и комментарии всех ворклогов — часто богаче самой задачи |
| Лимиты бесплатных моделей при первом массовом построении | Параллелизация ≤5; OpenRouter $10 поднимает лимит DeepSeek до 1000/день; Gemini как primary с лимитом 1500/день |
| Раздувание словаря — AI предлагает похожие имена для одного и того же | При показе кандидата проверка семантической близости с существующими темами (через embeddings) → мягкое предупреждение «похоже на тему X» |
| Faithfulness-валидатор отвергает валидный нарратив с округлениями | Допуск ±10% на числа; форсим в промпте использование точных чисел из findings |
| Пользователь забивает на курирование словаря, кандидаты копятся | Бейдж «★ N кандидатов» в правой колонке отчёта + накопительная вкладка в drawer-е словаря |
| Сбой AI на отдельных задачах ломает построение | Задачи с ошибкой попадают в «Требуют ручной разметки» — отчёт строится без них; кнопка «Повторить только эти» |
| Снимок устарел незаметно для пользователя | Бейдж «снимок мог устареть» возле даты + один клик «Пересчитать» |

| Map-фаза (разметка + кэш + воркер + параллелизация) | 2–3 |
| Reduce-фаза (синтез + faithfulness-валидатор + fallback) + снапшоты | 2 |
| Главная раскладка + дерево + donut + drill-down | 3 |
| Pivot + сейв layouts + панель кандидатов | 2 |
| Выбросы + рекомендация + бейджи + ручная разметка | 1–2 |
| XLSX + PDF + кросс-навигация | 2 |
| E2E-тесты + полировка | 1–2 |

**Итого: 15–19 рабочих дней.** Через subagent-флоу с двухстадийным ревью на main → ~2.5 недели.

## Открытые вопросы — на этап реализации

- Точный список колонок XLSX-выгрузки.
- Конкретный промпт Map-фазы (черновик есть в спеке Reduce-формата выше; финал на этапе реализации с проверкой на реальных данных).
- Чёрный список слов в нарративе (не только ФИО — также «команда работает усердно», «процесс улучшается» — пустые клише из research-документа).
- UI-решение для предупреждения о семантически похожих кандидатах (◇ вторая итерация, не блокирует MVP).

## Приложение: контекст из research-документов

В `TZ/Analitika_soprovod/` — два глубоких research-документа про executive engineering analytics. Использованные оттуда принципы:

- **30-секундная читаемость + AI-headline + триада тренда** — главная компоновка экрана.
- **Findings JSON + faithfulness validator** — защита от галлюцинаций.
- **Не сравнивать людей** в нарративах.
- **Бейдж низкой уверенности** при малом количестве данных.
- **Рекомендация = одно действие + ожидаемый эффект** в формате McKinsey one-pager.
- **Donut ≤5 сегментов**, остальное — horizontal bars.
- **Кросс-навигация** между виджетами и отчётами вместо дублирования.

Не использованы (вне скопа): DORA-метрики, Monte Carlo прогноз, CFD, sprint predictability, dependency graph, control charts, SLA-burnup. Это классический executive engineering health dashboard — отдельный продукт. Тематический отчёт по виду работ — узкая, прицельная задача для PM-а одного направления.
