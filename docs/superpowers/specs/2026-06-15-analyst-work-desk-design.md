# Рабочие столы аналитиков — дизайн

Дата: 2026-06-15
Статус: утверждён к реализации

## Цель

Персональная страница-монитор для аналитика, доступная по прямой ссылке без авторизации
(внутри корп-сети, чувствительность данных минимальна). Руководитель проектов (РМ) создаёт
стол под сотрудника своей команды, выбирает галочками набор виджетов, раздаёт ссылку.
РМ может отозвать или перевыпустить ссылку.

## Решения (из брейншторма)

- Доступ: `/desk/<токен>` — без логина, длинный случайный токен.
- Создаёт РМ — только для сотрудников своих команд (по `selected_teams` / membership).
- Набор виджетов настраиваемый — РМ отмечает галочками (Вариант B).
- Данные живые — страница читает текущие значения по токену, авто-обновление.
- Отзыв = два действия: «Отозвать» (ссылка → 404) и «Перевыпустить» (новый токен, старый мёртв).
- Один активный стол на сотрудника; пересоздание разрешено.

## Модель данных

Новая таблица **work_desk** (UUID, стандартные timestamps):

| Поле | Назначение |
|---|---|
| `id` | UUID PK |
| `employee_id` | FK → employee, чей это стол |
| `token` | случайная строка (≥32 симв., url-safe), уникальна, индекс |
| `enabled_widgets` | JSON-список ключей включённых виджетов |
| `created_by_user_id` | FK → user (РМ, который создал) |
| `revoked_at` | null = активна; not-null = отозвана |
| `last_viewed_at` | для виджета «что изменилось» |
| `created_at` / `updated_at` | стандартные |

Инвариант: один активный (`revoked_at IS NULL`) стол на `employee_id`. Перевыпуск отзывает
старый и создаёт новый.

Каталог виджетов — константа в коде (12 ключей), не таблица. `enabled_widgets` хранит
подмножество ключей.

## Доступ по токену (безопасность)

- Новая зависимость `get_desk_by_token(token)` — резолвит стол по токену, 404 если не найден
  или `revoked_at` заполнен. JWT/cookie не требуется.
- Эндпоинты стола НЕ используют `get_current_user`. Это единственная публичная (no-auth)
  поверхность в сервисе — изолирована в отдельном роутере `desk_public`.
- Токен — единственный секрет. Длинный, случайный (`secrets.token_urlsafe(32)`).
- Данные не уходят за периметр (корп-сеть), но в payload отдаём только то, что нужно
  виджету — без e-mail, без чужих зарплатных полей и т.п.

## API

**Публичные (по токену), роутер `desk_public`, без авторизации:**

- `GET /desk/{token}` — мета стола: имя/аватар сотрудника, команды, список включённых
  виджетов, период (текущий квартал). Обновляет `last_viewed_at` (но возвращает diff
  относительно предыдущего значения — см. виджет 12).
- `GET /desk/{token}/widget/{key}` — данные одного виджета. Лениво, по одному, чтобы
  страница грузилась по частям и обновлялась независимо.

Каждый виджет внутри переиспользует существующий сервис (см. таблицу ниже), передавая
`employee_id` и команды стола. Никакой новой бизнес-логики расчётов — только адаптеры.

**Управление (РМ), роутер `work_desks`, `Depends(get_current_user)`:**

- `GET /work-desks` — список столов сотрудников моих команд (статус, ссылка, виджеты).
- `POST /work-desks` — создать стол `{employee_id, enabled_widgets}`. Проверка: сотрудник
  в команде текущего пользователя. Если активный стол уже есть — 409 (или перевыпуск).
- `PATCH /work-desks/{id}` — изменить набор виджетов.
- `POST /work-desks/{id}/revoke` — отозвать.
- `POST /work-desks/{id}/regenerate` — перевыпустить токен (отзыв старого + новый).

Скоупинг по команде — как в остальных эндпоинтах: фильтр по `selected_teams` пользователя
через membership сотрудника.

## Каталог виджетов → переиспользуемые сервисы

| Ключ | Виджет | Источник |
|---|---|---|
| `my_tasks` | Мои задачи + плановые даты | ResourcePlanningService.get_gantt → ResourcePlanAssignment по employee_id |
| `weekly_load` | Загрузка план/факт по неделям | CapacityService (quarter capacity, разбивка по неделям) |
| `my_conflicts` | Конфликты планировщика по мне | ConflictAggregator / PlanConflict по employee_id |
| `hours_balance` | Баланс часов (переработки) | HoursBalanceService.compute_employee |
| `unlogged_days` | Незалогированное время | HoursBalanceService (DayCalc kind='skip') |
| `category_breakdown` | Факт по категориям/видам работ | HoursBreakdownService / AnalyticsService по employee_id |
| `team_absences` | Отпуска/отсутствия коллег | CapacityService / Absence по командам стола |
| `team_availability` | Доступность команды на неделе | ResourcePlanningService gantt + ScheduledBlock |
| `production_calendar` | Произв. календарь, остаток дней квартала | ProductionCalendarService |
| `quarter_deadlines` | Дедлайны квартала | BacklogService / BacklogItem.due по командам |
| `external_help` | «Помощь извне» (часы в чужие проекты vs свои) | ExecutiveDashboardService cross-team / alien-hours |
| `recent_changes` | Что изменилось в планах с прошлого визита | ResourcePlanAssignment.updated_at > desk.last_viewed_at |

Виджет 12 (`recent_changes`) — лёгкая версия: сравниваем `updated_at` закреплений сотрудника
с сохранённым `last_viewed_at`. Без отдельного снапшот-движка.

## Фронтенд

- Новый роут `/desk/:token` в `routes.tsx` — **вне** `AppLayout` и без `ProtectedRoute`
  (нет шапки/навигации/логина). Standalone-страница `DeskPage`.
- `DeskPage` грузит мету (`GET /desk/{token}`), затем рендерит включённые виджеты, каждый
  тянет свои данные лениво. Авто-обновление по таймеру (живые данные).
- Переиспользуем существующие dashboard-компоненты где возможно (NormWorkWidget и др.),
  адаптируя под single-employee режим. Где компонент завязан на auth/team-filter — делаем
  лёгкую desk-обёртку, принимающую данные пропсами.
- Тёмная/светлая Aurora-тема как на остальном сайте; читаемо с телефона.

**Управление РМ:** новая вкладка «Рабочие столы» на `/capacity` (рядом с TeamTab) —
таблица сотрудников команды: статус стола, копировать ссылку, чекбоксы виджетов (модалка),
«Отозвать» / «Перевыпустить».

## Тестирование

- Backend: токен-доступ (валидный/отозванный/несуществующий → 404), скоупинг по команде
  (нельзя создать стол чужому сотруднику), CRUD управления, перевыпуск отзывает старый.
- Каждый виджет-адаптер: возвращает данные по employee_id (smoke на seed-данных).
- E2E: открыть `/desk/<token>` без логина → виджеты рендерятся; отозвать → 404.

## Вне скоупа (v1)

- Конструктор drag&drop (Вариант C).
- Снапшот-движок истории изменений.
- Несколько столов на сотрудника одновременно.
- Самостоятельное создание столов аналитиком.
