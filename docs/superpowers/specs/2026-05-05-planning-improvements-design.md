# Улучшения календарного планирования — дизайн-док

**Дата:** 2026-05-05
**Контекст:** PyJobShop solver выдаёт «странную» раскидку. PM хочет несколько независимых улучшений в модель данных и солвер, чтобы план соответствовал реальности команды.

## Бизнес-цели

1. PM может прямо в реестре проектов выставлять приоритет задачи (1—10). Чем меньше число, тем раньше задача должна выполниться.
2. Задачи на Gantt группируются по исполнителю проекта из утверждённого сценария (✅ уже сделано в этой сессии).
3. ОПЭ-фаза раскладывается на две параллельные работы: одна на исполнителе с ролью аналитика, вторая на разработчике. Сейчас они идут последовательно.
4. Большие задачи декомпозируются: длинная фаза анализа делится на куски, чтобы разработка могла стартовать раньше после первого куска. Решение о декомпозиции принимает солвер только если без неё не уложиться в квартал.
5. Если на проекте задействовано несколько тестировщиков (или аналитиков, или разработчиков) одновременно — учитывать это: фаза заканчивается за меньшее время.
6. Из Jira уже синхронизируются 4 поля «вовлечённости по фазам» и 4 поля «длительности фазы в днях». Они должны учитываться в плане. Доля 0.6 значит, что сотрудник занят на этой задаче 60% дня и может одновременно вести другие задачи.

## Скоуп

В рамках 5 фаз. Каждая независима, ничего не ломает.

| Фаза | Что | Оценка |
|---|---|---|
| 1 | Редактируемый приоритет задачи | ½ дня |
| 2 | ОПЭ параллельно (двe одновременных строки) | ½ дня |
| 3 | Вовлечённость + длительности из Jira → BacklogItem → solver | 1.5 дня |
| 4 | Параллельные исполнители одной роли (на проекте + на задаче) | 1 день |
| 5 | Авто-split длинных фаз для соблюдения квартала | 1.5 дня |

## Ключевые решения

### Фаза 1
- Приоритет хранится на задаче, общий для всех сценариев. Поле в реестре уже есть — нужно сделать редактируемым.
- Диапазон: 1—10 (1 — высший). Пусто — задача в хвосте по умолчанию (что уже работает после F4).
- Редактирование: inline в строке backlog'а в /planning, без модалки.

### Фаза 2
- При построении плана из сценария ОПЭ-задача создаёт две строки одной фазы: одна на сотрудника с ролью аналитика, вторая на разработчика. Обе одновременно стартуют после конца тестирования (FS-зависимость к QA).
- Часы делятся согласно `opo_analyst_ratio` (как сейчас).
- Solver: в `PHASE_ROLE_MATCH` для ОПЭ нужны два разных набора ролей — для строки-аналитика и строки-разработчика. Различаем по `assignee_employee_id` назначенного человека или по новому `phase_subtype`.

### Фаза 3
- Поля `involvement_*` (4 штуки) и `duration_*_days` (4 штуки) уже в модели Issue и синхронизируются. Нужно: пробросить в BacklogItem, использовать в солвере.
- Solver-модель меняется: длительность задачи = `duration_days * 8` слотов, demand сотрудника = `involvement * 8` единиц/день. Capacity сотрудника остаётся 8.
- Это позволяет одному сотруднику одновременно вести 2+ задач (если сумма долей ≤ 1.0).
- Fallback: если из Jira пришёл NULL — `involvement = 1.0`, `duration_days = estimate_hours / 8`.

### Фаза 4
- Поле «количество исполнителей роли» на 3 фазы (analyst/dev/qa) — на уровне проекта (по умолчанию для всех задач) и переопределение на задаче.
- Solver: при N>1 фаза занимает в N раз меньше календарного времени или подключаются N сотрудников одной роли одновременно. Простейший вариант — масштабировать `duration / N`.

### Фаза 5
- Авто-split — эвристика, не CP-SAT optional split. Перед построением модели:
  1. Считаем суммарную длительность всех фаз / capacity квартала.
  2. Если больше 1.0 → начинаем дробить самые длинные фазы пополам, пока не уложимся или не дойдём до минимального куска (например 8ч).
  3. Каждый кусок становится отдельной задачей в солвере; FS-зависимость следующей фазы — на первый кусок.
- Видимо для пользователя: на Gantt фаза показывается несколькими блоками с пометкой «1 из N».

## Out of scope

- Возврат F5/F6/F7 в солвере (двойная цель, lag_days, warm start) — отдельной задачей после стабилизации SIGSEGV.
- Альтернативные солверы (Timefold, CPOptimizer).
- UI настройки порога авто-split.

## Приёмка

После всех 5 фаз PM может:
- Перетащить приоритет проекта вверх в реестре, нажать «Оптимизировать», увидеть что приоритетный проект подвинулся раньше по календарю.
- Включить в сценарий проект с длинной фазой анализа — солвер сам разделит её, чтобы дев-фаза стартовала раньше.
- Указать на задаче 2 тестировщика — фаза тестирования сократится в 2 раза.
- Изменить вовлечённость аналитика на 0.5 в Jira, увидеть что в плане задача растягивается на больше календарных дней.
- Видеть в Gantt задачи сгруппированными по исполнителю проекта.
- Видеть ОПЭ как две параллельные строки.
