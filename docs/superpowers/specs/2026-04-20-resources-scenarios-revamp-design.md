# Дизайн: Пересборка разделов «Ресурсы» и «Сценарии» (Этап 1+2)

**Дата:** 2026-04-20
**Статус:** Утверждён, готов к плану реализации
**Скоуп:** Этапы 1 и 2 из планирующей дорожной карты (см. `memory/project_planning_roadmap.md`). Этап 3 (календарное планирование инициатив в стиле Gantt) — отдельная эпопея, здесь только закладка модели.

---

## Контекст

На странице «Сценарии» ресурс считается по всему штату (>370 сотрудников), а не по команде, под которую строится сценарий. Правила обязательных работ живут на отдельной вкладке «Ресурсы / Правила» и негибко применяются (строго 100%-распределение времени). Клик по идее в списке бэклога пересчитывает правую панель через три последовательных сетевых запроса и занимает 3–5 секунд. Тестировщик для команды PM — внешний ресурс с фиксированным месячным/квартальным лимитом, а не сумма часов сотрудников из базы. Роль «Консультант» нужна как отдельный ресурсный пул, но в текущей модели роли зашиты в коде (Аналитик / Программист / Тестировщик / Другое).

Вкладка «Распределение» на странице «Ресурсы» — плоская таблица категорий ворклогов. PM её не использует: фактический анализ идёт через дашборды и аналитику.

## Цели

1. Сценарии считают ресурс по одной привязанной команде — не по всему штату.
2. Клик по идее в бэклоге — мгновенный, без сетевых запросов на пересчёт.
3. Правила обязательных работ редактируются прямо на сценарии, каждый сценарий имеет свой набор.
4. Виды работ получают переключатель «вычитать из доступных часов»: явно отделяем продуктивные виды от тех, что уменьшают пул.
5. Справочник ролей становится редактируемым; добавляется роль «Консультант».
6. Тестировщик — ручной ввод лимита на сценарий (перекрывает подсчёт по сотрудникам).
7. Вкладка «Распределение» удаляется целиком.
8. Под Этап 3 закладывается посуточная база ресурса и поле вовлечённости у идей сценария.

## Не-цели

- Календарное планирование инициатив, диаграмма Ганта, автораскладка дат — отдельный этап.
- Многоролевая работа сотрудника в явном виде (совмещение ролей регулируется Правилами, см. `memory/project_role_model_decision.md`).
- Переработка бэклога и колонок оценки часов по ролям — остаются как есть.
- Интеграция с Jira-полями для новых сущностей (роли, правила сценария хранятся только локально).

## Модель данных

### Реестр ролей (новая сущность)

Таблица ролей, заменяет захардкоженный список в коде и фронтовых константах. Поля:

- Код роли (идентификатор)
- Название (отображается пользователю)
- Цвет (для графиков и бейджей)
- Порядок сортировки
- Активна / скрыта
- Участвует в планировании ресурса (флаг)

Начальное наполнение (миграция из текущего состояния):

| Код | Название | Участвует в планировании |
|---|---|---|
| analyst | Аналитик | да |
| dev | Программист | да |
| qa | Тестировщик | да |
| consultant | Консультант | да |
| other | Другое | нет |

Существующая связь сотрудника с ролью остаётся (одна роль на сотрудника), но теперь ссылается на запись в справочнике.

### Виды работ — переключатель вычета

На сущности «Вид работ» добавляется флаг «вычитается из доступных часов при планировании».

- Включён — процент, назначенный этому виду работ в правилах сценария, вычитается из пула часов сотрудника.
- Выключен — вид работ есть в справочнике (для группировки факта), но в планировании ресурса не участвует.

При миграции: включается у всех видов работ, у которых сейчас нет ни одной связанной категории (то есть тех, что в текущей модели считаются «обязательными»).

### Правила обязательных работ — на сценарий

Вместо общих правил на квартал — у каждого сценария свой набор правил. Правило = (роль, вид работ, процент). Один сценарий содержит много правил.

При создании сценария правила копируются из общего шаблона квартала (чтобы PM не настраивал каждый раз с нуля). После создания правила сценария живут отдельно.

### Сценарий — новые поля

- Команда (привязка к одной из команд). Обязательное поле при создании.
- Ручной лимит часов Тестировщика на квартал для внешнего ресурса (опционально). Если заполнен — перекрывает подсчёт по сотрудникам с ролью Тестировщик.

### Идея сценария — закладка под Gantt

Добавляется опциональный коэффициент вовлечённости (0..1). UI в этом этапе не подключается; поле резервируется, чтобы на Этапе 3 не тянуть миграцию в середине работы.

### Что уходит

- Вкладка «Распределение» на странице Ресурсы и связанный эндпоинт — удаляются.
- Общие правила обязательных работ на квартал (их роль сводится к «шаблону для новых сценариев»).
- Вкладка «Правила» на странице Ресурсы — удаляется.

## Поведение и потоки данных

### Открытие сценария

1. Пользователь выбирает команду и квартал, открывает (создаёт) сценарий.
2. Сервер возвращает «базу ресурса команды» — по каждому сотруднику команды список дней квартала и сколько часов в каждый день он доступен (с учётом производственного календаря, отпусков и правил обязательных работ этого сценария).
3. Сервер также возвращает: суммарные часы по ролям, список идей с их спросом по ролям, правила сценария, ручной лимит Тестировщика.
4. Фронт кладёт всё это в память и рендерит правую панель.

Посуточная база — это закладка под Gantt. В текущей реализации правая панель показывает только суммы, но данные уже готовы для диаграммы дат.

### Клик по идее (включение/выключение)

1. Пользователь кликает на строку идеи в бэклоге.
2. Фронт локально переключает состояние включённости, моментально пересчитывает суммы спроса и заливку прогресс-баров, обновляет правую панель. Всё синхронно.
3. В фоне уходит сохранение на сервер; предыдущий незавершённый запрос отменяется. UI не блокируется, ошибки откатывают состояние с уведомлением.

Пересчёт базы ресурса команды не нужен — она зависит только от состава команды, квартала, отпусков и правил, а не от включённости идей.

### Изменение правил сценария

При правке правил в правой панели база пересчитывается: запрос уходит на сервер, возвращается обновлённая база, правая панель перерисовывается. Ожидаемое время запроса — до 500 мс, это редкое действие.

### Пересчёт часов на «Команде»

Кнопка «Пересчитать часы» на вкладке «Команда» страницы Ресурсы обновляет базу часов сотрудников команды с учётом производственного календаря и отпусков за выбранный период. После нажатия — обновлённые цифры в таблице.

### Роль Консультант в карточке ресурса

В карточке «Ресурс по ролям» (бывшая «Ёмкость по ролям») строка Консультанта показывает только доступные часы. Спроса нет (идеи бэклога не имеют колонки «часы консультанта»), бар заполнен на 0%. Строка подписана «информативно».

### Ручной лимит Тестировщика

В правой панели сценария поле «Часы тестировщика на квартал». Если заполнено — в карточке ресурса строка Тестировщика показывает введённое число как доступность (вместо суммы по сотрудникам с этой ролью). Если не заполнено — используется сумма из базы.

## UI

### Страница «Сценарии»

- В шапке — выбор команды (обязательное поле) и выбор квартала.
- Левая колонка — список идей бэклога (как сейчас), клик по любой части строки переключает включённость.
- Правая колонка — панель ресурса:
  - «Ресурс команды · Q{n}» (переименовано с «Ёмкость»).
  - «Ресурс по ролям» — бары по каждой роли с флагом «участвует в планировании», включая Консультанта (информативно).
  - «Расчёт ресурсов» — разложение: общий пул − отпуска − обязательные работы = доступно.
  - «По сотрудникам» — список сотрудников команды с их часами.
  - «Правила обязательных работ» — свёрнутая панель с таблицей (роль × вид работ × %), режим редактирования inline.
  - «Часы тестировщика (внешний ресурс)» — поле ввода.

### Страница «Ресурсы»

Остаются три вкладки:

- **Команда** — таблица сотрудников с фактом и планом; добавляется кнопка «Пересчитать часы».
- **Отсутствия** — без изменений.
- **Роли** (новая) — CRUD по реестру ролей: название, цвет, порядок, переключатель «участвует в планировании».

Удаляются вкладки «Распределение» и «Правила».

### Страница «Настройки» / виды работ

На существующей форме редактирования вида работ добавляется переключатель «вычитается из доступных часов при планировании».

## Миграция данных

1. Создаётся справочник ролей, заполняется текущими четырьмя значениями + Консультант.
2. У всех существующих сценариев формируется свой набор правил — копия общих правил квартала на момент миграции. После миграции общие правила квартала продолжают существовать только как шаблон для новых сценариев.
3. У видов работ, у которых сейчас нет ни одной связанной категории, включается флаг «вычитается из доступных часов».
4. У существующих сценариев поле команды заполняется пустым значением. PM при открытии старого сценария должен выбрать команду вручную; до выбора правая панель показывает предупреждение «команда не выбрана».
5. Ручной лимит Тестировщика — пустой у всех существующих сценариев.
6. Вкладка «Распределение» удаляется вместе со своим эндпоинтом.

---

## Обработка ошибок и граничные случаи

- **Сценарий без команды** (легаси после миграции): правая панель показывает заглушку «Выберите команду», кнопки редактирования правил и тумблеры недоступны.
- **Команда без сотрудников**: база ресурса нулевая, все бары пустые, подпись «В команде нет сотрудников».
- **Правило с процентом больше 100**: валидация на сохранении, ошибка «Сумма процентов не должна превышать 100%». Допускается сумма < 100.
- **Ручной лимит Тестировщика = 0**: считается явно заданным, в бар идёт 0 ч, в «По сотрудникам» Тестировщики не показываются для доступности.
- **Откат галочки при сетевой ошибке**: если сохранение включённости идеи упало — состояние на фронте откатывается, показывается уведомление об ошибке.
- **Удаление роли, используемой сотрудниками**: запрет на удаление, предупреждение «Роль используется N сотрудниками». Можно скрыть (is_active=false) — тогда она не появляется в выпадающих списках для новых сотрудников.

## Тестирование

### Юнит-тесты (бэк)

- Расчёт базы ресурса команды: правильно учитывает календарь, отпуска, правила сценария; корректно обрабатывает частичные периоды отпусков; сотрудник без роли игнорируется в ролевом разрезе.
- Правила сценария: копируются из шаблона при создании, независимы от шаблона после.
- Переключатель вычета у вида работ: включён → % уходит из пула; выключен → не влияет.
- Ручной лимит Тестировщика перекрывает подсчёт по сотрудникам.
- Справочник ролей: запрет удаления используемой роли, мягкое скрытие через is_active.

### Интеграционные тесты (API)

- Открытие сценария возвращает базу, правила, список идей единым запросом.
- Переключение включённости идеи не перечитывает базу.
- Изменение правила сценария триггерит пересчёт базы.

### Фронт / E2E

- Клик по любой части строки идеи переключает галочку и мгновенно обновляет правую панель (замер отзывчивости).
- Сетевая ошибка при сохранении галочки — состояние откатывается.
- Смена команды перечитывает базу; смена квартала перечитывает базу.
- Вкладка «Распределение» отсутствует; попытка открыть её URL редиректит на «Команду».
- Кнопка «Пересчитать часы» на «Команде» обновляет цифры без полной перезагрузки страницы.
- Новая вкладка/страница с реестром ролей: добавить, переименовать, скрыть.

## Совместимость и риски

- **Старые сценарии теряют правила обязательных работ**: нет — миграция копирует общие правила квартала в каждый существующий сценарий.
- **Фронт и бэк расходятся по списку ролей**: минимизируется тем, что справочник живёт на бэке и фронт запрашивает его при старте. Временное решение до первого запроса — фоллбэк на зашитый список для не-блокирующего рендера.
- **Удаление вкладки «Распределение»**: удаляется также API-эндпоинт и связанная схема. Тесты, которые его используют, тоже удаляются (не отключать — чистый снос).
- **Переход посуточной базы**: база возвращается посуточно, но в текущем UI агрегируется в суммы. Объём данных — ~90 дней × N сотрудников команды. Для команды в 30 человек это ~2700 точек, допустимо. Если окажется тяжело — группируем по неделям на бэке с возможностью отдать по дням при включении Gantt.

## Открытые вопросы

На момент утверждения дизайна открытых вопросов нет. Все развилки закрыты в ходе брейншторма:

- Одна команда на сценарий — да.
- Правила на сценарий, не на квартал — да.
- ОПЭ остаётся как часовой тип с распределением между Аналитиком и Программистом (не роль).
- Консультант — отдельная роль, без колонки оценки в идеях.
- Сотрудник — одна роль, совмещение через Правила.
- «Распределение» — удалить.
- Скорость клика — фронтовый пересчёт поверх посерверной базы.
