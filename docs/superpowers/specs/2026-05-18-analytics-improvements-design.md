# Доработки раздела «Аналитика»

Дата: 2026-05-18
Статус: утверждён, к реализации

## Контекст

Иерархический отчёт `/analytics` сейчас имеет жёсткую группировку (Команда → Роль → Сотрудник → Вид работ → Категория → Задача), колонку «% от итога» (от гранд-тотала) и тег «Чужая» без указания команды-владельца. PM нужны: гибкая группировка per-user, контекстный процент от родителя, оценка объёма чужой работы на агрегированных строках, идентификация команды-владельца чужой задачи, KPI-сводка и визуальная полоса доли в строке.

Существующая логика расчёта `plan_hours` НЕ изменяется — этот вопрос отложен до починки ресурсного планирования.

## Скоуп

### 1. Пользовательская группировка строк

**Что:** свободная иерархия из 6 уровней (team, role, employee, work_type, category, issue) — пользователь сам определяет порядок и видимость. Сохранение per-user.

**Поведение:**
- Drag-n-drop меняет порядок вложенности.
- Любой уровень можно скрыть («глаз»). Скрытый уровень не группирует строки и не отрисовывается в таблице.
- Если у видимого уровня в текущем срезе всего одно значение — авто-сворачивание (всегда раскрыт, экономим один клик).
- Пресеты: «По людям», «По категориям», «По задачам», «По видам работ». Один клик загружает раскладку.
- Пользователь может сохранить текущую раскладку как именованный пресет (хранится у пользователя).

**Уровни (внутренние коды):**
- `team`, `role`, `employee`, `work_type`, `category`, `issue` (последний нельзя скрыть — это листья дерева; точнее, его можно скрыть только если перед ним скрыты все остальные — иначе бессмысленно. Решение: `issue` всегда виден как последний уровень, скрыть нельзя.)

Уточнение: уровень `issue` всегда виден как лист, скрывается только при крайних раскладках с одним уровнем-листом. По умолчанию `issue` присутствует.

**Сохранение:**
- Per-user в `users.preferences` (JSON-поле в модели `User`, либо новая колонка `analytics_layout`).
- Структура preferences:
  ```json
  {
    "analytics": {
      "active_preset": "custom" | "people" | "categories" | "...",
      "group_order": ["employee", "category", "issue"],
      "hidden_levels": ["team", "role", "work_type"],
      "saved_presets": [
        {"name": "По людям без команд", "group_order": [...], "hidden_levels": [...]}
      ]
    }
  }
  ```

**UI:**
- Кнопка «Настройка столбцов» переименовывается в **«Настройка отчёта»**.
- Drawer (или модалка) с двумя секциями:
  1. **Группировка** — список активных уровней с drag-handle, переключателем «глаз», крестиком-скрытием. Ниже зона «Скрытые» — клик возвращает в активные. Сверху ряд пресетов + поле «Сохранить как…».
  2. **Столбцы** — текущий список столбцов с чекбоксами видимости (без изменений в логике).

### 2. Колонки процентов

**Что:** заменить одну колонку «% от итога» на две независимые:
- **«% в группе»** — `node.fact_hours / parent.fact_hours × 100`. Видима по умолчанию.
- **«% от итога»** — текущая, `node.fact_hours / grand_total_fact × 100`. Скрыта по умолчанию, включается в «Настройка отчёта».

**Бэкенд:**
- `NodeTotals` получает новое поле `pct_in_group: float | None`.
- Вычисляется при свёртке дерева — на каждом уровне делим на `parent_total`.
- Существующее `pct_total` оставляем.

**Фронт:**
- `AnalyticsTable.tsx` добавляет колонку «% в группе» рядом с фактом.
- Логика `useAnalyticsColumns` получает новый ключ `pct_in_group` (default visible) и обновлённую видимость `pct_total` (default hidden, при первом запуске у существующих пользователей оставляем как есть — миграция не обязательна, т.к. дефолт переключается только для новых).

### 3. Счётчик чужих задач на агрегированных уровнях

**Что:** на каждой агрегированной строке (Team, Role, Employee, WorkType, Category — если уровень виден) — оранжевый бейдж справа от часов: `12 задач / 85ч / 15%`.

**Формула:**
- `foreign_issue_count` — кол-во distinct `issue_id` под узлом, у которых `is_foreign=True`.
- `foreign_hours` — сумма часов на чужих задачах под узлом.
- `foreign_pct` — `foreign_hours / total_hours × 100` (если total > 0).

**Бэкенд:**
- `NodeTotals` получает поля: `foreign_issue_count: int`, `foreign_hours: float`, `foreign_pct: float`.
- В `calc_totals` агрегируем по флагу `v.get("is_foreign")` в бакете.

**Фронт:**
- В `AnalyticsTable.tsx` рендерится оранжевая «капсула» справа от ячейки «Часы факт» на агрегированных строках (kind != 'issue', kind != 'worklog-block'). Скрыта если `foreign_issue_count == 0`.
- Капсула: `{issues} зад. · {hours}ч · {pct}%`. Иконка ⚠ или цветная точка слева.
- На уровне Задача — не отрисовывается. На скрытых уровнях — нет таких строк, вопрос не возникает.

### 4. Команда-владелец чужой задачи

**Что:** на строке задачи с тегом «Чужая» добавить второй тег с названием команды-владельца. Контур цветом команды.

**Источник:** `Issue.team` (продуктовая команда) — уже отдаётся бэкендом в `AnalyticsIssueNode`.

**Фронт:**
- В `buildIssueNode` в `AnalyticsTable.tsx` если `i.is_foreign && i.team` — рендерить второй `<Tag>` после «Чужая». Контурный стиль (`bordered`, transparent bg, border = цвет команды).
- Цвет команды: использовать палитру `CHART_COLORS` (детерминистический хэш от имени) либо новый хелпер `teamColor(name)`.

### 5. (Отложено) Планы на «Проекты и развитие»

В этом цикле не меняется. Возвращаемся после починки ресурсного планирования.

### 6. KPI-плашки в шапке

**Что:** над таблицей — ряд из 5 крупных плашек:
1. **Σ часов факт** — `grand_totals.fact_hours`
2. **Σ часов план** — `grand_totals.plan_hours` (если есть)
3. **% выполнения** — `fact / plan` (если plan > 0)
4. **Сотрудников** — `grand_totals.employee_count`
5. **Чужих часов** — `grand_totals.foreign_hours` (новое поле, см. п. 3)

**Поведение:**
- Реактивно обновляются при смене периода / фильтра / группировки (через ту же `useAnalyticsReport` query).
- Скрыта, если выбран только режим «Новый вид» (`AnalyticsDetailWorkspace`) — там своя сводка. Для классического вида показываем всегда.

**Стиль:** AntD `Statistic`-стиль или кастом, в палитре dark-theme (`#0f2340` фон, cyan акценты).

### 7. Bar-in-cell для «Часы факт»

**Что:** под числом в ячейке «Часы факт» — горизонтальная мини-полоска. Длина = `pct_in_group` (доля от родителя). Цвет цианом `#00c9c8`.

**Поведение:**
- Только на агрегированных строках (kind != 'issue', kind != 'worklog-block').
- Высота полоски ~3px, ширина = % от полной ширины ячейки.
- Tooltip: «Доля от родителя: X%».
- На корневых строках (Team), где нет «родителя» — полоска относительно гранд-тотала, либо скрыта (решение по визуалу). По умолчанию: на корне используем `pct_total`, ниже — `pct_in_group`.

## Архитектура

### Бэкенд

**Файлы:**
- `app/services/analytics_service.py` — добавить агрегацию `foreign_*` и `pct_in_group` в `calc_totals`.
- `app/schemas/analytics_report.py` — расширить `NodeTotals`.
- `app/api/endpoints/analytics.py` — если есть отдельный endpoint для KPI, или брать из `AnalyticsReportResponse.grand_totals`.

**Контракт API:**
- `GET /analytics/report` остаётся идемпотентным; ответ обогащается новыми полями `pct_in_group`, `foreign_issue_count`, `foreign_hours`, `foreign_pct` в каждом `NodeTotals`.
- Группировка остаётся серверной по умолчанию (Team → Role → Employee → WorkType → Category → Issue). **Перестановка/скрытие уровней — на клиенте** (постпроцессинг ответа). Это упрощает бэк и кеширование.

### Фронт

**Файлы:**
- `frontend/src/components/analytics/AnalyticsTable.tsx` — рефакторинг построения дерева: вместо жёсткой цепочки `buildTeamNode → buildRoleNode → buildEmployeeNode → buildWorkTypeNode → buildCategoryNode → buildIssueNode` ввести универсальный `buildTreeFromLayout(data, layout)` где `layout = ['team', 'role', 'employee', ...]`.
- `frontend/src/components/analytics/AnalyticsColumnSettings.tsx` → переименовать в `AnalyticsReportSettings.tsx`, добавить секцию «Группировка» c drag-n-drop через `@dnd-kit/core` + `@dnd-kit/sortable` (уже в проекте).
- `frontend/src/components/analytics/AnalyticsKpiTiles.tsx` — новый компонент, ряд из 5 плашек.
- `frontend/src/hooks/useAnalyticsLayout.ts` — новый хук, читает/пишет `users.preferences.analytics`. Дефолт = стандартная раскладка.
- `frontend/src/pages/AnalyticsPage.tsx` — рендерит KPI выше таблицы (в classic view), пробрасывает `layout` в `AnalyticsTable`.
- `frontend/src/api/analytics.ts` — обновить типы под новые поля `NodeTotals`.

**Клиентский постпроцессинг дерева:**
1. Бэк отдаёт жёсткую цепочку (как сейчас).
2. Клиент собирает плоский список «листовых событий» (issue × его путь) из ответа, либо разворачивает дерево в плоский список бакетов.
3. По `layout` (порядок видимых уровней) пересобирает дерево, группируя по выбранным ключам.
4. Авто-сворачивание: если у уровня только один ребёнок — отрисовать его «склеенно» (один visual row с двумя метками) либо просто авто-раскрыть.

### Сохранение per-user preferences

- Текущая модель `User` имеет только `selected_teams_raw` (JSON-строка). Общего `preferences` поля нет.
- Добавить колонку `User.preferences: Text` (JSON-строка, default `'{}'`) через alembic migration. По аналогии с `selected_teams_raw` + property-сеттер/геттер `preferences: dict`.
- API: добавить endpoint `GET /me/preferences` и `PUT /me/preferences` (либо расширить существующий `/me`). Возвращает/принимает свободный JSON. Аналитика читает только ключ `preferences['analytics']`.
