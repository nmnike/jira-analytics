# Справочник вовлечённости по ролям + запись в задачи при утверждении

Дата: 2026-06-18

## Проблема

Вовлечённость по ролям (доля) в Jira заполнена не всегда. Когда поле пустое, у целевой
задачи нет значения вовлечённости, и планирование опирается на запасной расчёт. Нужен
способ задавать вовлечённость по ролям централизованно — на уровне команды и периода —
и фиксировать её в задачах при утверждении сценария.

## Решение — обзор

1. **Справочник вовлечённости** — отдельная таблица значений вовлечённости по ролям,
   с привязкой к команде и кварталу начала действия. Редактируется в разделе «Сценарии».
2. **Запись при утверждении** — при утверждении сценария пустые значения вовлечённости
   у включённых целевых задач заполняются из справочника. Локально, в Jira не пишем.

Длительность в справочник **не входит** — она расчётная для каждой задачи.

## Справочник вовлечённости

### Состав записи

- **Команда** — обязательна (строки «все команды» нет).
- **Роль** — одна из четырёх фаз: Анализ, Разработка, Тестирование, ОПЭ.
- **Действует с** — квартал и год начала действия.
- **Вовлечённость** — доля 0–1.

Уникальность: одна запись на (команда, роль, квартал+год начала).

### Действие во времени

Значение действует, начиная с указанного квартала, и до следующей записи по той же
команде и роли с более поздним «действует с».

Пример (команда A, роль Анализ):
- «с 1кв2026 = 0.8»
- «с 3кв2026 = 0.9»

При планировании 1–2кв2026 берётся 0.8; 3кв2026 и далее — 0.9.

### Поиск значения

Для пары (команда, роль) и планируемого квартала: берётся запись с самым поздним
«действует с», который не позже планируемого квартала. Если такой записи нет —
значения по умолчанию нет (поведение как сейчас при пустом поле).

Сравнение кварталов — по паре (год, номер квартала).

### Редактирование

Таблица в разделе «Сценарии» (страница планирования): добавление, правка, удаление
строк. Доступ — как у остального планирования (не публичная страница).

## Запись при утверждении сценария

Момент применения — **только утверждение** сценария. Черновой расчёт справочник не
затрагивает.

Алгоритм при утверждении:
1. Берём команду и квартал утверждаемого сценария.
2. Для каждой включённой целевой задачи сценария и каждой из четырёх ролей:
   - если у задачи вовлечённость по этой роли **пустая** —
   - находим значение в справочнике по (команда сценария, роль, квартал сценария);
   - если значение найдено — записываем его в задачу.
3. Непустые значения (из Jira или внесённые вручную) не трогаем.

Запись идёт в целевую задачу локально (в карточку бэклога, которая и есть «целевая
задача»). В Jira ничего не отправляется. Действующее правило синхронизации сохраняется:
если позже Jira пришлёт собственное значение этого поля, оно перезапишет наше.

Идемпотентность: повторное утверждение не перезаписывает — пустых полей уже нет.

### Откат утверждения

При откате сценария в черновик вписанные значения **остаются** в задачах. Они
трактуются как обычные данные. Повторное утверждение их не тронет (поля непустые).

## Граничные случаи

- Команда задачи не совпадает с командой сценария / нет записи в справочнике — поле
  остаётся пустым (старое поведение).
- Сценарий без указанного квартала или года — запись вовлечённости пропускается
  (нечем искать в справочнике).
- Роль ОПЭ — пишется в поле вовлечённости запуска целевой задачи.

## Технический аппендикс (для реализации)

- Роли справочника → поля целевой задачи: analyst→`involvement_analyst`,
  dev→`involvement_dev`, qa→`involvement_qa`, opo→`involvement_launch`.
- Запись при утверждении встраивается в `approve_scenario`
  ([app/api/endpoints/planning.py](../../../app/api/endpoints/planning.py)) по списку
  включённых `BacklogItem` (тех же, что идут в ревизию/снапшот).
- Цель записи — `BacklogItem.involvement_*` (карточка «целевой задачи»). Issue не
  трогаем: он отражает значение Jira, и запись туда выдавала бы наш дефолт за Jira-данные.
- Новая таблица справочника + Alembic batch-миграция. Поля: команда, роль,
  год начала, квартал начала, значение; UUID-ключ; стандартные timestamps.
- Лукап-хелпер: для (team, role, year, quarter) вернуть значение последней действующей
  записи или None.
- Эндпоинты CRUD справочника + фронт-таблица в разделе планирования.
- Тесты: лукап по датам начала (несколько записей, граница квартала), запись только в
  пустые поля при утверждении, идемпотентность повторного утверждения, сохранение
  значений при откате.
