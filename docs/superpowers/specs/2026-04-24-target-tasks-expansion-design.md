# Целевые задачи: расширение реестра на квартальные задачи

**Дата:** 2026-04-24  
**Статус:** Approved

## Проблема

Раздел «Бэклог» отслеживает только задачи с категорией `initiatives_rfa` («Инициативы и RFA»). Задачи, взятые в работу в текущем квартале (категория `quarterly_tasks`, «Квартальные задачи»), в планировании не участвуют — их нельзя включить в сценарий.

PM хочет создать сценарий из уже активных инициатив ITL/RFA, чтобы зафиксировать факт текущего квартала и в дальнейшем анализировать его.

## Решение

Расширить систему BacklogItem на вторую категорию (`quarterly_tasks`), переименовать раздел и реструктурировать вкладки.

---

## Секция 1 — UI / Навигация

**Раздел «Бэклог»** переименовывается в **«Целевые задачи»** (в сайдбаре и заголовках страницы).

### Вкладки

| Вкладка | Содержимое | Изменение |
|---|---|---|
| **Активные** | BacklogItem из категории `quarterly_tasks` — задачи в работе | Новая |
| **Бэклог** | BacklogItem из категории `initiatives_rfa` — очередь | Переименована (было «Активные») |
| **В работе** | Задачи, включённые в утверждённый сценарий | Без изменений |
| **Архив** | Архивированные задачи | Без изменений |

---

## Секция 2 — Данные и синхронизация

### Источник определения вкладки

Вкладка BacklogItem определяется по `category` связанной Jira-задачи (поле `Issue.category`). Ручное создание BacklogItem остаётся доступным только для «Бэклог» (initiatives_rfa) — как сейчас. Записи из `quarterly_tasks` попадают в систему исключительно через синхронизацию с Jira.

### Поведение при смене категории в Jira

| Было | Стало | Результат |
|---|---|---|
| `initiatives_rfa` | `quarterly_tasks` | Запись остаётся, переходит на вкладку «Активные» |
| `quarterly_tasks` | `initiatives_rfa` | Переходит обратно на «Бэклог» |
| `initiatives_rfa` или `quarterly_tasks` | `archive` / `archive_target` | Запись архивируется |
| `initiatives_rfa` или `quarterly_tasks` | Другая активная категория | Запись архивируется (больше не отслеживается) |

### Изменения в `refresh-from-jira`

`POST /backlog/refresh-from-jira` расширяет поиск кандидатов: ищет Issue с `category IN ('initiatives_rfa', 'quarterly_tasks')`.

- Новые задачи из обеих категорий → создаётся BacklogItem
- Задачи, сменившие категорию между двумя отслеживаемыми → обновляется без архивации
- Задачи, ушедшие в архив или другую категорию → BacklogItem архивируется

---

## Секция 3 — Сценарии

### Аллокации

При создании сценария (`POST /scenarios`) автоматически создаются аллокации для **всех неархивированных** BacklogItem — из обоих источников (`initiatives_rfa` и `quarterly_tasks`). Логика включения/исключения не меняется: PM вручную отмечает что взять.

### Колонка «Источник»

В таблице аллокаций добавляется колонка **«Источник»** (или тег) — показывает, из какой категории пришла задача:
- «В работе» — из `quarterly_tasks`
- «Бэклог» — из `initiatives_rfa`

Это не влияет на логику — только наглядность.

---

## Что не меняется

- Логика утверждения/отката сценария
- Логика архивации BacklogItem вручную
- Связь BacklogItem → Issue (one-to-one)
- Авто-добавление аллокаций в draft-сценарии при появлении новых BacklogItem
- Авто-удаление аллокаций при архивации BacklogItem

---

## Scope

- **Backend:** `refresh-from-jira` (расширить категории), список BacklogItem (фильтр по вкладке), ответ аллокаций (добавить поле источника)
- **Frontend:** переименование раздела в навигации, реструктура вкладок, тег «Источник» в таблице сценария
- **Миграция:** не требуется (категории уже существуют)
