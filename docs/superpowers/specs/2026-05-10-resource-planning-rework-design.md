# Дизайн-док — Переработка раздела «Ресурсное планирование»

**Дата:** 2026-05-10
**Скоуп:** `/resource-planning` (логика расчёта + интерфейс + панель конфликтов)
**Связанные памяти:** project_resource_planning_fixes_shipped, project_gantt_phase3_shipped, project_planning_improvements_5phases_shipped

## 1. Контекст и проблема

Текущий раздел работает, но имеет ряд критичных дефектов:

1. **Поломанный расчёт ёмкости.** Коэф вовлечённости влияет только на длительность фазы, но не на дневную ёмкость сотрудника. Часы укладываются по полной дневной норме (8 ч), а длительность считается отдельно с учётом коэфа. Результат — несовместимые цифры и перегрузки до 24000% на скрине.
2. **Жёсткая последовательность фаз.** `analyst → dev → qa → opo` зашита в коде. Реальные кейсы (тестирование параллельно с анализом, ОПЭ начинается с конца разработки) выразить нельзя.
3. **Нет ручной правки.** Пользователь не может изменить дату старта фазы, переопределить предшественника, разбить фазу на части. Авто-распределение — единственный способ получить расписание.
4. **Бесполезная панель конфликтов.** Простыня одинаковых строк с GUID, перегрузка одного сотрудника дублируется на каждый день. Невозможно понять что и где сломано.
5. **Слабый визуал графика.** Не видно конфликтов на барах, не видно недоступных дней внутри фазы, длинные имена проектов обрезаются, фазы нельзя свернуть.
6. **Узкие блокировки.** Поддерживаются три исключающих режима (один сотрудник / одна роль / вся команда) — нельзя выбрать «эти две роли + три конкретных человека».

## 2. Цель и не-цели

**Цель:** сделать раздел рабочим инструментом — корректные цифры, читаемый график, ручная правка с сохранением, понятная панель конфликтов.

**Не входит:**
- Изменения раздела `/planning` (отбор инициатив в сценарий) — отдельная история.
- Эстафета (`relay`) и связи между разными инициативами (`PlanItemDependency`) — оставляем как есть.
- Старые `/resource-planning-v2` и `/resource-planning-v3` — уже выпилены, не возвращаем.
- Алгоритмическая часть (RCPSP-leveler, CPM, форки/сравнение, PERT P50/P90) — концептуально не меняется. Меняем входные данные расчёта (формула вовлечённости, граф зависимостей) и постобработку конфликтов.

## 3. Решения по логике

### 3.1. Формула вовлечённости (фикс перегрузок)

**Новое правило укладки часов фазы в календарь:**

Дневная ёмкость роли в фазе = `доступно_по_календарю_сотрудника × коэф_вовлечённости_роли`.

Часы фазы льются по этой урезанной ёмкости. Длительность фазы вычисляется автоматически = `ceil(часы / дневная_ёмкость)`. Поля `duration_*_days` из Jira — игнорируются в расчёте (длительность в Jira хранится в часах в других полях, на которые мы и так смотрим через `estimate_*_hours`).

Параметры:
- `involvement_*` из `BacklogItem` (берётся из Jira для связанных задач, NULL → 1.0).
- Если `involvement = NULL` → дневная ёмкость = доступно_по_календарю (старое поведение для незаполненных задач).

**Пример:** Аналитик доступен 8 ч/день, `involvement_analyst = 0.7` → 5.6 ч/день по этой фазе. Часы фазы = 20 → длительность = 4 дня (ceil(3.57)). В каждый день аналитик «занят» 5.6 ч, оставшиеся 2.4 ч свободны под другие задачи.

**Что снимается:** функция `_resolve_phase_calendar_days` в текущем виде не нужна — длительность теперь не отдельный концепт, а следствие часов и ёмкости. `parallel_count_*` остаётся: при `parallel_count > 1` дневная ёмкость роли умножается на это число (несколько исполнителей одной роли работают параллельно).

### 3.2. Свободный граф зависимостей фаз

**Новая модель.** Каждая фаза имеет список предшественников (0..N) — других фаз той же инициативы. По умолчанию при создании новой фазы предшественник = последняя по порядку фаза цепочки `analyst → dev → qa → opo` (если она есть).

Фаза стартует не раньше окончания **всех** своих предшественников. Если предшественников нет — стартует с начала квартала.

**Хранение.** Новая таблица:

```
phase_predecessor (
  id UUID PK,
  successor_assignment_id UUID FK → resource_plan_assignments(id) CASCADE,
  predecessor_assignment_id UUID FK → resource_plan_assignments(id) CASCADE,
  created_at, updated_at
)
```

Уникальный индекс `(successor_assignment_id, predecessor_assignment_id)`. Циклы запрещены — проверка при добавлении.

**При первом расчёте плана** для инициативы создаются стандартные связи (analyst→dev, dev→qa, dev→opo) согласно текущей жёсткой цепочке. Дальше пользователь правит.

### 3.3. Закрепление и ручные правки

Расширяем существующий `is_pinned` флаг (булевый) до явного перечисления действий:

Новые поля в `resource_plan_assignments`:
- `pinned_start: bool` — старт зафиксирован руками, при `compute` не двигается.
- `pinned_employee: bool` — сотрудник закреплён руками (заменяет старый `is_pinned`).
- `pinned_split: bool` — разбив на части задан руками, при `compute` не объединяется.
- `manual_edit_at: datetime` — когда последняя ручная правка (для бейджа «ручное» в UI).

Существующий `is_pinned` мигрирует в `pinned_employee = is_pinned` для совместимости, потом удаляется.

Ручная правка триггерит **мини-пересчёт**: затрагиваются только цепочка наследников этой фазы (по графу предшественников) и заново строятся конфликты. Полный `compute` пересчитывает всё с нуля, кроме закреплённых правок.

### 3.4. Ручной разбив фазы

UI: контекст-меню фазы → «Разбить» → диалог:
- Выбор количества частей (2..10).
- Таблица «Часть N — часов __» (по умолчанию равномерно). Сумма = исходные часы. Последняя строка добирает остаток после редактирования.
- Чекбокс «Каскадно разбить последующие фазы пропорционально» (по умолчанию ВКЛ).

При применении:
- Фаза превращается в N записей `ResourcePlanAssignment` с разными `part_number` и часами.
- Каждая часть имеет своего предшественника: часть K зависит от части K-1 этой же фазы.
- Если каскад ВКЛ: каждая зависимая фаза разбивается на N частей пропорционально (округление до целых, остаток в последний кусок). Связи: часть K зависимой фазы зависит от части K разбитой фазы.
- Каждая часть получает `pinned_split = true`.

**Ограничение.** Повторный разбив уже разбитой фазы запрещён. Сначала «Объединить части» в контекст-меню, потом разбить заново.

### 3.5. Ручной старт фазы

Два способа:
1. **Drag бара** по горизонтали в графике — сетка дней.
2. **Контекст-меню → «Перенести старт»** — date-picker для точного ввода.

Действия:
- `start_date` записывается в фазу.
- `pinned_start = true`, `manual_edit_at = now`.
- Каскадно сдвигаются все наследники по графу — на ту же дельту дней. Уже закреплённые наследники (`pinned_start = true`) не двигаются (это закрепляет «островки»).
- Запускается мини-пересчёт CPM + leveler + конфликтов.

**Откат.** Контекст-меню → «Снять закрепление» сбрасывает `pinned_start`, при следующем `compute` фаза вернётся к авто-расчёту.

### 3.6. Расширенные блокировки

Текущая модель `ScheduledBlock`: `employee_id | role_id | team` (взаимоисключающие). Расширяем:

```
ALTER scheduled_blocks:
  - role_id (single FK) → DELETE
  - employee_id (single FK) → DELETE

NEW: scheduled_block_role (M:N)
  - block_id, role_id

NEW: scheduled_block_employee (M:N)
  - block_id, employee_id
```

Резолв при расчёте доступности:
- Если у блока есть роли И/ИЛИ люди — блок применяется к объединению (роль ИЛИ конкретный человек).
- Если ни ролей ни людей — блок применяется ко всей команде (старое поведение).

Миграция переноса данных: для каждой существующей записи с `role_id` создаётся одна строка в `scheduled_block_role`; аналогично для `employee_id`. Старые колонки удаляются.

### 3.7. ОПЭ (без изменений по существу)

ОПЭ остаётся как сейчас — две записи `phase="opo"` (одна для аналитика, одна для разработчика) с тем же `part_number=1` и разными `employee_id`, в пропорции `opo_analyst_ratio`. Но при свободном графе зависимостей пользователь может вручную перенаправить ОПЭ-аналитика, чтобы он стартовал параллельно с тестированием, а не после.

## 4. Решения по интерфейсу

### 4.1. Компоновка списка слева (двухстрочные строки)

Каждая инициатива — одна строка с двумя визуальными уровнями:
- Крупно: название инициативы (полностью, с переносом если очень длинное).
- Мелко под названием: ключ Jira · название проекта.

Слева от строки — индикатор `▶/▼` для свёртки/развёртки фаз.

Фазы при разворачивании показываются вложенными строками с отступом 32 px. Каждая фаза:
- Иконка фазы + короткое название («Анализ», «Разработка», «Тестирование», «ОПЭ»).
- Имя исполнителя (или «пул» для тестирования).
- Бейдж «✋ ручное» если есть `manual_edit_at`.
- Бейдж серьёзности конфликта (если есть).

Состояние свёрнутости фаз — per-user, сохраняется в `user_preferences` (см. 4.6).

### 4.2. Бары на графике — недоступные дни

Каждая фаза = один сплошной бар от первого до последнего использованного дня. Дни внутри бара, в которые сотрудник не работает по фазе, рисуются разными паттернами:

- **Серая косая штриховка** (≈ rgba(255,255,255,0.10)): выходной, праздник, перенос (общий для всех).
- **Оранжевая косая штриховка** (≈ rgba(245,158,11,0.5)): персональная недоступность сотрудника (отпуск, больничный, заблокированный период).

Источник данных: для каждой фазы фронт получает массив `[{date, status: working|weekend|absence}, ...]` для диапазона `[start_date..end_date]`.

### 4.3. Тумблер «Только рабочие дни»

В шапке графика — тумблер. Когда включён: столбцы выходных и праздников РФ скрываются, бары визуально «сжимаются» (но реальные даты сохраняются в логике расчёта). Персональные отсутствия НЕ скрываются — это часть бара.

Состояние per-user, сохраняется.

### 4.4. Подсветка конфликтов на графике

Тепловая полоса загрузки сотрудника + красная рамка на конфликтных барах:

- **Над барами фаз сотрудника** — отдельная строка-«термометр» с делениями по дням:
  - Зелёный (rgba(34,197,94,0.4)) при загрузке ≤90%.
  - Жёлтый (rgba(234,179,8,0.7)) при 90..110%.
  - Красный (rgba(239,68,68,0.85)) при >110%.
- **Сами бары конфликтных фаз** — обводятся красной рамкой (`box-shadow inset 0 0 0 2px #ef4444`).
- **Иконка ⚠ в строке** конфликтной фазы.

Тепловая полоса показывается всегда. Включается из той же модели данных, что и панель конфликтов.

### 4.5. Управление зависимостями фаз — три способа

Все три способа должны быть доступны:

**Drag-стрелки на графике.** На концах баров — точки (handles). Тащишь от точки конца одной фазы к точке начала другой → создаётся стрелка-связь. Удаление: клик по стрелке + Del.

**Боковая панель свойств.** Клик по фазе → открывается панель справа. Поля:
- Часы (read-only из BacklogItem).
- Старт (date-picker, при изменении → `pinned_start = true`).
- Сотрудник (select из команды, при изменении → `pinned_employee = true`).
- Предшественники: список чипов всех других фаз инициативы со чекбоксами. Меняешь чекбокс → правится граф зависимостей.
- Кнопка «Снять все ручные правки».

**Контекст-меню.** Правый клик по фазе → меню с пунктами:
- Перенести старт…
- Разбить на части… / Объединить части
- Зависит от: список фаз с галочками (✓ = выбран как предшественник)
- Без предшественников (старт от начала квартала)
- Снять ручную правку

### 4.6. Per-user предпочтения

Новая таблица или расширение `User`:

```
user_resource_planning_preferences (
  user_id PK FK,
  hide_weekends bool DEFAULT false,
  collapsed_initiative_ids JSON list of UUIDs,
  selected_view_mode str ('day'|'week'|'month'),
  show_relay bool DEFAULT true,
  updated_at
)
```

Записывается через debounced PATCH при изменении любого UI-стейта.

## 5. Панель конфликтов

### 5.1. Группировка с переключателем

Над таблицей — табы (radio): «По задачам / По людям / По типу». По умолчанию — «По задачам».

**По задачам.** Один уровень: инициатива (ключ + название). Раскрытие → список её конфликтов. Колонки: серьёзность, тип, диапазон дат, действия.

**По людям.** Один уровень: сотрудник. Раскрытие → его конфликты. Колонки: серьёзность, тип, инициатива, диапазон дат, действия.

**По типу.** Группы: Перегрузка, Не вмещается в квартал, Нет аналитика/разработчика, Старт позже плана, Принудительный сдвиг, Принудительная замена. Под каждой группой — список инцидентов с инициативой/сотрудником.

### 5.2. Дедупликация перегрузок по дням

Текущая логика создаёт по строке на каждый день перегрузки → 50+ строк. Заменяем на одну запись «Шутов С. перегружен 1–10 апреля» с массивом `affected_dates`. UI агрегирует по `(employee_id, type)` диапазоны (последовательные даты сливаются в один период).

В таблице `plan_conflicts` поля `window_start/window_end` уже есть — используем их как диапазон, дни при необходимости можно показать в раскрытии строки.

### 5.3. Понятные сообщения

Шаблоны вместо GUID:

- `OVERLOAD_HIGH/MED/LIGHT`: «Шутов С. перегружен на 28% (140%) в период 1–10 апреля. Конфликтуют: PRJ-10892 Разработка ↔ ITL-299 Разработка».
- `QUARTER_OVERFLOW`: «PRJ-10892 не вмещается в квартал — ОПЭ заканчивается 5 июля».
- `NO_ANALYST` / `NO_DEV`: «В команде "1С (ERP)" нет {аналитиков|разработчиков} — расписание {фазы анализа|фазы разработки} невозможно».
- `LATE_START`: «PRJ-10892 Анализ стартует с отставанием 5 дней».
- `LEVELING_DELAY`: «PRJ-10892 Разработка перенесена на 3 дня — выравнивание загрузки Шутова С.».
- `LEVELING_REASSIGN`: «ITL-299 Разработка переназначена с Шутова С. на Петрова И. — выравнивание».
- `SPLIT_REQUIRED`: «PRJ-10892 разбита на части — попадает на заблокированный период».

### 5.4. Действия на строке

На каждой строке конфликта — иконки-кнопки:
- 🎯 «К фазе на графике» — скролл графика и выделение фазы.
- 👤 «Назначить другого» — открыть панель свойств с фокусом на сотруднике.
- 🔇 «Заглушить» — переводит конфликт в `status = muted`, скрывается из активных.

Фильтры сверху:
- По серьёзности (Критические / Предупреждения / Информация / Все).
- По статусу (Активные / Заглушённые / Все).
- Поиск по тексту (название инициативы, фамилия).

## 6. API изменения

### 6.1. Существующие эндпоинты (расширение)

`POST /resource-planning/resource-plans/{id}/compute` — добавляется параметр `?scope=full|incremental`. По умолчанию — `full`. `incremental` пересчитывает только цепочку наследников от указанной фазы (передаётся в теле через `assignment_id`).

`GET /resource-planning/resource-plans/{id}/gantt` — ответ обогащается:
- На каждой фазе: `predecessors: [assignment_id, ...]`, `pinned_start/employee/split`, `manual_edit_at`, массив `unavailable_days: [{date, type: weekend|absence}]`.
- На каждом сотруднике: `daily_load: [{date, pct: 0..1.5}]` для тепловой полосы.

### 6.2. Новые эндпоинты

```
PATCH /resource-planning/assignments/{id}
  body: { start_date?, employee_id?, predecessor_ids? }
  → 200 + затронутые фазы (после каскадного сдвига)

POST /resource-planning/assignments/{id}/split
  body: { parts: [hours], cascade: bool }
  → 200 + новые фазы

POST /resource-planning/assignments/{id}/merge
  → 200 + объединённая фаза

DELETE /resource-planning/assignments/{id}/manual-edit
  → 200 + фаза без флагов закрепления

GET /resource-planning/resource-plans/{id}/conflicts?group_by=item|employee|type
  → агрегированный список с шаблонизированными сообщениями

PATCH /user/preferences/resource-planning
  body: { hide_weekends?, collapsed_initiative_ids?, view_mode?, show_relay? }
```

## 8. Тестирование

### 8.1. Unit-тесты бэкенда

- `test_planning_involvement_capacity.py` — формула вовлечённости: 8 ч × 0.7 = 5.6, длительность вычисляется корректно, перегрузки не вылетают за разумные границы.

## 9. Этапы выполнения

Один большой rollout (по решению пользователя). Деление на этапы — для удобства проверок:

1. **Бэкенд — модель и миграции.** Новые таблицы, миграции, обновление `ResourcePlanAssignment`.
2. **Бэкенд — формула вовлечённости.** Замена `_resolve_phase_calendar_days` и `_allocate_hours` на новую логику дневной ёмкости.
3. **Бэкенд — свободный граф зависимостей.** Замена линейного цикла фаз на топологический проход. Сохранение и применение `phase_predecessor`.
4. **Бэкенд — endpoints для ручных правок.** PATCH/split/merge/предпочтения.
5. **Бэкенд — переработка панели конфликтов.** Дедупликация диапазонов, шаблонизированные сообщения, группировка.
6. **Фронт — двухстрочные строки + свёртка.** Перерисовка списка слева.
7. **Фронт — паттерны недоступных дней + тумблер выходных.** Штриховка и скрытие колонок.
8. **Фронт — тепловая полоса + рамки конфликтов.** Подсветка перегрузок на графике.
9. **Фронт — три способа правки зависимостей.** Drag-стрелки, боковая панель, контекст-меню.
10. **Фронт — диалог разбива.** Модалка, каскад, бейдж «ручное».
11. **Фронт — переработанная панель конфликтов.** Табы группировки, фильтры, действия.
12. **E2E + регресс.** Прогон всех существующих + новых тестов.
13. **Smoke на реальных данных.** Прогон на текущей БД, верификация что цифры сошлись.

## 10. Риски и compromise

- **Данные миграции на свободный граф.** Существующие планы получат дефолтную цепочку — пользователю не придётся ничего настраивать заново. Но если у кого-то была кастомная разбивка через `parallel_count` — нужна точечная проверка.
- **Производительность тепловой полосы.** При большой команде (15+ сотрудников) и квартале (90 дней) — 1350 ячеек, нагрузка на рендер. Решение: виртуализация рядов или агрегация по неделям при масштабе «Месяц».
- **Drag-стрелки на dense графике.** Может быть сложно попасть в handle при тонком баре. Решение: handles раздуваются при наведении на бар, плюс альтернативные способы (панель + контекст-меню).
- **Каскад наследников при ручном старте.** Если у наследника есть свои закреплённые правки — он не двигается. Это может выглядеть странно (родитель уехал, наследник остался). Решение: визуальный warning «есть закреплённые наследники» при drag.

## 11. Открытые вопросы (на этап реализации)

Все основные вопросы закрыты на этапе брейнсторма. На этапе реализации могут возникнуть:

- Точный размер handle для drag-стрелок (UX-проба).
- Цветовая палитра тепловой полосы (текущие зелёный/жёлтый/красный — из дизайн-системы или кастомные).
- Масштаб «Месяц» — агрегация дней или показ каждого дня очень узкими столбцами.

## 12. Совместимость

- Старый эндпоинт `POST /compute` без параметров — продолжает работать (default `scope=full`).
- Старый `is_pinned` мигрирует в `pinned_employee`. Внешний API на чтение возвращает оба поля до полного перехода фронта.
- `ScheduledBlock` API: на чтение — отдаём массивы `roles[]`/`employees[]`. На запись — принимаем оба формата (legacy single + new multi) первые две недели, потом legacy выпиливается.
