# Виджет «Проекты квартала» — каскадный сбор часов по дереву эпика команды

Дата: 2026-06-15

## Проблема

Виджет «Проекты квартала» (`AnalyticsService.get_dashboard_projects`) считает факт-часы Инициативы (RFA) только по самой Инициативе и её **прямым детям** (один уровень). Реальная иерархия глубже:

```
RFA-2609 (Инициатива)         ← строка виджета
└─ PRJ-10766 (Эпик команды)   ← прямой ребёнок (код доходит сюда)
   └─ задачи                   ← НЕ считаются (часы здесь)
      └─ подзадачи             ← НЕ считаются (часы здесь)
```

Часы лежат на задачах/подзадачах на глубине 2+, поэтому виджет показывает «нет работ» для RFA, где работа фактически идёт.

Текущая реализация: `analytics_service.py:263-279` строит `child_to_parent` запросом `Issue.parent_id.in_(issue_ids)` (один уровень). Все агрегации (`epic_fact_hours`, `epic_team_fact_hours`, `epic_trend`, `epic_last_wl`, `epic_forecast`, weekly activity, assignees, subtasks count) работают по этому одноуровневому отображению.

## Целевое поведение (вариант B — «только дерево эпика команды»)

Для каждой Инициативы:

1. Найти внутри неё **эпики выбранной команды** на любой глубине.
2. Собрать **всё поддерево** под этими эпиками (задачи, подзадачи, любая вложенность).
3. Суммировать по этому поддереву: факт-часы, командные/чужие часы, тренд, прогноз, недельную активность, аватары, «сделано/всего».

### Правила области (scope)

- **Командный эпик** определяется по полю команды задачи (`Issue.team` — основная, либо `Issue.participating_teams` — участвующая). Узел-потомок Инициативы попадает в область, если **ближайший родитель-контейнер с проставленной командой** (включая сам узел), не доходя до Инициативы, принадлежит выбранной команде (основная ИЛИ участвующая).
- У эпиков команда проставлена всегда (подтверждено заказчиком) — fallback на «нет команды нигде» не требуется, но реализация не должна падать: узел без командного предка просто не попадает в область.
- **Помощь извне**: часы людей не из команды, залогированные внутри командного эпика, идут в «помощь извне» (split по сотруднику, как сейчас).
- **Эпики чужих команд** под той же Инициативой в виджет не попадают вовсе.
- **Часы напрямую на Инициативе** не включаются (на RFA логировать запрещено административно — заказчик подтвердил).
- **Фильтр команды не выбран** (`teams` пуст): область = всё поддерево Инициативы целиком, без ограничения по команде; всё считается командным (сохраняет текущее «нет раздела» поведение, но с каскадом).
- **План часов** — без изменений, из утверждённого сценария (`plan_by_issue`).

## Реализация

### Сбор поддерева (новое)

Заменить одноуровневый запрос `child_to_parent` на нисходящий обход (BFS) от Инициатив вниз до конца дерева:

- Старт: множество id Инициатив (`issue_ids`).
- На каждом уровне: `Issue.id, parent_id, team, participating_teams, status_category` где `parent_id IN <текущий уровень>`.
- Повторять, пока есть дети. Глубина дерева мала (~4 уровня), число Инициатив в утверждённом сценарии — десятки; запросов и строк немного, поле `parent_id` индексировано.
- Защита от циклов: не обрабатывать уже посещённые id.

Результат:
- `node_parent: dict[id, parent_id]` — для всех потомков.
- `node_team: dict[id, (team, participating_teams)]` — для определения командного владельца.
- `node_status: dict[id, status_category]`.

### Определение командного владельца и области

Для каждого потомка `N` подняться по цепочке родителей до (не включая) Инициативы, найти ближайший узел с непустым `team`. Узел в области, если этот владелец совпадает с выбранной командой (`team` основная или `participating_teams` содержит команду). При пустом фильтре команды — все потомки в области.

Построить `descendant_to_rfa: dict[id, rfa_id]` только для узлов в области. Заменить им `child_to_parent` во всех агрегациях:
- `all_wl_ids = set(issue_ids) | set(descendant_to_rfa.keys())` (id Инициатив нужны для последнего/первого ворклога и т.п., но факт-часы самой RFA = 0, т.к. логировать запрещено — включение безвредно).
- Все функции `epic_*` суммируют по `descendant_to_rfa` вместо `child_to_parent`.
- Привязка ворклога к строке: `descendant_to_rfa.get(issue_id, issue_id)`.

### Колонка «сделано / всего»

`subtasks_done/total` пересчитать по узлам в области (все потомки Инициативы в области; контейнеры-эпики включаются в счёт — простой и предсказуемый счёт). `done` — `status_category == "done"`.

## Затронутые компоненты

- `app/services/analytics_service.py` — `get_dashboard_projects` (метод 180-652). Изменяется только сбор иерархии и привязка ворклогов; формат ответа `DashboardProjectsResponse` / `ProjectItem` без изменений.
- Фронтенд не меняется (тот же контракт ответа).

## Тестирование

Новые pytest в `tests/` на `get_dashboard_projects`:

1. **Каскад вглубь**: RFA → Эпик(команда) → Задача → Подзадача с ворклогом на подзадаче. Ожидание: часы попадают в `fact_hours` строки RFA (раньше = 0).
2. **Чужой эпик исключён**: под RFA два эпика — команды A и B; выбрана команда A. Часы эпика B не входят в виджет (ни в команду, ни в помощь извне).
3. **Помощь извне внутри командного эпика**: ворклог сотрудника не из команды на задаче под командным эпиком → `alien_fact_hours`.
4. **Часы на RFA напрямую не считаются** (если такой ворклог есть — не входит).
5. **Фильтр команды не задан**: всё поддерево суммируется, всё командное.

Существующие тесты виджета не должны сломаться (формат ответа неизменен).

## Риски

- Производительность: BFS добавляет несколько запросов на запрос виджета. Дерево мелкое, Инициатив десятки — приемлемо. Если число потомков велико, запросы по индексированному `parent_id` остаются быстрыми.
- Корректность владельца команды зависит от проставленного поля команды на эпиках (подтверждено заказчиком).
