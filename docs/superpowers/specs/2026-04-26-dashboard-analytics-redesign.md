# Dashboard & Analytics Redesign — Design Spec

**Date:** 2026-04-26  
**Status:** Draft — pending user review

---

## Overview

Полная переработка разделов «Дашборд» и «Аналитика». Дашборд получает три новых виджета, ориентированных на квартальное управление проектами. Аналитика — новую страницу с master-detail анализом проектов, включая AI-резюме.

---

## 1. Общий фильтр периода (Dashboard + Analytics)

**Текущее состояние:** произвольный диапазон дат (DateRangePicker).

**Новое поведение:** заменяется на квартальный переключатель:
- Выбор года (число) + квартал Q1/Q2/Q3/Q4 (pills)
- По умолчанию = текущий квартал
- Опционально: детализация до конкретного месяца внутри квартала (дополнительный row с тремя месяцами, появляется при нажатии на квартал повторно или через отдельный toggle)
- Выбор (год + квартал + опционально месяц) сохраняется в URL-параметрах и переживает навигацию между страницами
- Командный фильтр (FactFilterBar) остаётся без изменений

## 2. Дашборд — Виджет 1: Обзор проектов квартала

### Что показывает
Статус проектов из **утверждённого сценария** на выбранный квартал. «Проект» = эпик (BacklogItem), включённый в сценарий со статусом `approved`.

### Визуальный формат
Hero-карточка (Direction C из брейншторма). Два ряда:

**Верхний ряд** (пончик + статусы + метрики):
- Donut-диаграмма статусов (зелёный = завершено, cyan = в работе, красный = просрочено, серый = не начато). Центр: общее число проектов.
- Колонка разбивки: N завершено / N в работе / N просрочено / N не начаты (с %-ами)
- Плитка «Прогноз»: N из 24 закроется к концу квартала (при текущем темпе), прогресс-бар
- Плитка «Средняя длительность завершённых»: X нед., медиана / мин / макс
- Плитка «Перебор оценки»: N из M проектов (факт часов > план)

**Нижний ряд** (два списка):
- **Требует внимания**: список проектов с тегами «просрочен Nд» / «тишина Nд», с именем и суммой часов. Кнопка «ещё N» если >5.
- **Перебор по часам**: список проектов где факт > план, с горизонтальным прогресс-баром (факт/план) и дельтой в часах.

### Drill-down
Клик на проект в списке → переход в Аналитику с этим проектом открытым в правой панели.

### Данные
- Список проектов: `ScenarioAllocation` → `BacklogItem` где сценарий `status=approved` и квартал = выбранный
- Статус проекта: `Issue.status_category` (done / indeterminate / new)
- Просрочено: дедлайн эпика (поле `due_date` из Jira — сейчас не синхронизируется, нужно добавить) < сегодня и status ≠ done. Если поле пустое — проект не считается просроченным.
- «Тишина»: последний ворклог по задачам проекта > N дней назад (N настраивается)
- Перебор: сумма ворклог-часов по задачам эпика vs плановые часы из BacklogItem (поле нужно уточнить — если оценка не хранится, считаем перебор как «факт > X ч» где X настраивается)
- Прогноз: линейная экстраполяция темпа закрытий (завершённых / дней прошло × дней осталось)

## 3. Дашборд — Виджет 2: План/факт нормированных работ

### Что показывает
Соотношение плановых и фактических часов по каждому **виду нормированных работ** из справочника (5 видов: организационные, управление и администрирование, поддержка и консультации, технический долг, технические задачи).

Связка: «Категория задачи» → «Вид нормированных работ» (M:1). Несколько категорий могут относиться к одному виду.

### Визуальный формат
Bullet-бары (Direction B). Для каждого вида:
- Горизонтальный трек (100% = план)
- Заливка = факт
- Вертикальная риска = цель (100%)
- Число %-соотношения справа + часы факт/план
- Внизу итоговая строка: Σ план, Σ факт, общая загрузка

**Цветовые пороги** (настраиваемые, defaults):
- ≤ 100% → зелёный (в норме)
- 101–110% → жёлтый (небольшой перегруз)
- > 110% → красный (перегруз)
- < 70% → жёлтый (недогруз)

Шестерёнка ⚙ на виджете открывает настройку порогов.

### Drill-down
Клик на строку → переход в Аналитику, вкладка детализации по виду работ (отдельный отчёт, разворачивающий вид по категориям и проектам).

### Данные
- **План**: для каждого вида работ берётся % из правил утверждённого сценария (`ScenarioAllocation` → правила вида работ) умноженный на доступные часы команды (производственный календарь − отсутствия) за выбранный период. Глобальные шаблоны (`RoleCapacityRule` / `EmployeeCapacityOverride`) используются только если для периода нет утверждённого сценария.
- **Факт**: сумма `Worklog.hours` где `Issue.category.work_type_id = <вид>` за выбранный период.
- Период: квартал (или месяц) из глобального фильтра.

## 4. Дашборд — Виджет 3: Метрики ворклогов по категориям задач

### Что показывает
Распределение фактических часов по **категориям задач** (справочник Category, без архивных) + meta-метрики по каждой категории.

### Визуальный формат
Treemap + meta-таблица (Direction A):
- **Левая часть (~55%)**: treemap где площадь прямоугольника = часы. Внутри: название категории, часы, %. Цвет уникален для каждой категории.
- **Правая часть (~45%)**: таблица-легенда. Колонки: цвет · категория · часы · ворклогов · задач · сотрудников · ср. ворклог (мин) · % от общего. Итоговая строка.

### Данные
Источник: `Worklog` + `Issue.category`. Период = глобальный фильтр (квартал/месяц). Команда = FactFilterBar.

## 5. Аналитика — Страница анализа проектов

### Общая структура
**Master-detail layout**: список проектов слева (~28%), карточка анализа справа (~72%).

**Фильтры страницы** (в шапке, синхронизированы с дашбордом):
- Квартальный переключатель (год + Q)
- Командный фильтр

### Левая панель — список проектов
Источник: утверждённые сценарии на выбранный квартал.

Каждый элемент списка:
- Цветная точка статуса (зелёный / cyan / красный / серый)
- Название проекта (bold)
- Epic key (мuted) + суммарные часы (right-aligned)
- Выбранный элемент: левая cyan-граница + светлее фон

Над списком: заголовок «Проекты Q2 · N утверждено» + поле поиска.

### Правая панель — карточка проекта

Вся карточка **помещается на один экран без прокрутки**.

**Шапка карточки:**
- Название проекта (22px bold), epic key + период
- KPI-чипы: N задач · N ч · N участников · N нед
- Статус-бейдж (ГОТОВ / В РАБОТЕ / ПРОСРОЧЕН)
- Кнопка «✨ ИИ Обновить»

**Сетка контента** (3 горизонтальных ряда, один разделитель на 50% по всем рядам):

**Ряд 1** (50% | 50%):
- Левая: «Цели проекта ✨» — AI-резюме целей из описания эпика
- Правая: «Основной результат ✨» — AI-резюме итогов + схема контура (текст) + бейджи

**Ряд 2** (50% | 50%, каждая половина поделена внутри):
- Левая половина: `[Структура трудозатрат | Статус проекта]`
  - Структура (58%): donut слева + легенда справа
  - Статус (42%): статус-текст + 2×2 KPI (задач / часов / участников / недель)
- Правая половина: `[Оценка | Итог ✨]`
  - Оценка (38%): звёздный рейтинг из Jira custom fields (3 строки: качество / скорость / результат)
  - Итог (62%): AI-текст вывода по проекту ✨

**Ряд 3** (50% | 50%):
- Левая: «Участники» — горизонтальные бары по сотрудникам (аватар + имя + bar + часы + %)
- Правая: «Топ задач» — топ-3 по часам + «Ключевые блоки» (2 горизонтальных бара по категориям)

## 6. AI-генерация в карточке проекта

### Что генерирует AI
Три блока (помечены ✨ в UI):
1. **Цели проекта** — резюме целей из описания эпика + поля goals
2. **Основной результат** — резюме итогов из описаний дочерних задач + ворклог-контекст
3. **Итог** — заключение: достигнуто ли, что выделяется, риски

### Механика
- **При первом открытии** карточки бекенд автоматически запускает генерацию (если кэша нет):
  1. Собирает контекст: описание эпика + описания дочерних задач (топ-N по часам) + сводку ворклогов
  2. Отправляет в Claude API (prompt-caching для повторных вызовов)
  3. Получает структурированный JSON `{цели, результат, итог}`
  4. Сохраняет в таблицу кэша
  5. Возвращает клиенту (пока грузится — spinner ~3-5 сек)
- **Последующие открытия** — мгновенно из кэша (без вызова API)
- **Кнопка «Обновить ✨»** — принудительная перегенерация с записью в кэш

### Что нужно добавить в синхронизацию из Jira
- Поле `description` для эпиков и дочерних задач (сейчас не синхронизируется)
- Кастомные поля оценки (Качество / Скорость / Результат) — PM предоставит ID полей

### Новая таблица кэша
`project_ai_analysis`: `issue_id (FK)`, `цели`, `результат`, `итог`, `generated_at`, `model_version`

## 7. Решения по открытым вопросам

1. **Поля оценки из Jira** — настраиваются пользователем в разделе Настройки (аналогично полям команды/целей). Хранятся как `AppSetting` ключи `jira_quality_field_id`, `jira_speed_field_id`, `jira_result_field_id`. Синхронизируются в `_upsert_issue` по той же схеме что `team` и `goals`.

2. **Порог «тишины»** — настраиваемый (`AppSetting` ключ `silence_threshold_days`, default 14).

3. **AI-провайдер для генерации** — настраиваемый. Поддерживаем два варианта:
   - **Claude API** (claude-haiku-4-5 — дёшево и быстро; NOT связан с Claude Code — отдельный биллинг Anthropic)
   - **OpenAI API** (GPT-4o-mini — ещё дешевле для резюмирования; требует отдельный API-ключ на platform.openai.com, ≠ ChatGPT Plus)
   
   Настройка через Настройки → вкладка «ИИ»: выбор провайдера + поле для API-ключа. Реализация: абстрактный `AIProvider` интерфейс с двумя реализациями. Default: Claude Haiku.

4. **Drill-down из виджета 2** — отдельный спек, реализуем позже.

5. **Drill-down из виджета 3** — отдельный спек, реализуем позже.

6. **Фильтр по статусу в левой панели** — не нужен в первой версии.

## 8. Что НЕ входит в этот спек

- Widget 4 (PM/sprint метрики) — PM не выбрал ни одного варианта, отложено
- Настройки внешнего вида (цвета ролей и токены) — отдельный спек
- Drill-down страницы в Аналитике для виджетов 2 и 3 — отдельный спек
- Компоновка всей страницы дашборда (расположение 3 виджетов относительно друг друга и существующих KPI-плиток) — уточнить при реализации
