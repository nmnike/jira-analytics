# RFA + Эпики: режим планирования, остаток часов, ручная правка плана

**Дата:** 2026-06-03
**Статус:** утверждён к реализации
**Автор:** brainstorm с PM

## Зачем

В категорию `initiatives_rfa` сейчас попадают и сами RFA-задачи, и их дочерние Эпики. В бэклоге они выводятся как две независимые строки. PM:
- не может сказать сервису «утверждайте Эпик, RFA-родителя считайте контейнером»;
- не видит остатка часов по длительной RFA (которая живёт несколько кварталов и постепенно «съедается» дочерними Эпиками);
- вынужден править плановые часы в Jira → ждать sync → видеть результат; не видит истории правок.

Все три проблемы поднял PM на конкретном примере RFA-6 / PRJ-10587 (короткая пара) и гипотетических длинных RFA на тысячи часов с эпиками по кварталам.

## Цели и не-цели

### Цели
- На каждой паре «RFA-родитель + дочерние Эпики» PM явно выбирает, что идёт в планирование: RFA целиком, по Эпикам, или смешанно (RFA «добивает» непокрытые кварталы).
- Расчётная таблица часов длинной RFA с шестью колонками: План / Факт прошлых Q / Факт текущий / Утверждено / Запланировать / Черновик. Период «прошлых/текущего» берётся из глобального фильтра-квартала сверху страницы.
- Ручная правка плановых часов любой задачи прямо в сервисе. История правок (кто, когда, было/стало, источник, комментарий). Разрешение конфликта sync-vs-ручная без потери данных.
- Изменения покрыты внутренней справкой (`docs/help/*.md`) и лентой «Что нового».

### Не-цели (YAGNI)
- Запись правок плановых часов обратно в Jira (только локально).
- Версионирование любых других полей задачи (статус, описание, исполнитель). Только плановые часы.
- Авто-определение «контейнер vs единица планирования» по эвристикам (тип задачи, наличие Эпиков). PM всегда решает явно.
- Двусторонний sync «черновики ↔ Jira».
- Графики сжигания/Burn-down по RFA. Только табличная сводка.

## Доменная модель

### Режим планирования группы RFA

Группа = RFA-родитель из категории `initiatives_rfa` + все её дочерние задачи той же или родственной планируемой категории (`initiatives_rfa`, `quarterly_tasks`).

У RFA-родителя появляется два состояния:
- **«RFA целиком»** (по умолчанию) — в Сценарий идёт RFA-родитель. Дочерние Эпики скрыты в списке кандидатов на утверждение.
- **«По Эпикам»** — в Сценарий идут дочерние Эпики. RFA-родитель доступен как опциональная отдельная отметка (для непокрытых кварталов длинной RFA).

В режиме «По Эпикам» для каждой задачи группы PM ставит/снимает индивидуальный флаг участия в утверждаемом сценарии:
- Дочерние Эпики по умолчанию `☑`.
- RFA-родитель по умолчанию `☐` (включается осознанно, для «довешивания» непокрытых кварталов).

Пересечение «RFA-родитель `on` + одна из дочек `on` + квартал совпадает» — оранжевый warning «часы посчитаются дважды». Решение остаётся за PM; система не блокирует.

### Таблица часов длинной RFA

Шесть колонок, считаются per-роль (Аналитик / Разработка / Тестирование / ОПЭ) и общим итогом. Период «выбранный квартал» берётся из глобального фильтра-квартала. «Поддерево RFA» — все потомки на любую глубину (дочери, внуки, подзадачи).

| Колонка | Что считаем | Период |
|---|---|---|
| **План** | Поле плановых часов самой RFA из Jira, с учётом ручной правки (см. ниже) | — |
| **Факт прошлых Q** | Σ worklog по всему поддереву RFA (включая чужие команды) | До выбранного квартала |
| **Факт текущий** | Σ worklog только по утверждённым Эпикам поддерева | В выбранном квартале |
| **Утверждено** | Σ плановых часов утверждённых Эпиков | В выбранном квартале |
| **Запланировать** | План − Факт прошлых Q − Утверждено | — |
| **Черновик** | Σ плановых часов Эпиков поддерева, которые ещё не утверждены и не начаты (нет ворклогов) | — |

Замечания:
- «Факт прошлых Q» — весь поддерево, включая out-of-scope ворклоги. PM согласовал: «Если кто-то работал в моём эпике, значит так надо было».
- «Утверждено» — это план утверждённых Эпиков целиком, не остаток. «Факт текущий» — рядом информационно (видно, сколько часов уже отработано в выбранном квартале по утв. эпикам). В прогресс-баре они визуально объединяются как «зарезервировано в текущем квартале», но в таблице цифры raw.
- «Запланировать» = План − Факт прошлых − Утверждено. Это то, что ещё можно положить в новый/текущий сценарий.
- «Черновик» вычитать из «Запланировать» не нужно: черновики ничего не резервируют, они кандидаты.

### Edge cases в таблице

- Запланировать < 0 → красная подсветка; при попытке добавить RFA в новый сценарий — warning «свободного часа не осталось».
- Черновик > Запланировать → оранжевый бейдж в колонке Черновик («черновики превышают остаток»).
- План RFA не задан в Jira → в строке прочерк, бейдж «План не задан», ячейка «Запланировать» пустая.
- Перерасход утв. Эпика в текущем квартале (Факт текущий > Утверждено per роль) → оранжевая подсветка обеих ячеек; в общую формулу «Запланировать» перерасход не входит, расчёт ведётся по плану утв. Эпиков. PM должен либо поднять план утв. Эпика, либо ужать черновики.
- В одной RFA включены и родитель, и дочка в одном квартале — оранжевый warning рядом с заголовком таблицы.

### Ручная правка плана + версионирование

Любую задачу (RFA, Эпик, подзадача — все Issue без исключений) можно править в сервисе по плановым часам каждой из 4 ролей.

Модель хранения на уровне `issues`:

| Поле | Назначение |
|---|---|
| `planned_<role>_hours_jira` | Последнее значение из Jira (синонимично нынешнему `planned_<role>_hours`, переименовываем) |
| `planned_<role>_hours_manual` | Ручная правка; `NULL` если правки нет |
| `planned_<role>_hours_effective` | Вычислимое: `manual ?? jira` (генерируемая колонка либо просчёт в сервисе) |

Новая таблица `plan_audit` — журнал правок:

| Поле | Что |
|---|---|
| `id` | UUID |
| `issue_id` | FK issues.id |
| `role` | analyst / dev / qa / opo |
| `value_before` | int / null |
| `value_after` | int / null |
| `source` | `jira_sync` / `manual_edit` / `manual_revert` |
| `user_id` | FK users.id (для `jira_sync` — NULL) |
| `comment` | nullable; обязателен при `manual_edit` |
| `created_at` | timestamp |

Логика:
- При sync из Jira: если значение в Jira отличается от `planned_<role>_hours_jira`, обновляем `_jira` и пишем запись в `plan_audit` с `source='jira_sync'`.
- Если `planned_<role>_hours_manual` уже задан И новое Jira-значение ≠ старого `_jira` → конфликт: ручная правка остаётся, в карточке RFA/Эпика появляется баннер «В Jira план изменили на N (роль X). Принять Jira-значение / Игнорировать». «Принять» обнуляет `_manual`. «Игнорировать» закрывает баннер до следующего sync, который снова поменяет Jira-значение.
- Если новое Jira-значение совпадает со старым `_jira` — конфликта нет, ручная правка живёт, ничего не показываем.
- Ручная правка: PM нажимает «✎ Редактировать план» (см. ниже), задаёт значения 4 ролей, обязательный комментарий → пишем `_manual`, добавляем строку в `plan_audit` с `source='manual_edit'`.
- Откат к Jira (любой строки истории) → `_manual = NULL`, запись `source='manual_revert'`.

## UI

### Бэклог

Сейчас бэклог — плоская таблица. Добавляем:

1. **Раскрытие строки RFA-родителя.** Стрелка ▶ слева у задач с дочками. Клик → разворачивается панель:
   - прогресс-бар по часам (Факт прошлых Q · Факт текущий · Утв. остаток · Запланировать; черновик — текстом отдельно);
   - таблица 6 колонок по ролям;
   - список дочерних Эпиков со статусом (утв. / черновик / без статуса) и их планом/фактом per-Эпик.

2. **Переключатель режима группы.** В шапке раскрытой панели — радио «RFA целиком ⟷ По Эпикам». В режиме «По Эпикам» рядом с RFA-родителем и каждой дочкой — индивидуальный чекбокс.

3. **Кнопка «✎ Редактировать план».** Открывает drawer (см. ниже). Доступна на любой задаче, не только RFA.

4. **Баннер конфликта.** Появляется над таблицей часов, если по этой задаче есть нерешённый sync-vs-ручная конфликт.

### Сценарий (/planning)

На строке RFA в сценарии — иконка `ℹ` рядом с ключом задачи. Клик → drawer справа с той же таблицей часов и прогресс-баром, что и в бэклоге. Drawer переиспользует компонент.

Радиопереключатель «целиком / по Эпикам» в Сценарии **не показываем** — режим уже выбран в бэклоге, в Сценарии видны только итоговые карточки участников.

### Drawer редактирования плана

- Заголовок: «Редактирование плана · {задача}».
- Таблица: роль / значение Jira (read-only) / значение правки (input).
- Обязательное поле «Комментарий» (минимум 3 символа).
- Кнопка «Сохранить» — сохраняет правку, закрывает drawer. «Отмена» — сбрасывает изменения.
- Под таблицей — ссылка «История правок (N)» — раскрывает журнал той же задачи (дата / автор / роль / было / стало / источник / комментарий, кнопка «Вернуть» на каждой строке).
- Доступ — все аутентифицированные пользователи.

### Карточка задачи (опционально)

Если/когда появится детальная страница задачи — блок «Часы» сверху с той же таблицей и кнопкой правки. В этой итерации не делаем.

## API

Новые / меняющиеся эндпоинты (детальная схема в плане реализации):

- `GET /api/v1/issues/{id}/hours-breakdown?quarter=YYYY-Q` — возвращает 6 колонок по ролям и итог + флаги edge-кейсов.
- `GET /api/v1/issues/{id}/plan-history` — журнал правок.
- `PATCH /api/v1/issues/{id}/plan` — body: `{role_hours: {...}, comment: "..."}` — пишет правку, возвращает обновлённый план.
- `POST /api/v1/issues/{id}/plan/revert` — body: `{audit_id?: ...}` — откат к Jira или к конкретной точке истории.
- `POST /api/v1/issues/{id}/plan/conflict-resolve` — body: `{action: "accept_jira" | "ignore"}`.
- `PATCH /api/v1/backlog/{id}/planning-mode` — body: `{mode: "whole" | "by_epics"}` — переключает режим группы RFA.
- `PATCH /api/v1/backlog/{id}/included` — body: `{included: bool}` — отдельный индивидуальный флаг участия (для режима «по Эпикам»).

## Миграции

- Alembic-миграция, переименовывает `planned_<role>_hours` → `planned_<role>_hours_jira`; добавляет `_manual` (nullable); `_effective` — генерируемая колонка либо просчёт в `BacklogService`.
- Создание таблицы `plan_audit`.
- Backfill: для всех существующих задач `planned_<role>_hours_jira = ныне записанное значение`, `_manual = NULL`. Истории до миграции нет — это нормально.
- Поле «режим планирования» — на уровне `backlog_items` (`planning_mode` enum + per-item `included_in_planning` bool). По умолчанию `planning_mode='whole'`, `included_in_planning=true`.

## Открытые вопросы

- Где хранить `planning_mode` группы — на `backlog_items.id` родителя или отдельной таблицей `rfa_group_settings`? Решим в writing-plans (склоняемся к первому).
- Перформанс расчёта 6 колонок для всех RFA в бэклоге за один запрос — нужно ли кэшировать? Решим после первой реализации, если медленно.
- «Утверждённые Эпики поддерева» — считаем любого потомка RFA на любой глубине, чей `BacklogItem` отмечен в `approved`-сценарии. Если из утв. эпика спущена обычная подзадача (не Эпик) — её ворклог тоже в «Факт текущий»? Склоняемся к да: всё в поддереве утв. узла считаем как «работу по утв. эпику».
- Что делать с перерасходом утв. Эпика в формуле «Запланировать» — оставляем текущую формулу (без перерасхода) или брать `max(Утверждено, Факт текущий)` per роль? Зафиксировано в edge cases как открытый вопрос, по умолчанию — формула из спецификации.
