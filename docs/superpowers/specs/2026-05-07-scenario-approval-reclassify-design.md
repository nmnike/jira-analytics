# Дизайн: утверждение сценария → перевод задач + фильтрация планирования

**Дата:** 2026-05-07  
**Статус:** approved

---

## Контекст

Пользователь вручную отбирает инициативы из бэклога в черновой сценарий квартала, расставляет галочки «включено», затем утверждает сценарий. После утверждения задачи должны автоматически сменить категорию — из «Бэклога инициатив» в «Активный стек» (квартальные задачи).

Параллельно: в экране планирования чернового сценария отображался мусор — задачи уже переведённые в предыдущих кварталах, и дочерние подзадачи, которые не являются самостоятельными инициативами.

---

## Проблемы

### Фича: нет автоматического перевода категории при утверждении

При нажатии «Утвердить» создаются снэпшоты ресурсов и ёмкости, но категории задач не меняются. Пользователь обязан вручную переводить каждую задачу из «Инициатив и RFA» в «Квартальные задачи».

### Баг 1: задачи из прошлых кварталов видны в новом сценарии

Задача ITL-304 была включена в Q2 2026 (утверждённый сценарий) и вручную переведена в «Квартальные задачи». Несмотря на это, она отображается в черновом Q3 2026, так как в базе остаётся её запись в таблице аллокаций и endpoint не фильтрует задачи с категорией «Квартальные задачи».

### Баг 2: дочерние задачи (OS-*) видны в сценарии планирования

Дочерние задачи наследуют категорию от родительской инициативы. Сервис бэклога создавал для них записи и автоматически добавлял их в черновые сценарии. В экране планирования квартала они не нужны — планирование ведётся на уровне корневых инициатив.

---

## Решение

### 1. Хук на утверждение сценария

При утверждении сценария (`POST /planning/scenarios/{id}/approve`), после фиксации статуса:

1. Получить все аллокации с галочкой «включено» для данного сценария.
2. Для каждой задачи: если её категория — «Инициативы и RFA», сменить на «Квартальные задачи».
3. Запустить `BacklogService.sync_from_issue` для каждой задачи — BacklogItem перемещается из бэклога в «Активный стек».
4. Удалить аллокации этих задач из всех других черновых сценариев — они больше не являются кандидатами бэклога.
5. Бэкенд публикует `entity_changed` для backlog и categories — фронтенд инвалидирует кэши.

Задачи с галочкой «не включено» не трогаются — остаются в бэклоге для следующего квартала.

### 2. Фильтр списка задач в черновом сценарии

Endpoint `GET /planning/scenarios/{id}/allocations` для **черновых** сценариев возвращает только аллокации, где:
- эффективная категория задачи = «Инициативы и RFA»
- задача не имеет родительской задачи (корневая инициатива)

Для **утверждённых** сценариев фильтр не применяется — исторический снимок показывается полностью.

### 3. Запрет добавления дочерних задач в черновые сценарии

В `BacklogService` при автоматическом добавлении задачи в черновые сценарии (`_ensure_draft_allocations`) — пропускать задачи с родителем. Новые дочерние задачи больше не попадут в аллокации при синхронизации.

---

## Что не меняется

- Существующие данные OS-* в таблице аллокаций не удаляются и не архивируются. Они скрыты через фильтр в endpoint'е.
- Категория задач без галочки при утверждении не меняется.
- Утверждённые сценарии остаются read-only, отображают полный исторический состав.
- Архив бэклога не используется — дочерние задачи остаются активными во всех аналитических отчётах.

---

## Затронутые компоненты

| Компонент | Изменение |
|---|---|
| `app/api/endpoints/planning.py` | Хук на approve: рекласификация + удаление из черновиков |
| `app/api/endpoints/planning.py` | Endpoint аллокаций: фильтр для черновых сценариев |
| `app/services/backlog_service.py` | `_ensure_draft_allocations`: пропускать дочерние задачи |
| `frontend/src/` | Инвалидация кэшей backlog + categories после approve |

---

## Ретроактивная обработка

Не требуется. Пользователь уже вручную перевёл задачи утверждённых сценариев в «Квартальные задачи». База данных в корректном состоянии. Фильтр endpoint'а автоматически скроет некорректные исторические аллокации.
