# Лента «Что нового» — дизайн

**Дата:** 2026-06-01
**Статус:** утверждён к реализации
**Автор:** brainstorm session с PM

## Зачем

Сейчас пользователи сервиса (аналитики, PM) не узнают что меняется между релизами. Технический changelog в GitHub Release для них непригоден — формулировки коммитов вида `feat(categories): primary_only filter` им непонятны. Нужна лента «Что нового» внутри сервиса на доменном языке.

После каждого релиза пользователь должен:
1. Сразу узнать что появилось — при первом заходе модалка.
2. Иметь возможность вернуться к истории всех версий когда удобно.
3. Получать только бизнес-значимые изменения, не технический шум.

## Цели и не-цели

### Цели
- Лента релизных изменений, видимая всем авторизованным пользователям.
- Заметность нового релиза без раздражения: модалка при первом заходе + красная точка пока не отмечено как прочитанное.
- Группировка внутри версии: тип (Новое / Улучшения / Исправления) с вложенным тегом раздела сервиса.
- Полу-автоматический сбор заметок: Claude добавляет в ходе работы через CLI, PM редактирует/публикует в админке.

### Не-цели (YAGNI)
- Email-рассылка о новом релизе.
- RSS-канал заметок.
- Реакции пользователей (👍/👎), комментарии.
- Связь с системой обратной связи (типа «эта заметка отвечает на запрос #42»).
- Поиск по ленте.
- Метки «бета» / «экспериментально».
- Локализация (пока только русский).

## Жизненный цикл записи

Заметка проходит 3 состояния:

1. **Черновик** — создаётся при работе над фичей/фиксом через CLI-скрипт. Версия пустая. Видна только в админке PM.
2. **Опубликована** — на релизе скрипт привязывает все черновики к новой версии. Показывается пользователям.
3. **Скрытая** — PM пометил «не показывать» (тех-фикс без бизнес-смысла и т.п.). Хранится для аудита, не отображается в ленте и модалке.

Между черновиком и публикацией PM:
- Открывает админку «Что нового» в `/settings`.
- Видит блок «Готовится к выпуску — N записей».
- Правит формулировки, удаляет лишнее, добавляет от себя что забыли.
- Жмёт «Выпустить под версию vX.Y.Z» — записи становятся опубликованными.

## Содержимое одной заметки

Поля (в доменных терминах):

- **Тип** — одно из:
  - Новое (зелёный)
  - Улучшение (синий)
  - Исправление (серый)
- **Раздел** — одно из: Сценарии / Ресурсы / Аналитика / Анализ / Дашборд / Бэклог / Синхронизация / Настройки / Общее
- **Заголовок** — одно короткое предложение что появилось.
  - Пример: «Метки статуса Jira в Сценариях».
- **Описание** — 1-3 предложения зачем нужно и где смотреть.
  - Пример: «Рядом с задачей в сценарии теперь видно её статус из Jira — больше не нужно открывать задачу чтобы проверить готова ли она. Доступно на странице Сценарии.»
- **Ссылка на раздел справки** (необязательно) — прыжок в HelpDrawer на соответствующую статью.

Без скриншотов (по предыдущему опыту в Drawer мелко и нечитаемо).

## Что видит пользователь

### Первый заход после нового релиза — модалка

Поверх страницы открывается модалка «Что нового в v1.1.0»:
- Заголовок: «Что нового в v1.1.0» + дата релиза.
- Блок «Новое» (зелёный акцент) → пункты с тегом раздела.
- Блок «Улучшения» (синий) → пункты с тегом раздела.
- Блок «Исправления» (серый, свёрнут по умолчанию) → пункты с тегом раздела.
- Каждый пункт: иконка типа + тег раздела + заголовок жирным + описание серым.
- Внизу — кнопка «Понятно» (помечает версию прочитанной) + ссылка «Все версии» (открывает полную ленту).

**Не показывается** на странице логина и неавторизованным пользователям.

### Несколько релизов подряд

Если пользователь не заходил неделю и за это время вышло несколько версий — при заходе **одна** модалка с обеими версиями подряд. Заголовки версий внутри как разделители. Не каскад из двух модалок.

### Красная точка на иконке справки

Если пользователь закрыл модалку крестиком (не дочитал) или не открывал её ещё — на иконке справки в шапке висит красная точка. Точка исчезает только когда пользователь зашёл в «Что нового» и нажал «Понятно».

### Полная лента «Все версии»

Отдельная вкладка в HelpDrawer рядом с существующими 14 разделами справки (так пользователь привыкает: «всё про сервис в одном месте»). Свёрнутые блоки по версиям сверху вниз, новые первыми. Каждый блок: заголовок версии + дата + развёрнутое содержимое. Фильтр-чекбокс «Скрыть исправления» вверху.

## Состояние «прочитано» — per-user

У каждого пользователя хранится максимальная версия которую он видел.

**Логика показа модалки**: при заходе сравниваем последнюю опубликованную версию с тем что у пользователя в «прочитано». Если новее — показываем модалку с записями всех версий от (прочитано+1) до текущей. После «Понятно» обновляем «прочитано» на текущую.

**Новый пользователь после регистрации**: «прочитано» инициализируется текущей последней версией. Модалка ему не сваливается на голову при первом входе — он начинает с чистого листа.

## Админка PM (`/settings` → вкладка «Что нового»)

Доступ только админам.

### Верхний блок — «Готовится к выпуску»

- Список черновиков (записей без версии) с прошлого релиза.
- Каждая строка: тип / раздел / заголовок / описание — всё inline-редактируемо.
- Кнопка «Удалить» или «Скрыть от пользователей» на строке.
- Кнопка «+ Добавить запись» — если PM хочет дописать что-то от себя или то что Claude пропустил.
- Кнопка «Выпустить под версию vX.Y.Z» — открывает модалку подтверждения с предложенной версией (берётся из текущего значения app_version). После «Выпустить» — записи становятся опубликованными. Кнопка неактивна если черновиков нет (пустую версию не создаём).

### Нижний блок — «История версий»

- Свёрнутые блоки по версиям сверху вниз.
- Внутри блока — все записи, можно править пост-фактум (опечатки и т.п.).
- Тогглы «Скрыть / Показать» на каждой записи.
- Внизу блока — кнопка «Удалить версию» (откатить публикацию — записи возвращаются в черновики).

### Превью

Кнопка «Посмотреть как пользователь» — открывает ту самую пользовательскую модалку в режиме чтения для проверки финального вида.

## CLI-скрипт для Claude

`scripts/release_note.py`:

```
py -3.10 scripts/release_note.py add \
    --type fix \
    --section sync \
    --title "Кнопка «Остановить синхронизацию» работает мгновенно" \
    --description "..."
```

Что делает: вставляет запись со статусом «черновик», версия пустая.

Параметры:
- `--type` — `new` / `improvement` / `fix` (обязателен)
- `--section` — slug раздела (обязателен)
- `--title` — заголовок одной строкой (обязателен)
- `--description` — 1-3 предложения (обязателен)
- `--help-link` — ссылка на статью справки (опционально)

### Когда Claude добавляет заметку

**Пишу всегда:**
- Новый раздел / страница / вкладка.
- Новая функция или режим работы.
- Заметное улучшение UX (новые фильтры, переработка экрана, ускорение).
- Исправление бага что мешал работать.

**Пропускаю:**
- Рефакторинг без видимого эффекта.
- Мелкие тех-долги.
- Изменения в тестах, доки для разработчика.
- Миграции БД без бизнес-эффекта.
- Стилистические правки.

Если сомневаюсь — пишу. PM при ревью в админке удалит лишнее.

## Связь с релиз-скриптом

`scripts/release.py` после успешного `make release` (тэг создан) вызывает новый шаг: «привязать черновики к новой версии».
- Если черновики есть — все получают версию = только что выпущенную.
- Сообщение в консоли: «Привязано N заметок к vX.Y.Z. Открой /settings → Что нового для финальной проверки перед публикацией.»
- При этом записи ещё **не публикуются автоматически** пользователям — публикация = ручное действие PM в админке (иначе сырые заметки утекут).

**Альтернатива (на будущее)**: автопубликация после привязки с лимитом времени на правку (24 ч). Пока YAGNI.

## Ретроспектива v1.1.0

Релиз уже выпущен без заметок. План одноразовой акции:

## Данные

Описание состояний и связей — в доменных терминах:

**Запись** хранит: версию (или пусто = черновик), тип, раздел, заголовок, описание, ссылку на справку, флаг «скрыта», даты создания и обновления, автора создания, автора последнего редактирования.

**Версия** хранится только как строка вида `v1.1.0` на каждой записи. Отдельной сущности версии нет — список версий вычисляется из distinct по записям.

**У пользователя** добавляется поле «прочитано до версии» — текстовая строка с номером версии. Обновляется при нажатии «Понятно».

## Доступ

- **Лента «Что нового» и модалка** — доступны всем авторизованным пользователям.
- **Админка вкладки «Что нового»** — только админам (как другие вкладки `/settings`).
- **CLI-скрипт `release_note.py`** — для меня (Claude), запускается локально перед коммитом.
