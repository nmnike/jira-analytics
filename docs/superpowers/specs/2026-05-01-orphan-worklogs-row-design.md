# Спек: строка-улавливатель для ворклогов без вида работ

**Дата:** 2026-05-01
**Виджет:** «Нормированные работы» на дашборде

## Проблема

Виджет «Нормированные работы» теряет ворклоги в двух случаях:

1. У задачи нет категории (`Issue.category IS NULL`).
2. У категории задачи не привязан вид работ (`Category.work_type_id IS NULL`) — сейчас это «Архив» и «Незаполненные / сомнительные».

В таких случаях часы молча выкидываются из подсчёта факта. Пример: Шутов Сергей, апрель 2026 — 176 ч в ворклогах, виджет показывает 167 ч; разница = 9 ч в категории «Архив».

## Цель

Все ворклоги сотрудника за период должны попасть в виджет: либо в нормированный вид работ, либо в «Прочие / Чужие задачи», либо в новую строку **«Не указана категория/вид работ»**. Сумма факта в виджете = сумма ворклогов сотрудника за период (без потерь).

## Решение

### Виртуальная строка «Не указана категория/вид работ»

В ответе `/dashboard/norm-work` для каждого сотрудника добавляется виртуальная строка вида работ:

- `work_type_id`: специальный sentinel (`"__unmapped__"`)
- `label`: «Не указана категория/вид работ»
- `plan_hours`: всегда 0
- `fact_hours`: сумма часов ворклогов сотрудника, попавших в orphan-bucket
- `pct`: 0 (план = 0)

Строка показывается только если `fact_hours > 0`.

### Условие попадания в orphan-bucket

В цикле обхода `wl_rows` (`analytics_service.get_dashboard_norm_work`):

1. Если `is_foreign` → идёт в `other_foreign` (как сейчас). **Приоритет other_foreign выше orphan**: ворклог по чужой команде без категории идёт в «Прочие / Чужие задачи», а не в orphan.
2. Иначе если `cat_code is None` → orphan.
3. Иначе если `code_to_wt.get(cat_code) is None` → orphan.
4. Иначе если `wt_id == project_wt.id` → пропускаем (учитывается отдельно через scenario allocations).
5. Иначе → нормированный вид работ.

### Размер БД

Никаких миграций. Виртуальная запись формируется на лету. Если в будущем понадобится назначать на неё процент норматива — добавим реальную запись `mandatory_work_type` через ручной insert/seed.

### Сортировка строк в блоке сотрудника

Существующий порядок видов работ — по `MandatoryWorkType.sort_order`. Виртуальный orphan встаёт перед `other_foreign` (sort_order 98 — где `other_foreign = 99`).

### Сводные итоги виджета

`total_fact` (заголовок виджета) = Σ всех видов работ включая orphan.
`total_plan` = Σ plan_hours (план orphan = 0, не влияет).
`total_pct` = `total_fact / total_plan * 100` — может вырасти, потому что раньше факт был занижен.

То же самое на уровне роли (`role.total_plan/total_fact/total_pct`).

### Подсветка

Существующая логика `overflowZeroPlan` (`plan=0, fact>0`) уже красит строку красным. Дополнительной правки фронта не требуется. Подпись и проценты остаются те же.

### Drill-down (на будущее)

Когда появится новый аналитический отчёт (Спек 2), клик по orphan-строке должен открыть отчёт с фильтром «без вида работ» по этому сотруднику. В рамках текущего спека drill-down не реализуется.

## Тесты

Backend integration test (`tests/test_dashboard_norm_work.py` или аналогичный):

1. Сотрудник с ворклогом на задаче без категории → orphan_row.fact_hours = N.
2. Сотрудник с ворклогом на задаче в категории «archive» (work_type_id=NULL) → orphan_row.fact_hours = N.
3. Сотрудник с ворклогом на чужой команде на задаче без категории → НЕ orphan, а `other_foreign` (приоритет foreign).
4. Сотрудник без orphan-ворклогов → orphan-строка отсутствует в `work_types`.
5. Σ fact по всем видам работ = Σ ворклогов сотрудника за период (минус project_wt, если нет approved scenario).

## Out of scope

- Перепривязка категории «Технический долг» с `support_consult` на `tech_debt` (отдельная задача — это правка справочника, а не кода).
- Drill-down с виджета в новый аналитический отчёт — Спек 2.
- Серверная или фронтовая настройка sort_order orphan-строки.
- UI для редактирования виртуального вида работ.
