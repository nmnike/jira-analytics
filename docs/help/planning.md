# Планирование сценариев

Раздел `/planning` — рабочее место PM, где собирается **квартальный сценарий**: какие инициативы из бэклога мы планируем сделать, силами какой команды, в каких ролевых пропорциях, и хватает ли на это ресурса. Один сценарий = один план на один квартал одной команды.

---

## 1. Зачем раздел

Бэклог инициатив (`/backlog`) — это всё, что хотелось бы сделать. Capacity (`/capacity`) — это сколько часов на самом деле есть у людей. Сценарий — это **обещание**: вот эти конкретные инициативы войдут в квартал, вот так распределится нагрузка, и вот это мы реально потянем.

Раздел отвечает на пять вопросов PM перед утверждением квартала:

- Что входит в квартал, а что мы честно скажем «не успеем»?
- Хватает ли ресурса каждой роли (аналитик / разработчик / тестировщик / ОПЭ)?
- Кто конкретно отвечает за каждую инициативу?
- Что переходит из прошлого квартала и сколько на это осталось доделать?
- Сколько ресурса съедают обязательные работы (сопровождение, встречи, PM), а сколько остаётся под инициативы?

Сценарий проходит две стадии: **черновик** (можно крутить как угодно) и **утверждён** (зафиксирован, изменения только через возврат в черновик с созданием новой ревизии).

---

## 2. Когда сюда заходить

- В начале квартала — собрать новый сценарий, согласовать с командой и руководством, утвердить.
- В середине квартала — если в Capacity изменилась доступность (новый отпуск, новый сотрудник), посмотреть индикатор «Доступность изменилась» и решить, переутверждать ли сценарий.
- В конце квартала — посмотреть, что доделалось из плана, какие инициативы переезжают (откроется метка «продолжение» в следующем сценарии).
- Когда нужно сравнить два варианта плана («с инициативой Х» vs «без неё») — кнопка «Сравнить».
- Когда руководство просит «а что если» — сделать черновик-альтернативу, посмотреть Diff с утверждённым.

## 3. Элементы интерфейса

| № | Элемент | Что показывает |
|---|---------|----------------|
| 1 | Шапка «Сценарии» | Выпадающий список всех сценариев + кнопки «Сравнить» и «Новый сценарий» |
| 2 | Карточка сценария | Имя, квартал, статус (Черновик / Утверждён), бейдж дефицита, счётчик «включено N из M» |
| 3 | Кнопки управления | Утвердить / В черновик / Диаграмма / Diff / История / Экспорт / Удалить |
| 4 | Индикатор «Доступность изменилась» | Жёлтая полоса, если после утверждения у кого-то в команде поменялся Capacity |
| 5 | Панель «Ресурс команды» (справа) | Сколько часов команды доступно, сколько уже забрали обязательные работы, сколько осталось под инициативы |
| 6 | Вкладка «Распределение» | Список инициатив бэклога с галочками включения, исполнителем, заказчиком, ролевыми часами |
| 7 | Вкладка «Правила» | Обязательные работы (сопровождение, встречи и т. д.) — сколько процентов времени каждой роли они занимают |
| 8 | Внешний QA | Поле «часы внешнего тестирования», если QA-ресурс берётся со стороны (не из команды) |

Подробности — в подразделах 3.1–3.8.

### 3.1. Выпадающий список сценариев

Показывает все сценарии, видимые по глобальному фильтру команды. Подпись каждого: `Имя · Квартал Год` + галочка `✓` у утверждённых. Переключение мгновенно перерисовывает всё окно. Если сценариев нет — показывается заглушка с кнопкой «Новый сценарий».

### 3.2. Создание сценария

Кнопка «Новый сценарий» открывает модалку: имя, квартал, год, команда (можно выбрать позже). Сразу после создания сценарий открывается в черновике, инициативы из бэклога этой команды автоматически подтягиваются как «не включённые» — PM проставляет галочки.

Если команда не выбрана — сценарий показывает заглушку с просьбой выбрать команду. Ресурс не считается, пока команды нет.

### 3.3. Статусы: Черновик и Утверждён

- **Черновик** — можно: ставить/снимать галочки, менять исполнителя, менять приоритет, перетаскивать строки, переоценивать часы, менять правила обязательных работ, менять команду, удалять. Все изменения сразу сохраняются на сервере.
- **Утверждён** — всё заблокировано на редактирование. Доступны: возврат в черновик (создаёт новую ревизию при следующем утверждении), сравнение с другим сценарием, экспорт, открытие диаграммы ресурсов, история ревизий.

Утверждение фиксирует **снапшот** Capacity на момент утверждения. Если потом в Capacity что-то поменяется — появится индикатор «Доступность изменилась» (см. 3.7).

### 3.4. Вкладка «Распределение»

Главная вкладка. Колонки слева направо:

| Колонка | Что значит |
|---------|------------|
| Перетащить | Иконка-ручка для перетаскивания строки. Активна только в черновике. |
| Включено | Галочка — войдёт ли инициатива в квартал. Клик по строке тоже переключает. |
| Приоритет | Число 1–10. 1–3 подсвечены бирюзовым. Меняется по клику в поле. |
| Идея | Название инициативы + ключ Jira (кликабельный). Метки: «переоценка» (золотая), «⚠ продолжение» (красная), «В работе» (если инициатива из категории «квартальные задачи» — уже была в работе). |
| Исполнитель | Кто ведёт инициативу. Выбирается из списка сотрудников команды. |
| Заказчик | Из поля Jira «заказчик» (если есть). |
| АН / ПР / ТС / ОПЭ | Часы по ролям: аналитик, разработчик, тестировщик, опытно-промышленная эксплуатация. |
| Всего часов | Сумма + процент от доступного ресурса под инициативы. |

Подсветка строки:
- Бирюзовая — включённая инициатива.
- Красноватая — есть дефицит ресурса.
- Серая — оценки не проставлены (часы по всем ролям = 0).
- Зелёная плашка «В работе» — продолжение из прошлого квартала.

Снизу панель «Ресурс команды» (на правой стороне) пульсирует тем ролевым прямоугольником, который ты только что задел галочкой — визуально видно, какую роль ты сейчас нагрузил.

### 3.5. Метка «⚠ продолжение» и переоценка

Если инициатива уже была в каком-то прошлом сценарии (значит, часть часов уже отработана), при попадании в новый сценарий она помечается «⚠ продолжение». Это сигнал: исходные часы из бэклога не актуальны — надо переоценить остаток.

Клик по иконке-карандашу справа от названия открывает попап **переоценки**:
- Текущие часы (из бэклога).
- Сколько уже фактически потрачено по ролям (из ворклогов).
- Поля «остаток» по каждой роли — что ещё надо сделать в этом квартале.

Сохранённая переоценка маркируется золотой меткой «переоценка». В часах строки и в ресурсе команды используется именно переоценка, исходная оценка бэклога — игнорируется. Снять переоценку — открыть тот же попап и очистить поля.

### 3.6. Вкладка «Правила»

Здесь живут **обязательные работы**: то, что съедает время команды не на инициативы (сопровождение продукта, оргвстречи, техдолг, PM-активности, коммуникации).

Каждая строка правила:
- Категория работ (из справочника `/settings` → «Категории работ»).
- Доля времени роли (в процентах) — например, «аналитик тратит 30% времени на сопровождение».
- Можно сделать **переопределение для конкретного сотрудника**, если у кого-то доля сильно отличается от командного норматива (например, дежурный аналитик — 70% на сопровождение).

Правила копируются из шаблона при создании сценария, но потом редактируются под этот конкретный квартал. Изменение правила сразу пересчитывает «доступный под инициативы» ресурс справа.

### 3.7. Индикатор «Доступность изменилась»

Появляется только у утверждённых сценариев. Если после утверждения в Capacity у кого-то из команды:
- Появилось/удалилось отсутствие,
- Сменился рабочий календарь,
- Изменилась активность,

— система сравнивает снапшот, зафиксированный при утверждении, с актуальным состоянием и показывает жёлтую полосу: «Доступность изменилась (N чел.)». Раскрытие — посотрудничный список изменений (было / стало / разница в часах).

Две кнопки:
- **Пересмотреть сценарий** — возвращает в черновик, чтобы переотметить состав и переутвердить (создаст новую ревизию).
- **Игнорировать** — закрыть индикатор, оставить как есть.

### 3.8. Правая панель «Ресурс команды»

Sticky-панель справа, не уезжает при скролле бэклога:
- **Всего ресурса** на квартал по календарю минус отсутствия.
- **Минус обязательные работы** — по правилам из соответствующей вкладки.
- **Доступно под инициативы** — что осталось на план.
- **По ролям** — отдельно аналитик / разработчик / тестировщик / ОПЭ, с дефицитом, если включённые инициативы превышают доступное.
- **По сотрудникам** — кому сколько часов уйдёт в случае утверждения текущего набора.

Если включённые инициативы перегружают какую-то роль — рамка соответствующей роли становится красной, в шапке сценария появляется бейдж дефицита.

Ниже — поле **внешний QA**: если тестировщик берётся не из команды (например, ручной заказ внешнего ресурса), сюда вписываются часы. Они вычитаются из требуемого QA-ресурса и не нагружают команду.

## 4. Откуда цифры

- **Список инициатив сценария** — из бэклога (`/backlog`), куда они попадают автоматически из задач с категорией «Инициативы (на потом)». Категория настраивается в `/categories`.
- **Ресурс команды** — из Capacity (`/capacity`): рабочий календарь минус отсутствия + активные сотрудники + их роль из реестра ролей.
- **Реестр ролей** — справочник в `/settings` → «Роли». Одна активная роль на сотрудника. Совмещение реализуется через распределение времени в правилах обязательных работ, а не через две роли.
- **Обязательные работы** — из правил сценария (вкладка «Правила»), формируются от шаблона при создании, дальше редактируются вручную.
- **Часы инициативы по ролям** — оценка из бэклога, либо переоценка, сохранённая в этом конкретном сценарии (если стоит метка «переоценка»).
- **Снапшот при утверждении** — копия Capacity на момент утверждения, фиксируется в БД, используется для индикатора «Доступность изменилась».
- **Снапшот ревизии** — каждое утверждение/возврат в черновик создаёт ревизию в истории. Видна по кнопке «История».

## 5. Типовые задачи

### Сценарий А: собрать черновик нового квартала

1. Открыть `/planning`, нажать «Новый сценарий».
2. Указать имя (например, «Q3 2026 — основной план»), квартал, год, команду.
3. Открыть вкладку «Правила» — проверить, что обязательные работы соответствуют плану: какой процент времени аналитик тратит на сопровождение, разработчик на техдолг, и т. д. Поправить, если надо.
4. Перейти на вкладку «Распределение». В правой панели посмотреть «Доступно под инициативы по ролям».
5. Пройтись по списку инициатив сверху вниз (отсортирован по приоритету бэклога). Ставить галочку — следить, как нагрузка по ролям растёт справа.
6. Когда какая-то роль уходит в дефицит (красная рамка) — либо снять галочку с какой-то инициативы, либо переоценить часы (если они завышены), либо принять дефицит осознанно.
7. Назначить исполнителя каждой включённой инициативе (выпадающий список в колонке «Исполнитель»).
8. Когда план собран — нажать «Утвердить» в шапке. Сценарий зафиксирован.

### Сценарий Б: утвердить сценарий

1. Убедиться, что в правой панели нет красных рамок (либо явно принят дефицит).
2. У всех включённых инициатив проставлен исполнитель.
3. У переходящих инициатив («⚠ продолжение») сделана переоценка (иначе используется исходная оценка бэклога — обычно завышенная).
4. Нажать «Утвердить» — сценарий перейдёт в статус «Утверждён», поля заблокируются, под капотом сохранится ревизия + снапшот Capacity.

### Сценарий В: сравнить два сценария

1. В шапке нажать «Сравнить».
2. В Drawer выбрать второй сценарий из списка.
3. Сравнение покажет: какие инициативы есть в одном и нет в другом, чем отличаются по ролям, по исполнителям, по итоговой нагрузке.
4. Альтернатива — открыть `/resource-planning/compare` через кнопку «Диаграмма» (только для утверждённых) — там сравнение визуальное, на диаграмме Ганта.

### Сценарий Г: переоценить переходящие инициативы

Если в новом сценарии у инициативы стоит «⚠ продолжение»:

1. Нажать на иконку-карандаш справа от названия.
2. Посмотреть «Уже потрачено по ролям» (это факт из ворклогов).
3. В полях «Остаток» проставить, сколько часов по каждой роли осталось доделать.
4. Сохранить. Метка инициативы сменится с «⚠ продолжение» на «переоценка». Часы строки и нагрузка справа пересчитаются под новый остаток.

Если переоценка не нужна (например, инициатива закрыта и попала сюда по ошибке) — снять с неё галочку «включено» либо вообще убрать её из бэклога через `/categories`.

### Сценарий Д: вернуть утверждённый сценарий в черновик

1. Нажать «В черновик» в шапке утверждённого сценария.
2. Появится баннер: сценарий снова редактируется.
3. Поправить состав / правила / переоценки.
4. Нажать «Утвердить» — создастся **новая ревизия**. Прошлая остаётся в истории (кнопка «История»).

## 6. Ограничения

- **Одна активная роль на сотрудника.** Совмещение ролей в этой системе не моделируется через две записи — только через перераспределение времени в правилах. Если человек совмещает аналитика и PM-функции, его основная роль — аналитик, а PM-нагрузка живёт в обязательных работах.
- **ОПЭ — отдельный лимит.** Опытно-промышленная эксплуатация — это четвёртая ролевая колонка отдельно от аналитика/разработчика/тестировщика. Часть оценки распределяется между аналитиком и разработчиком по настраиваемому соотношению.
- **Внешний QA — ручной.** Если тестировщик берётся со стороны, его часы вписываются в поле «внешний QA» руками. Из ресурса команды они автоматически не списываются — они уменьшают требование, а не запас.
- **Утверждённый сценарий не редактируется.** Для изменений — обязательно возврат в черновик, что создаёт новую ревизию. Прошлая ревизия остаётся видна в истории.
- **Снапшот Capacity** фиксируется при утверждении. Если ничего не менялось — индикатор не появляется. Если поменялось — он не блокирует ничего, просто сигнал.
- **Список инициатив** в сценарии формируется автоматически из бэклога этой команды. Добавить инициативу «только в этот сценарий» нельзя — сначала она должна быть в бэклоге (= иметь категорию «Инициативы (на потом)» в `/categories`).
- **Удаление сценария** удаляет и все его раскладки. Отменить нельзя.
- **Сравнение сценариев** работает между любыми двумя — черновик с черновиком, черновик с утверждённым, утверждённый с утверждённым.

## 7. Частые проблемы

| Симптом | Причина | Что делать |
|---------|---------|------------|
| Инициатива не появляется в списке сценария | У задачи нет категории «Инициативы (на потом)» либо она не той команды | Открыть `/categories`, найти задачу, проставить категорию «Инициативы (на потом)». Проверить фильтр команды в шапке сценария. |
| Ресурс команды показывает 0 | У сценария не выбрана команда, либо в Capacity у команды нет активных сотрудников | Выбрать команду в шапке сценария. Проверить `/capacity` — есть ли активные сотрудники на этот квартал. |
| Роль перегружена, хотя инициатив мало | Слишком большая доля в правилах обязательных работ (например, 80% времени аналитика на сопровождение) | Открыть вкладку «Правила», уменьшить долю обязательных работ. Либо принять — это объективное ограничение команды. |
| «⚠ продолжение» появилось у инициативы, которая в прошлый квартал не входила | Инициатива могла быть в более раннем сценарии. Помечается, если по ней есть хоть какой-то факт ворклогов | Открыть переоценку — посмотреть «Уже потрачено». Если потрачено мало или ноль — переоценить остаток равным исходной оценке либо принять и не переоценивать. |
| «Доступность изменилась» висит, я уже всё посмотрел | Индикатор сам не закрывается, ждёт явного действия | Нажать «Игнорировать» — индикатор пропадёт. Если изменения значимые — «Пересмотреть сценарий» и пересобрать. |
| Утвердил, но не вижу диаграмму | Кнопка «Диаграмма» доступна только в утверждённых сценариях | Проверить, что статус «Утверждён» (бейдж в шапке). Если черновик — сначала утвердить. |
| Удалил сценарий случайно — как откатить | Никак — удаление необратимо | Создать новый сценарий с теми же параметрами и пройти заново. Если успели сделать ревизии — они тоже удалены. |
| В Diff показывает много изменений, хотя я почти ничего не трогал | Diff сравнивает черновик с **последним утверждённым**, а не с прошлым состоянием черновика | Это норма. Если черновик отличается от утверждённого — все различия показаны. |
| Заказчик пустой в строке | В Jira у инициативы не заполнено поле «заказчик» (кастомное поле) | Заполнить в Jira → дождаться синхронизации → откроется в строке. |

## 8. Связанные разделы

**Откуда приходят данные:**
- **Категоризация** (`/categories`) — категория «Инициативы (на потом)» определяет, попадёт ли задача в бэклог инициатив, а значит и в сценарий.
- **Бэклог инициатив** (`/backlog`) — список инициатив, из которого формируется состав сценария.
- **Capacity** (`/capacity`) — рабочий календарь + отсутствия + активные сотрудники → ресурс команды.
- **Settings → Роли** — реестр ролей, из которого берётся роль каждого сотрудника.
- **Settings → Категории работ** — справочник, из которого выбираются обязательные работы в правилах сценария.
- **Глобальный фильтр команды** (в шапке приложения) — фильтрует список сценариев в выпадающем списке.

**Куда уходит результат:**
- **Диаграмма ресурсов** (`/resource-planning`) — визуальный план Ганта по утверждённому сценарию: кто чем занят по дням.
- **Сравнение сценариев** (`/resource-planning/compare`) — параллельный просмотр двух сценариев.
- **Дашборд** (`/`) — KPI квартального плана берутся из утверждённого сценария квартала.
- **Executive Dashboard** (`/executive`) — план/факт верхнего уровня тоже опирается на утверждённый сценарий.
- **Capacity** (`/capacity`) — таблица «план/факт по сотрудникам» сопоставляет факт ворклогов с тем, что зафиксировано в утверждённом сценарии.

Сценарий — это **обещание команды квартала**. Утверждённый сценарий становится единым источником истины для всех отчётов «что мы планировали».
