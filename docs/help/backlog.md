# Бэклог инициатив

Раздел `/backlog` — реестр инициатив, кандидатов на следующий квартал и архив прошлых периодов. Это **единственный источник** идей для квартального планирования: всё, что не лежит в бэклоге, в сценарии не попадёт.

---

## 1. Зачем раздел

Между разбором задач и планированием сценариев нужен «склад» инициатив. Бэклог собирает кандидатов на квартал в одном месте: что-то прилетает из Jira автоматически (по категории «Инициативы (на потом)»), что-то PM заводит руками как идею-черновик. В бэклоге PM сравнивает инициативы между собой, проставляет приоритеты, оценки по ролям и параметры планирования, переводит выбранные в работу, остальное отправляет в архив.

Дальше эти данные подхватывает планирование сценариев: оценки, приоритеты, исполнители и параметры (вовлечённость, длительность, параллельность) — всё уже посчитано здесь. В сценариях PM решает только «брать / не брать».

---

## 2. Когда сюда заходить

- В начале квартального планирования — собрать список инициатив и проставить приоритеты.
- После синхронизации с Jira — посмотреть, какие новые инициативы прилетели из категории «Инициативы (на потом)».
- После того, как в `/categories` появились новые задачи — убедиться, что они отразились в бэклоге.
- Когда нужно сравнить нагрузку инициатив (часы по ролям, доля от общего ресурса).
- Когда нужно отправить инициативу в архив или вернуть её обратно.
- Когда в сценарии пропала инициатива — проверить, не ушла ли она в архив.

---

## 3. Элементы интерфейса

В шапке — название «Целевые задачи», подзаголовок, кнопка «Обновить с Jira» (сходить в Jira за свежими значениями полей: заказчик, тип затрат, исполнитель — без полной синхронизации) и «Идея вручную» (добавить идею-черновик без привязки к Jira).

Три вкладки:

| Вкладка | Что лежит |
|---------|-----------|
| **Активные** | Инициативы, уже принятые в работу в текущем квартале — закреплены за утверждёнными сценариями. Редактирование заблокировано (только параметры планирования). |
| **Бэклог** | Кандидаты на следующий квартал. Здесь идёт основная работа PM: приоритеты, оценки, параметры. |
| **Архив** | Инициативы, которые в работу не пошли (отложены, отменены, перенесены). Можно вернуть. |

Цифра в скобках — количество инициатив в каждой вкладке с учётом глобального фильтра команды.

### 3.1. Вкладка «Бэклог» (основная)

Главная рабочая поверхность. Инициативы упорядочены по приоритету (число в первой колонке). Порядок фиксированный — перетаскивание отключено, приоритет меняется только вводом числа в колонке «Prio».

Если у инициативы есть дочерние Эпики, слева от строки появится стрелка ▶. Клик раскрывает строку — дочки показываются прямо в таблице с отступом и префиксом `└─`. Дочерний эпик показывает свой Jira-ключ и наследует проект от родителя.

| Колонка | Что показывает |
|---------|----------------|
| **Prio** | Приоритет. Целое число, чем меньше — тем выше. Меняется вводом значения в поле. |
| **Идея** | Название инициативы. Под названием — ссылка на задачу в Jira (если инициатива привязана). |
| **Исполнитель** | Для идей-черновиков — выпадающий список сотрудников. Для привязанных к Jira — assignee из задачи (только просмотр). |
| **Заказчик** | Текстовое поле для черновиков, значение из Jira для привязанных. |
| **Статус** | Статус задачи в Jira + сколько дней назад менялся. ≥180 дней — жёлтым, ≥365 дней — красным (сигнал «зависшая инициатива»). |
| **АН / ПР / ТС / ОПЭ** | Оценка часов по четырём ролям: Аналитик / Разработка / Тестирование / Опытно-промышленная эксплуатация. Клик по ячейке открывает редактор. Цветная полоска внутри ячейки показывает долю роли в общей оценке. |
| **ОПЭ→АН** | Какая доля часов ОПЭ идёт на аналитика (остальное — на разработку). По умолчанию 0.5. |
| **Всего часов** | Сумма всех ролей + процент от общего объёма бэклога. Помогает оценить «вес» инициативы. |
| **Проект** | Ключ проекта Jira, к которому относится инициатива. |
| **Действия** | См. 3.4. |

### 3.2. Вкладка «Активные»

Те же колонки, но без перетаскивания приоритета и без редактирования (инициативы уже в работе). Добавлена колонка **Сценарий** — список утверждённых сценариев, в которые инициатива включена. В действиях остаётся только «Параметры планирования».

### 3.3. Вкладка «Архив»

Те же колонки + **Квартал** — метка квартала, в котором инициатива была отправлена в архив. Сверху — переключатель «Группировать по кварталам»: показывает архив отдельными секциями по периодам. Сортировка по умолчанию — последний квартал сверху.

### 3.4. Кнопки в строке (колонка «Действия»)

Активная вкладка («Бэклог»):

- **Стрелка** — открыть карточку проекта (если инициатива привязана к Jira).
- **Связать с Jira** или **Отвязать** — привязать идею-черновик к существующей задаче или разорвать связь.
- **Карандаш** — редактировать черновик (название, заказчик, описание).
- **Шестерёнка** — параметры планирования: вовлечённость по фазам, длительность, параллельность. Подробнее в 3.5.
- **Папка-стрелка вниз** — отправить в архив.
- **Корзина** — удалить совсем. Если инициатива была в черновиках сценариев — система предупредит, из каких именно сценариев её уберёт.

Вкладка «Архив»:
- **Стрелка-возврат** — вернуть в активный бэклог.
- **Карандаш** — редактировать черновик.
- **Шестерёнка** — параметры планирования.
- **Корзина** — удалить совсем.

### 3.5. Параметры планирования (шестерёнка)

Для каждой роли задаётся:
- **Вовлечённость** — какую долю своего рабочего времени человек отдаёт этой инициативе (0.5 = пол-дня в день).
- **Длительность в днях** — сколько календарных рабочих дней занимает фаза.
- **Параллельность** (для аналитика / разработки / тестирования) — сколько таких инициатив один человек может вести параллельно.

Каждое поле имеет два значения: эффективное (использует планировщик) и пришедшее из Jira (только для просмотра). Бирюзовый тег «Jira» — значение совпадает с Jira-источником. Жёлтый тег «вручную» — PM переопределил вручную. Кнопка «Подтянуть из Jira» возвращает значение Jira.

---

## 4. Откуда цифры

- **Бэклог-инициативы из Jira** — приходят автоматически после того, как задача в `/categories` получила категорию «Инициативы (на потом)». Снять категорию = инициатива выпадет из бэклога.
- **Идеи вручную** — создаются кнопкой «Идея вручную» (черновик без привязки). Можно потом привязать к появившейся задаче Jira через «Связать с Jira».
- **Оценки и параметры из Jira** — длительность фаз и вовлечённость подтягиваются из кастомных полей задачи. Можно переопределить вручную (тег «вручную» в шестерёнке).
- **Фильтр команды** — глобальный, из заголовка приложения. Видны только инициативы, попадающие под выбранные команды.
- **Архивирование инициативы** — автоматически убирает её из всех черновиков сценариев. Утверждённые сценарии не трогаются.
- **Возврат из архива** — инициатива снова попадает в активный бэклог, но в сценарии её придётся добавить заново (через `/planning`).

## 5. Типовые задачи

### Сценарий А: провести квартальный отбор

1. Открыть `/backlog`, вкладка «Бэклог».
2. Убедиться, что в шапке стоит твоя команда.
3. Пройтись сверху вниз: проставить приоритеты (или перетащить строки).
4. Для каждой инициативы открыть шестерёнку и проверить вовлечённость / длительность по фазам.
5. Прикинуть «Всего часов» и долю в проценте — что укладывается в ресурс, что нет.
6. Заведомо непроходные — отправить в архив (папка-стрелка).
7. Перейти в `/planning` или `/resource-planning` — там выбрать инициативы для сценария.

### Сценарий Б: отправить инициативу в архив

1. Найти инициативу во вкладке «Бэклог».
2. В колонке «Действия» — кнопка с папкой-стрелкой вниз.
3. Подтвердить. Инициатива переезжает в «Архив», убирается из всех черновиков сценариев. Связь с Jira сохраняется.

### Сценарий В: вернуть инициативу из архива

1. Открыть вкладку «Архив».
2. Найти инициативу (можно сгруппировать по кварталам).
3. Кнопка «стрелка-возврат». Подтвердить.
4. Инициатива возвращается в «Бэклог» с прежним приоритетом. Если нужно — добавить её в сценарий через `/planning`.

### Сценарий Г: завести идею до появления задачи в Jira

1. Кнопка «Идея вручную» в шапке.
2. Заполнить название, заказчика, оценки по ролям.
3. Когда в Jira появится соответствующая задача — открыть строку, кнопка «Связать с Jira», ввести ключ задачи.
4. После связки часть полей перейдёт в режим «только просмотр» (assignee, заказчик, статус — из Jira).

## 6. Ограничения

- **Связь с категорией обязательна для Jira-инициатив.** Если в `/categories` снять с задачи категорию «Инициативы (на потом)» — она пропадёт из бэклога автоматически.
- **Привязанные к Jira инициативы редактируются ограниченно** — название, статус, исполнитель и заказчик берутся из Jira; в бэклоге можно менять только оценки часов, приоритет и параметры планирования.
- **Архивирование убирает только из черновиков сценариев.** Если инициатива уже в утверждённом сценарии — её оттуда никто не уберёт, придётся править сценарий вручную.
- **Удаление через корзину — необратимо.** Если инициатива привязана к Jira, после удаления она снова появится в бэклоге при следующей синхронизации (пока стоит категория «Инициативы (на потом)»). Если это идея-черновик — пропадёт навсегда.
- **Возврат из архива не возвращает в сценарии.** После восстановления инициативу придётся вручную добавить в нужные сценарии.
- **Группировка по кварталам в «Архиве»** — только в рамках сессии (настройка локальная).
- **Перетаскивание строк не используется.** Приоритет меняется только вводом числа в колонке «Prio».

## 7. Частые проблемы

| Симптом | Причина | Что делать |
|---------|---------|------------|
| Инициатива пропала из бэклога | В `/categories` с задачи сняли категорию «Инициативы (на потом)» или поменяли её на другую | Открыть `/categories`, найти задачу, вернуть категорию «Инициативы (на потом)», сохранить |
| Инициатива есть в бэклоге, но не появляется в сценарии | Инициатива в архиве, либо сценарий привязан к другой команде | Проверить вкладку «Архив» и восстановить; убедиться, что фильтр команды в `/planning` совпадает |
| В бэклоге пусто, хотя в Jira инициативы есть | Глобальный фильтр команды отсекает все задачи, либо новые инициативы ещё не категоризованы | Проверить фильтр команды в шапке; зайти в `/categories` → «К разбору», проставить категорию «Инициативы (на потом)» |
| Не могу поменять название / исполнителя | Инициатива привязана к Jira — эти поля редактируются только в Jira | Поправить в Jira, нажать «Обновить» в шапке бэклога — данные подтянутся |
| Оценка часов в шестерёнке отличается от Jira | PM переопределил значение вручную (жёлтый тег «вручную») | Нажать «Подтянуть из Jira», если нужно вернуть значение источника |
| Инициатива в «Активных», но я её не утверждал | Она лежит в одном из утверждённых сценариев | Открыть колонку «Сценарий» в строке — там список; если нужно убрать — править сценарий в `/planning` |
| После архивации инициатива всё ещё в утверждённом сценарии | Архивация убирает только из черновиков; утверждённые сценарии PM правит руками | Открыть сценарий в `/planning`, убрать инициативу |

## 8. Связанные разделы

**Откуда приходят данные:**
- **Категории** (`/categories`) — единственный источник Jira-инициатив для бэклога. Категория «Инициативы (на потом)» = инициатива в бэклоге.
- **Синхронизация** (`/sync`) — без свежей синхронизации новые задачи из Jira до бэклога не дойдут.
- **Глобальный фильтр команды** — режет, что отображается в каждой вкладке.
- **Настройки → Поля Jira** — определяет, из каких кастомных полей берутся длительность, вовлечённость и заказчик.

**Куда уходит результат:**
- **Планирование сценариев** (`/planning`) — берёт инициативы из «Бэклога» как кандидатов для сценариев.
- **Ресурсное планирование** (`/resource-planning`) — визуализирует загрузку по выбранным инициативам.
- **Дашборд** (`/`) и **Аналитика** (`/analytics`) — инициативы в работе попадают в общую картину по своим задачам Jira.
- **Проекты** (`/projects`) — карточка проекта показывает связанные инициативы.

Если что-то в бэклоге пошло не так — обычно проблема в `/categories` (категория или фильтр команды). Все правки логичнее делать там, бэклог только отражает результат.

## Часы длинной RFA: таблица в шестерёнке

Часы по ролям, режим планирования и правка плана для RFA с дочерними Эпиками живут внутри модалки **«Параметры планирования»** (кнопка-шестерёнка в колонке «Действия»). Открывается одной модалкой и для самой RFA, и для её дочерних Эпиков. В модалке доступно:

- **Прогресс-бар часов** сверху: Факт прошлых кварталов · Факт текущий квартал · Утверждено (остаток) · Запланировать (свободно). Под полосой — отдельная заметка о черновиках.
- **Таблица по ролям** (Аналитик / Разработка / Тестирование / ОПЭ): шесть колонок — План / Факт прошлых Q / Факт текущий / Утверждено / Запланировать / Черновик. Период «текущего» квартала берётся из глобального фильтра наверху страницы. Отменённые Эпики и ITL (статусы Отменено / Rejected / Won't Do) в «Черновик» не учитываются.
- **Переключатель режима планирования группы** (см. ниже).
- **Кнопка «✎ Редактировать план»** для ручной правки часов (см. ниже).

В самой таблице бэклога дочерние Эпики раскрываются прямо в строке RFA (стрелка ▶). Это нужно, чтобы видеть состав группы и заходить в шестерёнку каждой дочки отдельно, если требуется.

## Режим планирования группы

В модалке «Параметры планирования» — переключатель:

- **RFA целиком** (по умолчанию): в сценарий идёт сама RFA, дочерние Эпики скрыты от выбора.
- **По Эпикам**: в сценарий идут отмеченные дочерние Эпики. RFA-родитель остаётся доступным как отдельная галочка — её включают для «непокрытых» кварталов длинной RFA.

Если в режиме «По Эпикам» одновременно отметить RFA и одну из её дочек, а их квартал совпадает — появится оранжевый знак: часы будут посчитаны дважды. Это допустимо в редких случаях; решение остаётся за вами.

## Ручная правка плановых часов

Кнопка «✎ Редактировать план» в модалке «Параметры планирования» открывает drawer справа:

- Слева — текущее значение из Jira (read-only).
- Справа — поле ввода вашей правки.
- Обязательное поле «Комментарий» (объясните, зачем правка).
- Сохранение — мгновенное, без перезагрузки страницы.
- «Сбросить к Jira» — возвращает все значения к синхронизированным с Jira.
- «Показать историю» — журнал всех правок с авторами и комментариями.

Колонка «План» в таблице часов будет показывать вашу правку.

## Конфликт с Jira

Если в Jira план изменили *после* вашей правки, при следующей синхронизации в модалке «Параметры планирования» над таблицей часов появится баннер:

> **В Jira план изменили на N часов (Роль). Сейчас активна ручная правка.**
> [Принять Jira]  [Игнорировать]

- **Принять Jira** — заменяет вашу правку значением из Jira.
- **Игнорировать** — оставляет вашу правку, баннер исчезает.

Ваша ручная правка никогда не теряется автоматически — её можно потерять только через явное «Принять Jira» или «Сбросить к Jira».
