# Категоризация задач

Раздел `/categories` — рабочее место аналитика, где задачи из Jira распределяются по категориям. От этой разметки зависит, попадёт ли задача в аналитику, в бэклог инициатив или в архив. Без разметки задача висит в очереди «К разбору» и не учитывается ни в одном отчёте.

---

## 1. Зачем раздел

Jira присылает в систему сотни задач каждый квартал. Часть из них — реальная работа, часть — оргвстречи, часть — техдолг, часть — мусор. Аналитика и планирование работают только с задачами, у которых проставлена категория. Этот раздел — единая точка, где задачи разбираются по полочкам один раз и навсегда.

Категория не равно «тип задачи в Jira». Категория — это **что мы делаем, когда тратим время на эту задачу**: сопровождаем продукт, тащим квартальную фичу, встречаемся, гасим техдолг, болтаем в чате и т. д. Одна и та же Story в Jira может быть в одном квартале «квартальной задачей», а в другом — «техдолгом».

---

## 2. Когда сюда заходить

- После полной или инкрементальной синхронизации, чтобы разобрать новые задачи (плашка «N ждут разбора» в шапке).
- Когда в Dashboard или Аналитике видна странная цифра «незаполненные worklog» — обычно это значит, что у части задач нет категории.
- Когда задача переехала из одной активности в другую (была квартальной — стала техдолгом) — перекатегоризовать вручную.
- Когда в Бэклоге инициатив пропала ожидаемая инициатива — проверить, стоит ли у неё категория «Инициативы (на потом)».
- Перед выпуском отчёта руководству, чтобы убедиться, что все важные ветки разобраны (см. «Подтвердить» в очереди «К разбору»).

## 3. Элементы интерфейса

| № | Элемент | Что показывает |
|---|---------|----------------|
| 1 | Заголовок «Разбор задач» | Название рабочего места + плашка-счётчик «N ждут разбора» |
| 2 | Пять карточек-очередей | Фильтр по статусу разбора (см. 3.1) |
| 3 | «Скрытые статусы» | Какие Jira-статусы не показывать. По умолчанию ничего не скрыто — «Отменено» видны |
| 4 | «Развернуть всё / Свернуть всё» | Раскрывает или схлопывает дерево целиком |
| 5 | Поиск по ключу/названию | Прыжок к задаче: вводишь ключ/название, дерево автоматически разворачивает цепочку предков и подсвечивает найденную строку (см. 3.2) |
| 6 | «Категория для отмеченных (N)» | Применить категорию ко всем чекбокс-выделенным задачам сразу + одновременно отметить их как подтверждённые |
| 7 | Дерево задач | Иерархия проект → эпик → задача → подзадача с цветовой подсветкой по глубине |
| 8 | Колонка «Категория» | Селектор категории для каждой строки |
| 9 | Колонка «В анализ» | Чекбокс — учитывать ли задачу в worklog-аналитике |
| 10 | Колонка «Действие» (только в «К разбору») | Кнопка «Подтвердить» — применяет выбранную в строке категорию (или принимает унаследованную) и сразу подтверждает её каскадом вниз |

Изменения категорий и галочек применяются мгновенно, без отдельного шага «Сохранить». Подробности по каждому пункту — ниже в подразделах.

### 3.1. Пять очередей разбора

Все задачи в дереве попадают ровно в одну очередь. Очередь определяется по **эффективной категории** задачи (своя или унаследованная от ближайшего предка с категорией) и флагу «верифицирована».

| Очередь | Когда задача попадает сюда |
|---------|----------------------------|
| **К разбору** (жёлтая, в фокусе) | У задачи нет ни своей категории, ни унаследованной от родителя; либо стоит унаследованная, но аналитик ещё не подтвердил её галочкой «Подтвердить». **Это очередь по умолчанию — главная задача аналитика расчистить её до нуля.** |
| **Активные** (зелёная) | Стоит любая «рабочая» категория (Сопровождение, Квартальные, Встречи, Коммуникации, Техдолг, PM, Незаполненные worklog). Эти задачи участвуют в аналитике. |
| **Инициативы** (оранжевая) | Стоит специальная категория «Инициативы (на потом)». Эти задачи попадают в раздел «Бэклог инициатив» и доступны для планирования в сценариях. Не учитываются как факт работы. |
| **Архив квартальных целей** | Квартальные цели старых периодов. Задача была в работе, но не закроется в этом квартале. Не учитывается в плане квартала, но история сохраняется. |
| **Архив неактуальных задач** | Старые и не актуальные задачи. Не относится к работе команды вообще (стороннее, мусор, старое). Не учитывается нигде. |

Цифра на карточке = сколько реальных задач в этой очереди в загруженном дереве (без учёта группировочных строк и контекстных предков). Клик по карточке переключает вкладку.

### 3.2. Структура дерева и поиск

Дерево строится по иерархии задач из Jira (родитель → дети). Глубина подсвечивается цветом фона: чем глубже задача, тем темнее фон строки. Группировочные строки (например «Эпики», «Истории») выделены жирным и не имеют категории — они только для группировки.

**Контекстные строки** (курсивом) — родители вне фильтра команды, показанные для иерархии:
- Чужие эпики, под которыми лежат твои задачи — раньше эти ветки терялись, теперь видны как read-only якоря, можно работать с собственными дочками внутри.
- Архивные родители в архивных вкладках — нужны, чтобы было видно, в какой ветке висит твоя задача.

У контекстных строк нельзя проставить категорию, нельзя включить/выключить из анализа, нет кнопки «Подтвердить». Чекбокс используется только как «выделить всё в этой ветке».

**Поиск работает в режиме «прыжка к задаче»**: вводишь полный Jira-ключ (например `ITL-571`) — система находит задачу, разворачивает цепочку предков, прокручивает к строке и подсвечивает её жёлтой пульсацией. Полный ключ ищется строго (не путать `ITL-57` и `ITL-571`); если введена подстрока — поиск идёт по ключу или названию. Само дерево при этом не фильтруется — все соседи остаются видны.

### 3.3. Унаследованная категория и кнопка «Подтвердить»

Если у задачи нет своей категории, но она есть у предка (эпика или ещё выше) — задача наследует её. В колонке «Категория» это видно как полупрозрачное значение с префиксом «↑». На очередь это влияет так же, как если бы категория была проставлена напрямую: например, дочерние истории эпика с категорией «Архив» попадают в «Прочий архив».

В очереди «К разбору» у такой задачи доступна кнопка **«Подтвердить»** (а на родителе — «Подтвердить +N», где N — количество неверифицированных потомков):

- **Если в селекторе категории ничего не выбрано** — кнопка подтверждает унаследованную категорию каскадом по всему поддереву (трогает только тех потомков, у которых нет собственного ручного выбора).
- **Если в селекторе выбрана категория** — кнопка одновременно применит её к строке и пройдёт каскадом вниз, подтверждая её на потомках без собственного выбора. Один клик = и категория, и подтверждение.

Без подтверждения задача с унаследованной категорией остаётся в «К разбору».

### 3.4. Колонка «Категория»

Селектор с поиском. Выбор категории применяется мгновенно к самой задаче, **а её потомкам без собственной категории та же категория проставится каскадом**. Граница каскада — потомок с ручной категорией PM (его поддерево не трогается). В уведомлении после изменения видно «включая потомков: N».

Если задача переведена в архивную категорию, флаг «В анализ» у её потомков, попавших под каскад, тоже снимется.

Очистка значения (крестик) — снять категорию (вернуться к наследованию или в «К разбору»).

Категории берутся из общего справочника в разделе `/settings` → «Категории». Системные категории (Архив, Инициативы, Незаполненные worklog) — нельзя переименовать или удалить.

### 3.5. Колонка «Статус» и «Статус изменён»

Цвет тега статуса берётся из Jira `statusCategory` (новая / в работе / готово). «Статус изменён» — дата последнего перехода статуса в Jira. Подпись «N д назад» подсвечивается:
- **жёлтым** — задача висит в одном статусе ≥180 дней;
- **красным** — ≥365 дней.

Это сигнал «зомби-задачи». Колонка сортируется кликом по заголовку.

### 3.6. Колонка «Цели»

Фиолетовые теги — значения кастомного поля «Цели» из Jira (если в команде используется). Несколько целей через запятую — несколько тегов. Сортировка по алфавиту первого тега.

### 3.7. Колонка «В анализ»

Чекбокс. Управляет тем, попадают ли worklog по этой задаче в аналитику и план/факт. При выключении на эпике — каскадом отключаются все дети. При выключении на одной задаче — только она.

Архивные категории (Архив квартальных целей, Архив неактуальных задач) **автоматически** снимают этот флаг при сохранении.

### 3.8. Тег с количеством детей

В колонке «Название» рядом с именем задачи появляется серый тег с числом — это количество подчинённых задач в иерархии на текущей вкладке (тот же фильтр статусов и команды, что и для дерева). Помогает оценить, сколько ещё разобрать в ветке, не разворачивая её. Когда раздел развёрнут и нужно ориентироваться по «весу» эпика — этого достаточно.

## 4. Откуда цифры

- **Список задач** — из локальной БД, заполняется синхронизацией с Jira (раздел «Синхронизация»). Бэк строит дерево по полю «родитель» (Parent Link) с учётом правил иерархии типов задач из раздела Settings → Иерархия.
- **Фильтр команды** — глобальный, из заголовка приложения. В команду включаются задачи, где сотрудник команды значится в поле «Команда» (основная или участвующая, настраивается в Settings → Поля Jira).
- **Контекстные строки** — те родители, которые не проходят фильтр команды, но нужны, чтобы показать иерархию. Подгружаются автоматически.
- **Эффективная категория** — собственная категория задачи; если её нет, берётся ближайшая категория предка. Используется и для отнесения к очереди, и для всей даунстрим-аналитики.
- **Унаследованная категория** маркируется стрелкой `↑` в селекторе и **не сохраняется** в БД задачи — это вычисляемое значение, оно автоматически отвалится, если у предка поменять категорию.

## 5. Типовые задачи

### Сценарий А: расчистить очередь «К разбору» после синхронизации

1. Открыть раздел `/categories`.
2. Убедиться, что в шапке стоит твоя команда (глобальный фильтр).
3. Если очередь «К разбору» — оранжевая, кликнуть на карточку.
4. Развернуть дерево («Развернуть всё»), пройтись сверху вниз:
   - Эпик целиком про техдолг → выбрать у эпика категорию «Техдолг»: применится мгновенно и каскадом протянется на потомков без своей категории.
   - Эпик смешанный → проставить категорию каждому ребёнку по отдельности.
   - Задача — болталка/коммуникация → «Внутренние коммуникации».
   - Задача — оргвстреча → «Встречи продуктовых команд».
   - Задача — мусор/случайно попала → «Архив прочих задач».
5. Где категория унаследована от эпика и аналитика с этим согласен — кнопка «Подтвердить» (она же подтвердит и неверифицированных потомков ниже).
6. Очередь «К разбору» уменьшается на каждом действии. Никаких кнопок «Сохранить» — изменения применяются сразу.

### Сценарий Б: массово назначить категорию

1. Кликнуть чекбоксы на нужных задачах (можно ставить на эпик — каскадно выделятся дети).
2. Кнопка «Категория для отмеченных (N)» в правом верхнем углу — нажать.
3. В модалке выбрать категорию из списка.
4. Применить — категория проставится и сразу будет подтверждена для всех выделенных задач.

Контекстные строки (родители вне фильтра команды) в массовое назначение не попадают — они просто работают как «выделить ветку».

### Сценарий В: перенести задачу из активных в архив

1. Открыть вкладку «Активные».
2. Найти задачу (можно скрыть лишние статусы через «Скрытые статусы» или прыгнуть к ней через поиск по ключу).
3. В колонке «Категория» выбрать «Архив квартальных задач» или «Архив прочих задач».
4. Задача мгновенно исчезает из «Активных» и появляется в соответствующей очереди архива. Флаг «В анализ» автоматически снимается у задачи и у её потомков, попавших под каскад.

### Сценарий Г: пометить инициативу на следующий квартал

1. Найти задачу, обычно это эпик уровня инициативы.
2. Категория → «Инициативы (на потом)».
3. Задача появляется в очереди «Инициативы» **и** автоматически попадает в раздел `/backlog` как доступная для планирования инициатива.

## 6. Ограничения

- **Групповые строки** (виртуальные «Эпики», «Истории» как заголовки секций) **не имеют категории** — их нельзя категоризовать, чекбокса нет.
- **Контекстные строки** (чужие эпики над твоими задачами или архивные родители) — нельзя категоризовать, нельзя выключить из анализа. Только подсветка иерархии.
- **Архивные категории** автоматически снимают «В анализ» и у самой задачи, и у потомков, попавших под каскад. Вернуть обратно можно только через перекатегоризацию + ручную галочку.
- **Унаследованная категория не сохраняется в БД** — если у эпика сменят категорию, задача автоматически перейдёт в другую очередь. Если нужно «зафиксировать» категорию у дочерней задачи — поставить её явно.
- **Каскад на потомков** работает только на тех, у кого нет собственной ручной категории. Любая задача с уже выбранной категорией PM — граница каскада, её поддерево не трогается.
- **Колонки можно изменять по ширине** (потянуть за правый край заголовка), но настройка не сохраняется между сессиями.
- **Скрытые статусы** не сохраняются между сессиями (только в рамках открытой вкладки). По умолчанию ничего не скрыто.

## 7. Частые проблемы

| Симптом | Причина | Что делать |
|---------|---------|------------|
| «К разбору» показывает много задач, хотя кажется, я уже разобрал | Свежая синхронизация принесла новые задачи; либо у эпиков с новыми детьми категория эпика не унаследована до подтверждения | Открыть «К разбору», пройтись «Подтвердить» по тем, где категория уже унаследована корректно |
| Поставил категорию — задача всё ещё в «К разбору» | Категория стоит у родителя, у задачи — пусто, флага верификации нет | Найти задачу в «К разбору», нажать «Подтвердить». Либо явно выбрать категорию селектором и нажать «Подтвердить» — категория применится и сразу будет подтверждена. |
| В дереве вообще пусто | Фильтр команды отсекает все задачи; либо нет настроенных проектов в Scope | Проверить глобальный фильтр команды в шапке и Settings → Scope. Если scope пуст — добавить проекты |
| Селектор категории — пустой список | Справочник категорий пуст (редкость, обычно сразу после первой миграции) | Settings → Категории, проверить, что системные категории есть |
| Стрелка `↑` на категории, не понимаю откуда | Категория унаследована от ближайшего предка с категорией | Развернуть дерево вверх, найти предка с категорией — это и есть источник |
| После архивации задачи в Аналитике осталась её история | Архив снимает «В анализ» вперёд, исторические worklog остаются в фактах прошлых месяцев | Это нормальное поведение — снимаем учёт только с этого момента, не задним числом |
| Кнопка «Подтвердить +N» — что значит +N | На этой задаче-родителе есть N неподтверждённых потомков; нажатие подтвердит всю ветку каскадом (до границ — потомков с собственной ручной категорией) | Если на родителе уже выбрана новая категория, тот же клик заодно применит её к подходящим потомкам |
| Не могу найти отменённые задачи | Раньше «Отменено» по умолчанию скрывалось — сейчас они видны без настройки | Если их слишком много — открыть «Скрытые статусы» и скрыть вручную |
| Под чужим эпиком моя задача — но эпик нельзя категоризовать | Это контекстный родитель (вне твоего фильтра команды); он показан только для иерархии | Это норма. Категорию ставь на свою задачу — на её отображение в очереди контекстный родитель не влияет |
| Ввёл точный ключ задачи в поиске — не находит | Возможно, фильтр команды отсекает её или она лежит вне scope-проектов | Проверить глобальный фильтр команды; если задача чужой команды — она может оказаться видна только как контекст под твоей веткой |

## 8. Связанные разделы

**Откуда приходят данные:**
- **Синхронизация** (`/sync`) — без свежей синхронизации в дереве не появится новых задач из Jira.
- **Settings → Scope** — определяет, какие проекты Jira вообще видны в дереве.
- **Settings → Иерархия задач** — определяет, какие типы задач считаются «родителями», какие — «контейнерами». Меняет структуру дерева.
- **Settings → Категории** — справочник, из которого берётся список в селекторе.
- **Settings → Поля Jira** — определяет, какое поле Jira считается «командой» (для глобального фильтра).
- **Глобальный фильтр команды** (в шапке приложения) — режет, что попадает в дерево.

**Куда уходит результат:**
- **Дашборд** (`/`) — KPI и виджеты считают только задачи с проставленной категорией и флагом «В анализ».
- **Аналитика** (`/analytics`) — иерархический отчёт показывает worklog в разрезе категорий.
- **Отчёт по видам работ** (`/analytics/work-type-report`) — тематический отчёт работает только на задачах с категориями.
- **Проекты** (`/projects`) — карточка проекта группирует задачи по категориям.
- **Бэклог инициатив** (`/backlog`) — автоматически забирает задачи с категорией «Инициативы (на потом)».
- **Планирование сценариев** (`/planning`, `/resource-planning`) — работают только с инициативами из бэклога → значит, через категорию `initiatives_rfa`.
- **Capacity** (`/capacity`) — план/факт по сотрудникам разбит по категориям задач, на которые они логировали.
- **Executive Dashboard** (`/executive`) — все KPI верхнего уровня идут через категории.

Ошибка категоризации в этом разделе → ошибка в каждом из перечисленных разделов. Это самый «ответственный» экран в системе.

## Большие команды

Когда в выбранной команде больше 1000 задач, дерево раскрывается лениво. Сначала видны только верхнеуровневые задачи и эпики — этого достаточно, чтобы открыть страницу за секунду даже при 6000+ задач. Чтобы увидеть подзадачи эпика, кликните на стрелочку рядом с ним — потомки подтянутся отдельно. Поиск работает на сервере: введите часть ключа или названия — отфильтруются только подходящие корни.

Кнопка «Развернуть всё» отключена для больших деревьев (создаст слишком много запросов). Раскрывайте интересующие эпики поштучно. Кнопка «Свернуть всё» сбрасывает ранее загруженные потомки — следующий клик «развернуть» подтянет их заново.
